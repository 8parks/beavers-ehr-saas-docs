# Multi-tenant EHR SaaS 보안 아키텍처 가이드

> AWS 네이티브 멀티 테넌트 EHR SaaS를 위한 아키텍처 결정 기록 및 보안 설계 문서입니다.

## 목적

해당 가이드라인은 가상의 AWS 기반 EHR(전자건강기록) SaaS 플랫폼을 위한 아키텍처 결정 사항, 보안 설계 지침, 위협 모델링 및 컴플라이언스 연계 항목을 체계적으로 정리한 문서입니다.

여러 병원 또는 의료 기관이 동일한 SaaS 플랫폼을 공유하는 멀티 테넌트 환경에서 테넌트 격리, 감사 추적성, 보안 컨트롤을 어떻게 설계할 수 있는지를 탐구합니다. 

> 선택된 규제 및 보안 요구사항을 기반으로 EHR SaaS 레퍼런스 모델을 AWS 상에서 구현할 때 고려할 사항을 제공하고자 합니다.

### 핵심 설계 목표
| 목표 | 구현 방식 |
| --- | --- |
| 멀티 테넌트 격리 | 테넌트 간 PHI 접근 방지, 스키마/서비스 레이어 격리 |
| PHI 보호 | 저장 및 전송 시 암호화, 키 관리, 안전한 인증/인가 |
| 감사 추적성 및 로그 무결성 | CloudTrail, S3 Object Lock, KMS 기반 로그 보존 |
| 컴플라이언스 인식 아키텍처 | HIPAA, 개인정보보호법, ISO 27001 등 연계 |
| 보안 자동화 및 대응 | AWS Config, Security Hub, EventBridge, SSM 자동화 |

## 대상 독자

- AWS 기반 보안 아키텍처에 관심 있는 학생 및 개발자
- 헬스케어 SaaS 플랫폼의 멀티 테넌트 설계에 관심 있는 보안 엔지니어 
- 컴플라이언스를 고려한 클라우드 아키텍처에 관심이 있는 분 

## 가이드라인 범위 

::: tip 이 문서가 다루는 것
- 멀티 테넌트 EHR SaaS 레퍼런스 아키텍처
- 테넌트 격리 전략 및 계정 분리 전략
- ID 및 접근 관리
- PHI 데이터 보호
- 감사 로그 및 로그 무결성
- KMS 기반 암호화 및 서명
- 보안 이상 탐지 및 자동 대응 흐름
- 연구 데이터셋 생성 및 가명처리 흐름
- 위협 모델링 및 위협-컨트롤 매핑
- 컴플라이언스 인식 설계 매핑
- AWS 네이티브 보안 컨트롤 카탈로그
- OWASP ASVS 기반 검증 체크리스트
- 한국어 우선, 영어 지원 이중 언어 문서 구조
:::

::: warning 이 문서가 다루지 않는 것
- 실제로 돌아가는 완성된 프로덕션 애플리케이션은 만들지 않았어요.
- 법적으로 컴플라이언스를 완전하게 보장한다고 말하는 문서는 아니에요.
- 규정을 공식적으로 법률 해석하는 건 범위 밖이에요.
- 전체 침투 테스트까지는 진행하지 않았어요.
- 비용 최적화는 보안에 비해 상대적 우선순위가 낮아 깊게 고려하지 않았아요.
:::

자세한 사항은 [시스템 컨텍스트 & 가정](/ko/01-system-context)를 확인하세요!


## 문서 한 눈에 보기
