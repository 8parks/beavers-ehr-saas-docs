# 시스템 컨텍스트 & 가정

> 원활한 논의를 위해 보안, 컴플라이언스, 아키텍처를 논하기 전에 먼저 설계 대상 시스템을 정의합니다.

## 서비스 개요

Baby Beavers EHR SaaS는 여러 병원 또는 의료 기관이 공통 플랫폼 인프라를 공유하는 멀티 테넌트 SaaS 모델의 가상 전자건강기록 플랫폼입니다.

자세한 서비스 시나리오는 [시나리오 기반 보안 설계](/ko/02-scenarios/)에서 다룹니다.

## 사용자 및 역할

| 역할 | 설명 | 테넌트 소속 |
|------|------|------------|
| 환자 | 자신의 진료 기록 조회 가능 | 테넌트 (병원) |
| 의사 | 환자 기록 작성, 조회, 서명 | 테넌트 (병원) |
| 간호사 | 환자 상태 기록 및 일부 정보 조회 | 테넌트 (병원) |
| 병원 관리자 | 병원 내 사용자 및 역할 관리, 병원 단위 설정 | 테넌트 (병원) |
| 연구원 | 가명처리된 연구 데이터셋 접근 | 테넌트 (연구기관) |
| SaaS 운영자 | 테넌트 생성, 시스템 운영 및 모니터링 | SaaS 제공자 |

### 역할별 접근 범위 및 보안 고려사항

**병원 관리자**
- 병원 내부 사용자 계정 및 역할을 관리하고 병원 단위 설정을 확인할 수 있음
- 개별 환자의 상세 진료기록에는 접근하지 않는 것을 원칙으로 함
- 권한 범위가 넓기 때문에 계정 탈취 또는 권한 오남용 시 전체 테넌트 데이터 노출로 이어질 수 있어 계정 수를 최대 3개로 제한

**의사**
- 환자 정보 조회, 진료 기록 작성, 처방, 서명 등 대부분의 의료 데이터를 생성하고 접근할 수 있는 주요 사용자
- 접근 제어가 제대로 이루어지지 않을 경우 다른 환자 또는 다른 테넌트 데이터 접근으로 이어질 수 있음

**간호사**
- 환자 상태 기록 및 일부 정보 조회를 담당하며 의사보다 제한된 권한을 가짐
- 데이터 접근 경로 자체는 의사와 유사하며, 권한 검증이 불완전할 경우 의사 권한으로 상승할 가능성이 존재

**연구원**
- 비식별화 또는 가명처리된 연구용 데이터셋에만 접근 가능하며 원본 PHI에는 접근할 수 없음

**SaaS 운영자**
- 테넌트(병원)를 생성하고 인프라 운영 및 모니터링을 담당
- 인프라 접근 권한은 있지만 테넌트 내부 의료 데이터에는 직접 접근하지 않는 것을 기본 가정으로 설정
- 접근 불가 증빙은 테넌트별 KMS 키 기반 암호화 및 CloudTrail 로그를 통해 확보
- 내부자 위협 및 권한 오남용 가능성에 대해 Break-glass 절차와 별도 모니터링 적용

## 설계 가정 및 제한사항
### 테넌트 모델

#### 테넌트란 무엇인가?

이 설계에서 **테넌트는 의료 기관 단위**로 정의됩니다. 각 병원은 고유한 `tenant_id`를 가지며, 하나의 SaaS 환경에서 여러 테넌트가 동시에 운영됩니다.

#### 격리 수준

물리적 분리가 아닌 **논리적 격리**를 기준으로 설계합니다.

- 모든 데이터는 `tenant_id`를 기준으로 구분되며 DB 레벨에서 필터링됨
- API 요청 시 인증 정보(JWT 등)에서 `tenant_id`를 추출하여 요청마다 검증 수행
- 테넌트 검증이 누락되거나 잘못 구현될 경우 다른 병원 데이터 접근이 가능해지므로 주요 공격 포인트로 간주

#### 테넌트 계층

| 역할 유형 | 설명 |
|-----------|------|
| 테넌트 관리자 | 병원 내 사용자 계정 및 역할 관리, 설정 확인 |
| 애플리케이션 사용자 | 의사, 간호사, 환자 등 실사용자 |
| SaaS 운영자 | SaaS 제공자 측 테넌트 생성 및 인프라 운영 |

#### 테넌트 간 접근 가정

- 기본적으로 테넌트 간 PHI 접근은 **허용되지 않습니다**.
- 플랫폼 관리자도 테넌트 PHI에 직접 접근하지 않는 것을 원칙으로 합니다.
- 비상 접근이 필요한 경우 별도 절차를 거치며 모든 접근은 즉시 기록됩니다. (→ [S6. 운영자 비상 접근](/ko/02-scenarios/s6-break-glass))
