# 위협 모델링

이 플랫폼은 멀티테넌트 의료 SaaS라는 특성상 일반적인 웹 애플리케이션보다 더 넓은 공격면을 가진다. 병원 사용자 경로, 연구 데이터셋 제공 경로, 운영자 예외 접근 경로가 서로 다르고, 하나의 통제 실패가 곧바로 PHI 노출 또는 테넌트 격리 붕괴로 이어질 수 있다. 따라서 위협 모델링은 단순한 설명 부록이 아니라, 어떤 자산을 어떤 경계에서 어떻게 보호할 것인지 정리하는 설계 기준으로 사용해야 한다.

이 섹션의 보안 요구사항은 4장 컴플라이언스에서 도출된 요구사항을 기준으로 하며, 시나리오 `S1`부터 `S7`까지의 실제 워크로드를 분석 대상으로 삼는다.

## 보안 목표 {#security-objectives}

위협 모델링은 다음 다섯 가지 목표를 동시에 만족시키는 방향으로 수행한다.

| 목표 | 이 플랫폼에서의 의미 |
|------|--------------------|
| 기밀성 | 한 테넌트의 PHI, 첨부파일, 임상 메타데이터는 인가된 동일 테넌트 사용자 외에는 접근할 수 없어야 한다. 운영자도 기본 경로로는 원본 PHI에 접근할 수 없어야 한다. |
| 무결성 | 진료 기록, 첨부파일 메타데이터, 승인 상태, 테넌트 매핑 정보는 무단으로 변경되거나 조작되어서는 안 된다. 장애 복구 이후에도 같은 무결성 속성이 유지되어야 한다. |
| 가용성 | 병원 운영과 연구 데이터셋 제공에 필요한 핵심 기능은 장애 상황에서도 서비스 수준을 유지해야 하며, 복구 과정이 새로운 격리 실패를 유발해서는 안 된다. |
| 프라이버시 | 연구 데이터셋 제공 시 원본 PHI 직접 제공을 금지하고, 제공 범위 최소화, 가명처리, 재식별 위험 저감, 목적 기반 승인 절차를 갖춰야 한다. |
| 책임추적성 | PHI 접근, 연구 데이터셋 반출, 운영자 예외 접근, 주요 권한 변경은 사후에 행위자·목적·대상 자산·시간을 재구성할 수 있을 정도의 증적을 남겨야 한다. |

## 위협 모델링 방법론 {#methodology}

이 플랫폼은 기술적 보안 위협과 프라이버시 위협을 동시에 다뤄야 하므로, `STRIDE`와 `LINDDUN`을 함께 사용한다.

| 방법론 | 적용 목적 | 이 플랫폼에서 중점적으로 보는 항목 |
|--------|----------|------------------------------|
| STRIDE | 기술적 공격면 식별 | 위장, 권한 상승, 데이터 변조, 비밀정보 노출, 서비스 거부, 테넌트 격리 붕괴 |
| LINDDUN | 프라이버시 침해 식별 | 데이터 연계 가능성, 재식별 가능성, 정보 주체 비인지 처리, 연구 목적 외 활용, 규제 위반 |

### 적용 방식

1. 먼저 분석 대상을 `S1. 환자 진료 기록 작성`, `S2. 환자 진료 기록 조회`, `S3. 연구 데이터셋 생성`, `S4. Tenant 온보딩`, `S5. Tenant 오프보딩`, `S6. 운영자 비상 접근`, `S7. 데이터베이스 장애 및 복구`로 고정한다.
2. 각 워크로드에서 PHI, 연구 데이터셋, 테넌트 매핑 정보, 운영자 예외 경로처럼 사고 파급력이 큰 자산을 식별한다.
3. 인증 정보, tenant context, 데이터 저장소, dataset delivery, 운영자 접근 경로가 교차하는 지점을 신뢰 경계로 정의한다.
4. 각 신뢰 경계마다 성립 가능한 공격 또는 실패 시나리오를 작성한다.
5. 각 시나리오에 대해 필수 예방 통제, 탐지 통제, 대응 통제, 운영 증빙을 연결한다.

### 위험도 분류 기준

이 섹션의 위험도는 단순 취약점 심각도가 아니라, 실제 운영 환경에서 사고로 이어질 가능성과 파급 범위를 함께 고려한다.

| 위험도 | 분류 기준 |
|--------|-----------|
| 높음 | 단독으로 PHI 노출, 크로스 테넌트 접근, 운영자 예외 경로 악용, 데이터 무결성 훼손, 조사 불가능 상태를 만들 수 있는 경우 |
| 중간 | 단독으로 즉시 대규모 사고를 만들지는 않지만, 다른 통제 실패와 결합될 때 사고 범위를 크게 확대하거나 복구를 어렵게 만드는 경우 |
| 낮음 | 운영 효율이나 관리 복잡도에는 영향을 주지만, 단독으로 PHI 노출 또는 격리 실패로 직결되지는 않는 경우 |

## 자산 식별 {#assets}

위협 모델링의 기준 자산은 다음과 같다.

| 자산 ID | 자산 | 설명 | 주요 위협 관점 | 위험도 |
|--------|------|------|---------------|--------|
| `A-01` | 환자 PHI 및 임상 첨부파일 | 환자 기본정보, 진료 기록, 진단, 처방, 검사 결과, 첨부 문서 및 영상 | 크로스 테넌트 노출, 무단 조회, 무단 수정, 유출 사고 | 높음 |
| `A-02` | 연구 데이터셋 산출물 | 가명처리되었거나 비식별화된 연구용 데이터셋, 다운로드 아티팩트, 제공 이력 | 재식별, 과다 제공, 승인 우회, 만료 후 잔존 접근 | 높음 |
| `A-03` | 가명처리 규칙 및 매핑 정보 | 가명식별자 생성 규칙, 일반화 기준, 재식별 위험 저감 정책, 매핑 테이블 | 재식별 공격, 목적 외 활용, 규제 위반 | 높음 |
| `A-04` | 인증 토큰 및 사용자 속성 | JWT, tenant claim, role claim, tenant type, group, scope, 세션 상태 | claim 불일치, 세션 재사용, 비활성 계정 우회 | 높음 |
| `A-05` | 테넌트 매핑 정보 | `tenant_id`, schema 또는 prefix 매핑, 상태 정보, 온보딩/오프보딩 상태 | 잘못된 매핑, 비활성 테넌트 우회, 온보딩 오류 | 높음 |
| `A-06` | 데이터 평면 인가 상태 | DB session context, RLS 상태, 스키마 바인딩, 객체 경로 제약 | 테넌트 컨텍스트 혼용, RLS 우회, prefix 우회 | 높음 |
| `A-07` | 비밀정보 및 키 자산 | DB 자격증명, 운영 비밀정보, KMS 키, 서명 키, 예외 접근용 비밀 | 비밀정보 노출, 과도한 decrypt/sign 권한, 운영자 남용 | 높음 |
| `A-08` | 감사 증적 및 승인 기록 | CloudTrail, 애플리케이션 감사 로그, 다운로드 이력, approval trail, 세션 로그 | 조사 불능, 행위 부인, 영향 범위 산정 실패 | 중간 |
| `A-09` | 운영자 예외 접근 경로 | SSM 기반 접근 경로, 관리용 bastion, 임시 권한, 승인 티켓 | break-glass 상시화, 승인 우회, 수동 질의 | 높음 |
| `A-10` | 백업, 스냅샷, 오프보딩 잔존물 | DB 백업, 스냅샷, 보존 사본, 만료 전 dataset, 오프보딩 전후 아티팩트 | 오프보딩 후 잔존 접근, 복구 후 격리 붕괴 | 중간 |

### 자산 식별 시 유의사항

- `A-01`과 `A-02`는 모두 데이터 자산이지만, 보호 방식은 같지 않다. 원본 PHI는 의료 서비스 운영 경로에서만 처리하고, 연구 데이터셋은 승인·가명처리·반출 통제까지 포함해 별도 자산으로 다뤄야 한다.
- `A-05`와 `A-06`은 물리적 저장 데이터가 아니더라도, 멀티테넌트 격리 실패를 설명하는 핵심 자산이다. 테넌트 매핑 정보와 세션 컨텍스트가 틀리면 그 아래 모든 통제가 함께 무너질 수 있다.
- `A-08`은 즉시 PHI를 노출하지는 않더라도, 사고 후 원인 분석과 규제 대응을 불가능하게 만들 수 있으므로 별도 관리 대상이 된다.

## 신뢰 경계 {#trust-boundaries}

이 플랫폼에서 신뢰 경계는 단순 네트워크 선이 아니다. 동일한 인증 상태나 동일한 정책 집합을 더 이상 그대로 신뢰할 수 없는 지점, 즉 다시 검증해야 하는 지점을 뜻한다.

| 경계 ID | 신뢰 경계 | 경계를 넘는 데이터 | 핵심 검증 포인트 | 대표 위협 |
|--------|-----------|-------------------|-----------------|----------|
| `TB-01` | 인터넷 ↔ CloudFront / WAF / API Gateway | 로그인 요청, API 요청, access token | TLS, WAF 정책, rate limiting, authorizer 적용 여부 | 위조 요청, 웹 공격, 토큰 탈취 |
| `TB-02` | Cognito ↔ API Gateway / Lambda | JWT claim, scope, group, tenant type | 서명 검증, claim 정합성, scope 검증, 계정 상태 반영 | claim 불일치, 토큰 재사용, 역할 우회 |
| `TB-03` | Lambda ↔ tenant registry | `tenant_id`, schema, prefix, tenant 상태 | 사용자 입력이 아니라 서버 관리 allowlist 사용, 상태 전이 검증 | 잘못된 매핑, 비활성 테넌트 우회 |
| `TB-04` | Lambda ↔ RDS Proxy ↔ Aurora | SQL, tenant context, query result | 세션마다 tenant context 재설정, RLS 적용, schema binding 검증 | 크로스 테넌트 조회, 세션 혼용 |
| `TB-05` | Lambda ↔ S3 / DynamoDB / Secrets / KMS | object key, dataset 상태, secret, key operation | prefix 제한, least privilege IAM, key usage 분리 | 객체 경로 남용, 비밀정보 노출, KMS 남용 |
| `TB-06` | 병원 테넌트 ↔ 연구기관 테넌트 | dataset request, approval state, dataset artifact | 목적 기반 승인, 제공 범위 제한, 재식별 위험 저감 | 과다 제공, 재식별, 승인 우회 |
| `TB-07` | 운영 평면 ↔ 데이터 평면 | SSM 세션, admin action, break-glass credential | 승인 여부, JIT access, 세션 로깅, 사후 검토 | 운영자 남용, standing privilege |
| `TB-08` | 연구기관 클라이언트 ↔ dataset delivery path | presigned URL, dataset object, 만료 정보 | 짧은 만료시간, 다운로드 시점 재검증, 반출 로그 | URL 재사용, 외부 공유, 만료 후 접근 |
| `TB-09` | 운영자 단말 ↔ SSM tunnel ↔ 관리용 bastion ↔ Aurora | 세션 시작 이벤트, 포트 포워딩, 수동 질의 | 승인 티켓, 세션 로그, DB 작업 증적 연계 | break-glass 상시화, 조사 불능 |

### 신뢰 경계 해석

- `TB-02`는 인증 경계다. 토큰이 있다는 사실만으로 충분하지 않으며, `tenant_id`, role, group, tenant type, scope가 함께 일관돼야 한다.
- `TB-03`은 멀티테넌트 격리의 기준 경계다. schema나 객체 경로를 사용자 입력에서 파생하지 않고, 서버가 관리하는 registry에서 읽어야 한다.
- `TB-04`는 데이터 평면 경계다. RDS Proxy 또는 커넥션 재사용 환경에서는 이전 요청의 tenant context가 다음 요청으로 누수되지 않도록 매 요청마다 context를 재설정해야 한다.
- `TB-08`은 외부 반출 경계다. 승인 상태와 실제 다운로드 행위는 다른 사건이므로, URL 발급 시점과 다운로드 시점 모두를 통제해야 한다.
- `TB-09`는 운영자 예외 경계다. 네트워크상 private path라는 사실만으로 안전하다고 볼 수 없으며, 세션 승인, 대상 인스턴스, 이후 DB 작업 증적이 하나의 체인으로 연결돼야 한다.

## 위협 시나리오 카탈로그 {#threat-scenarios}

위협 시나리오는 단순 공격 예시가 아니라, 설계 검토와 배포 전 점검에 직접 사용할 수 있는 수준으로 정의한다. 각 시나리오는 관련 워크로드, 성립 조건, 영향, 필수 통제를 함께 본다.

| 위협 ID | 위협 시나리오 | 위험도 | 관련 워크로드 |
|--------|--------------|--------|--------------|
| `T-01` | Cross-Tenant Access | 높음 | `S1`, `S2`, `S3` |
| `T-02` | 동일 테넌트 내부 권한 상승 | 높음 | `S1`, `S2`, `S4`, `S6` |
| `T-03` | 의료 기록 무단 수정 및 전자서명 우회 | 높음 | `S1`, `S2` |
| `T-04` | Presigned URL 및 객체 경로 남용 | 높음 | `S1`, `S3` |
| `T-05` | 인증 토큰 재사용 및 Claim 불일치 | 높음 | `S1`, `S2`, `S3`, `S6` |
| `T-06` | 재식별 공격 | 높음 | `S3` |
| `T-07` | 비밀정보 및 자격증명 노출 | 높음 | `S1`, `S3`, `S6`, `S7` |
| `T-08` | KMS 권한 남용 | 높음 | `S1`, `S3`, `S6`, `S7` |
| `T-09` | Break-glass 남용 | 높음 | `S6`, `S7` |
| `T-10` | 감사 증적 상실 | 중간 | `S1`, `S2`, `S3`, `S6`, `S7` |
| `T-11` | `tenant_registry` 오염 및 온보딩 오류 | 높음 | `S4`, `S1`, `S2`, `S3` |
| `T-12` | 오프보딩 실패 및 잔존 접근 | 중간 | `S5`, `S3`, `S6` |
| `T-13` | 데이터베이스 장애 및 복구 시 격리 붕괴 | 높음 | `S7` |
| `T-14` | PHI 유출 사고 대응 실패 | 높음 | `S1`, `S2`, `S3`, `S6` |

### T-01 Cross-Tenant Access

- 한 테넌트의 사용자가 다른 테넌트의 PHI, 첨부파일, 연구 산출물에 접근하는 시나리오다.
- 주된 실패 지점은 잘못된 tenant claim 신뢰, `tenant_registry` 대신 사용자 입력으로 schema 또는 prefix를 해석하는 로직, DB 세션 컨텍스트 재설정 누락, 객체 경로 검증 누락이다.
- 이 위협이 성립하면 단일 버그가 곧바로 대규모 PHI 유출과 규제 위반으로 이어진다.
- 필수 통제는 claim 정합성 검증, `tenant_registry` allowlist 조회, 매 요청 단위 tenant context 재설정, RLS 적용, tenant prefix 강제, 음성 테스트 자동화다.

### T-02 동일 테넌트 내부 권한 상승

- 동일 병원 내부에서 간호사, 병원 관리자, 연구자, 승인자 등이 자신에게 허용되지 않은 진료 기록 작성·조회·승인 기능에 접근하는 시나리오다.
- 멀티테넌트 격리가 강하더라도 동일 테넌트 내부 인가가 느슨하면 의료 기록 무단 열람과 업무 분리 실패가 발생할 수 있다.
- 특히 병원 관리자는 사용자 관리와 통계 조회 권한을 갖지만, 개별 환자 차트 접근까지 기본 허용되어서는 안 된다.
- 필수 통제는 서버 측 role whitelist, patient-level relation 검증, 동일 테넌트 내부 ABAC 확장, 관리자 계정 최소화, 권한 변경 감사 기록이다.

### T-03 의료 기록 무단 수정 및 전자서명 우회

- 진료 기록 본문, 첨부파일 메타데이터, 서명 대상 payload가 무단으로 수정되거나, 기록 저장 후 검증 없이 소비되는 시나리오다.
- 의료 기록은 단순 조회 데이터가 아니라 법적 효력을 가지는 근거 자료이므로, 저장 시점뿐 아니라 조회와 반출 시점에도 무결성을 설명할 수 있어야 한다.
- 대표 실패 지점은 클라이언트가 만든 payload를 그대로 신뢰하는 경우, 서명과 검증 키의 권한 분리가 약한 경우, 서명 검증이 읽기 경로에서 생략되는 경우다.
- 필수 통제는 서버 측 canonicalization, 서명 대상 범위 고정, `Sign`과 `Verify` 권한 분리, 감사 가능한 수정 이력, 복구 후 무결성 점검이다.

### T-04 Presigned URL 및 객체 경로 남용

- 임상 첨부파일 업로드 또는 연구 데이터셋 다운로드 과정에서 presigned URL이 의도와 다르게 재사용되거나, 다른 tenant prefix로 바뀌어 사용되는 시나리오다.
- presigned URL은 편리하지만, 발급 범위와 수명이 넓으면 애플리케이션 인가를 우회하는 간접 경로가 된다.
- 대표 실패 지점은 object key를 클라이언트가 자유롭게 정하는 구조, 긴 만료시간, 승인 취소 이후에도 유효한 다운로드 URL, URL 공유에 대한 추적 부재다.
- 필수 통제는 서버 생성 key만 허용하는 구조, 짧은 TTL, 반출 대상 tenant·patient·dataset 상태 재검증, 다운로드 로그 수집, 만료 후 즉시 재발급 금지다.

### T-05 인증 토큰 재사용 및 Claim 불일치

- 계정 비활성화, 역할 변경, tenant 상태 변경 이후에도 기존 access token 또는 refresh token이 남아 계속 사용되는 시나리오다.
- claim 자체가 올바르더라도 group, role, tenant type, scope가 서로 일치하지 않으면 우회 경로가 생길 수 있다.
- 대표 실패 지점은 긴 토큰 수명, 역할 변경 후 세션 무효화 부재, authorizer 캐시 과신, Lambda 내부 재검증 누락이다.
- 필수 통제는 짧은 토큰 수명, 강제 세션 무효화 절차, claim coherence 검증, tenant 상태 기반 요청 차단, 계정 상태 변경에 대한 음성 테스트다.

### T-06 재식별 공격

- 연구용 데이터셋이 외부 공개 데이터 또는 반복 추출된 내부 데이터셋과 결합돼 특정 환자를 다시 식별할 수 있게 되는 시나리오다.
- 이 위협은 크로스 테넌트 접근과 달리 기술적 침입 없이도 발생할 수 있으며, 제공 범위와 데이터 품질이 높을수록 위험이 커진다.
- 대표 실패 지점은 준식별자 일반화 부족, 소수 집단 억제 기준 부재, 반복 추출 누적 위험 미검토, 연구 목적과 무관한 필드 제공이다.
- 필수 통제는 필드 allowlist, 목적 기반 승인, 최소 집계 기준, suppression 및 generalization 정책, dataset release review 기록이다.

### T-07 비밀정보 및 자격증명 노출

- DB 접속 정보, 운영 비밀정보, API 자격증명, 예외 접근용 비밀이 코드, 로그, 과도한 권한, 장기 보관으로 인해 노출되는 시나리오다.
- 한번 노출된 비밀정보는 애플리케이션 인가 로직을 우회해 데이터 평면에 직접 접근하는 데 사용될 수 있다.
- 대표 실패 지점은 secret 하드코딩, 지나치게 넓은 `GetSecretValue`, 디버그 로그에 비밀정보 출력, break-glass용 장기 자격증명 보관이다.
- 필수 통제는 Secrets Manager 중심 운영, secret별 최소 권한, 자동 회전, 로그 마스킹, 예외 비밀정보의 JIT 발급과 사용 후 폐기다.

### T-08 KMS 권한 남용

- 암호화 또는 서명용 KMS 키에 과도한 `Decrypt`, `GenerateDataKey`, `Sign`, `Verify` 권한이 부여되어 원본 PHI 복호화 또는 위조 서명이 가능한 시나리오다.
- 키 분리와 사용 목적 분리가 약하면, 원래는 일부 워크로드를 위한 권한이 다른 데이터 평면 전체로 확장될 수 있다.
- 대표 실패 지점은 연구 데이터셋 처리 경로가 임상 원본용 키까지 사용할 수 있는 구조, 운영자가 직접 decrypt 가능한 키 정책, sign과 verify가 뒤섞인 설계다.
- 필수 통제는 키 목적별 분리, key policy 최소화, 서비스별 grant 범위 제한, 키 사용 로그 검토, 운영자 기본 decrypt 금지 원칙이다.

### T-09 Break-glass 남용

- 비상 접근 경로가 실제로는 일상 운영 경로처럼 사용되거나, 승인·만료·세션 기록 없이 사용되는 시나리오다.
- 운영자 예외 접근은 정상 사용자보다 더 강한 통제를 받아야 하며, 느슨한 운영 편의 수단이 되어서는 안 된다.
- 대표 실패 지점은 2인 승인 부재, standing privilege, 세션과 작업 증적의 분리, 사후 검토 없는 반복 사용이다.
- 필수 통제는 승인 티켓 기반 JIT access, 짧은 만료 시간, 세션 로그와 DB 작업 증적 연계, 사용 후 회수, 월간 예외 접근 검토다.

### T-10 감사 증적 상실

- PHI 접근 또는 운영자 행위를 조사할 수 있을 정도의 로그와 증적 체인이 남지 않는 시나리오다.
- 직접적인 데이터 노출이 아니더라도, 사고 발생 후 영향 범위 산정과 규제 대응을 불가능하게 만든다는 점에서 운영 위험이 크다.
- 대표 실패 지점은 CloudTrail data event 미수집, 애플리케이션 감사 로그 부재, 보존 기간 부족, 상관관계 ID 부재, 세션 로그와 승인 기록 분리다.
- 필수 통제는 인프라 로그와 앱 로그의 병행 수집, 변경 불가능한 보존, 상관관계 ID, 정기 로그 복원 테스트, 사고 대응 체크리스트다.

### T-11 `tenant_registry` 오염 및 온보딩 오류

- 테넌트 매핑 정보가 잘못 생성되거나 무단으로 변경되어 schema, prefix, tenant 상태가 잘못 연결되는 시나리오다.
- 이 위협은 개별 요청 수준이 아니라 구조 전체의 격리를 흔들기 때문에, 데이터 저장소의 내용보다도 더 높은 영향도를 가질 수 있다.
- 대표 실패 지점은 수동 매핑 입력, 승인 없는 상태 변경, 중복 tenant ID, `ACTIVE`·`SUSPENDED`·`OFFBOARDED` 전이 관리 부재다.
- 필수 통제는 registry를 단일 source of truth로 관리하는 구조, 상태 전이 승인 절차, 온보딩 검증 체크리스트, 변경 감사 로그, 샘플 데이터 음성 테스트다.

### T-12 오프보딩 실패 및 잔존 접근

- 테넌트 계약 종료 이후에도 계정 세션, presigned URL, dataset artifact, 스냅샷, 백업 식별 정보가 남아 접근이 계속되는 시나리오다.
- 오프보딩은 단순 삭제 작업이 아니라, 접근 차단과 보존 정책 이행을 함께 증명해야 하는 라이프사이클 통제다.
- 대표 실패 지점은 tenant 상태만 바뀌고 세션 무효화가 누락된 경우, 배포 산출물이나 dataset 사본이 남는 경우, 보존 기간과 파기 시점이 불분명한 경우다.
- 필수 통제는 상태 전이와 세션 무효화의 순서 고정, URL와 export artifact 회수, 백업 보존 정책 문서화, 오프보딩 완료 증적 보관이다.

### T-13 데이터베이스 장애 및 복구 시 격리 붕괴

- 장애 조치, 스냅샷 복원, 리더 승격, 수동 복구 과정에서 RLS, schema binding, key access, 감사 로깅 같은 보안 속성이 일부 빠지는 시나리오다.
- 복구가 성공했다는 사실만으로 충분하지 않으며, 복구 이후에도 동일한 격리와 무결성이 유지되는지 확인해야 한다.
- 대표 실패 지점은 복구 후 보안 설정 점검 누락, 긴급 복구용 수동 계정 사용, 스냅샷 테스트에서 tenant context 검증 생략이다.
- 필수 통제는 복구 후 보안 검증 체크리스트, RLS 및 schema 검증 자동화, 복구 drill, 운영자 수동 접근 최소화, RTO/RPO와 별개인 보안 복구 목표 정의다.

### T-14 PHI 유출 사고 대응 실패

- PHI 유출 의심 상황에서 차단, 증적 보존, 영향 범위 산정, 통지, 재발방지가 늦어지거나 누락되는 시나리오다.
- 의료 데이터 사고는 기술적 차단뿐 아니라 규제상 통지와 재발방지 계획까지 요구하므로, 대응 실패 자체가 별도 위협으로 간주된다.
- 대표 실패 지점은 초기 15분 내 차단 실패, 60분 내 증적 확보 실패, tenant별 영향 범위 산정 불가, 관련 로그 산재, 커뮤니케이션 소유자 부재다.
- 필수 통제는 초기 차단 절차, 증적 보존 순서, 영향 범위 산정 기준, 통지 책임자 지정, 사고 후 개선 과제 추적이다.

## 위협별 대응 전략 {#mitigations}

위협별 대응 전략은 개별 통제를 나열하는 데서 끝나지 않아야 한다. 어떤 위협을 어떤 통제 묶음으로 줄일 것인지, 그리고 무엇을 증빙으로 확인할 것인지를 함께 정의해야 한다.

| 통제 영역 | 주요 내용 | 대응 위협 | 운영 증빙 예시 |
|----------|----------|----------|---------------|
| 인증 및 claim 정합성 | JWT 서명 검증, role·group·tenant type·scope 정합성 검증, 세션 무효화 | `T-01`, `T-02`, `T-05` | authorizer 설정, 음성 테스트 결과, 계정 비활성화 테스트 기록 |
| tenant mapping 거버넌스 | `tenant_registry` allowlist, 상태 전이 승인, 온보딩 검증 | `T-01`, `T-11`, `T-12` | 상태 변경 로그, 온보딩 체크리스트, 샘플 검증 결과 |
| 데이터 평면 격리 | DB session context 재설정, RLS, schema binding, tenant prefix 검증 | `T-01`, `T-03`, `T-04`, `T-13` | DB 정책 검토 기록, cross-tenant 음성 테스트, 객체 키 검증 로그 |
| 연구 데이터 제공 통제 | field allowlist, 목적 기반 승인, 가명처리, 재식별 위험 검토, short-lived delivery | `T-04`, `T-06`, `T-12` | 승인 기록, 데이터셋 명세, 재식별 위험 검토 결과, 다운로드 로그 |
| 비밀정보 및 키 관리 | secret 분리, 자동 회전, 최소 권한 IAM, 목적별 KMS 키 분리 | `T-07`, `T-08`, `T-09` | rotation 기록, key usage log, IAM 정책 검토 결과 |
| 운영자 예외 접근 통제 | 2인 승인, JIT access, 세션 로깅, 사후 검토 | `T-09`, `T-14` | 승인 티켓, Session Manager 로그, 사후 검토 보고서 |
| 감사 증적 체계 | 앱 감사 로그, CloudTrail, S3 data event, immutable retention, correlation ID | `T-10`, `T-14` | 로그 수집 정책, 보존 설정, 조사 리허설 결과 |
| 라이프사이클 및 복구 검증 | 오프보딩 절차, 스냅샷/복구 점검, 보안 속성 검증, DR drill | `T-12`, `T-13`, `T-14` | 오프보딩 완료 증적, 복구 drill 결과, 보안 체크리스트 |

### 배포 전 최소 확인 항목

- 크로스 테넌트 음성 테스트를 통과하지 못하면 배포를 진행하지 않는다.
- 병원 관리자, 간호사, 연구자, 운영자 계정에 대한 역할 경계 테스트가 없으면 배포를 진행하지 않는다.
- 연구 데이터셋 제공 경로에 가명처리 기준과 반출 승인 절차가 정의되지 않으면 외부 제공 기능을 활성화하지 않는다.
- break-glass 경로에 승인, 세션 로그, 권한 자동 회수가 연결되어 있지 않으면 운영 경로로 승인하지 않는다.
- PHI 접근 및 예외 접근에 대한 감사 증적을 한 번의 조사 절차로 연결할 수 없으면 운영 승인을 보류한다.

### 운영 검토 우선순위

1. 먼저 `T-01`부터 `T-06`까지의 데이터 평면 위협을 검토한다. 이는 직접적인 PHI 노출과 프라이버시 사고를 유발할 가능성이 가장 높다.
2. 다음으로 `T-07`부터 `T-10`까지의 비밀정보, 키, 예외 접근, 감사 증적 위협을 검토한다. 이 영역은 단일 사고의 규모를 확대하거나 조사 자체를 어렵게 만든다.
3. 마지막으로 `T-11`부터 `T-14`까지의 라이프사이클, 복구, 사고 대응 위협을 검토한다. 이 영역은 장기 운영 안정성과 규제 대응 가능성을 좌우한다.
