# ADR-002. 계정 분리 전략

## 상태

제안됨

## 컨텍스트

AWS 멀티 계정 전략은 보안 경계 강화, 폭발 반경 제한, 권한 분리를 위한 기본 설계 패턴입니다. 이 EHR SaaS 플랫폼에서 어떤 계정 구조를 채택할 것인지 결정해야 합니다.

<!-- 작성 예정: AWS Organizations, Control Tower, Landing Zone과의 연관성 -->

## 결정

<!-- 작성 예정: 선택된 계정 분리 전략 -->

**초안:** 관리 계정(보안 서비스 집중), 멤버 계정(애플리케이션 리소스), 선택적으로 보안 전용 계정 분리.

## 고려된 대안

| 대안 | 설명 |
|------|------|
| 단일 계정 | 모든 리소스를 하나의 AWS 계정에서 운영 |
| 2-계정 모델 | 관리 + 멤버 계정 분리 |
| 멀티 계정 (권장) | 관리, 멤버, 보안, 로그 아카이브 등 역할별 계정 분리 |

## 근거

<!-- 작성 예정: 근거 상세 -->

## 보안 함의

<!-- 작성 예정: 보안 함의 분석 -->

- 계정 간 역할(Cross-account IAM Role) 설계 필요
- 관리 계정의 보안 서비스가 멤버 계정을 모니터링하는 구조
- 멤버 계정 내 대응 IAM 역할 최소 권한 원칙 적용

## 컴플라이언스 매핑

| 요구사항 범주 | 관련 설계 결정 |
|--------------|--------------|
| 접근 제어 | 관리자와 애플리케이션 운영자 분리 |
| 감사 추적성 | 계정별 CloudTrail 활성화 및 중앙 집계 |

## 미해결 질문

- 테넌트별 계정 분리 필요 여부 (테넌트 수 증가 시)
- AWS Organizations SCP(서비스 제어 정책) 설계 범위
- 계정 간 IAM 역할의 최소 권한 경계 설정 방법
