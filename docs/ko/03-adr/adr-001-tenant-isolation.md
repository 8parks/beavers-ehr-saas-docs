# ADR-001. 테넌트 격리 전략

## 상태

제안됨

## 컨텍스트

멀티 테넌트 EHR SaaS 플랫폼에서 여러 병원(테넌트)이 동일한 AWS 인프라를 공유합니다. 한 테넌트의 사용자가 다른 테넌트의 PHI에 접근하는 것을 방지하는 것은 핵심 보안 요구사항입니다.

어떤 수준에서 테넌트 격리를 구현할 것인지 결정해야 합니다.

<!-- 작성 예정: 컨텍스트 상세, 규제 요구사항과의 연관성 -->

## 결정

<!-- 작성 예정: 선택된 테넌트 격리 접근법 -->

**초안:** 서비스 레이어 테넌트 검증 + Schema-per-Tenant 데이터베이스 격리 + Row-Level Security 심층 방어 조합을 채택합니다.

## 고려된 대안

| 대안 | 설명 | 트레이드오프 |
|------|------|------------|
| 완전 공유 테이블 | tenant_id 컬럼으로 모든 테넌트 데이터 공유 | 구현 단순, 격리 취약 |
| Schema-per-Tenant | 테넌트별 PostgreSQL 스키마 분리 | 격리 강화, 운영 복잡도 증가 |
| Database-per-Tenant | 테넌트별 독립 DB 인스턴스 | 최강 격리, 비용 매우 높음 |
| RLS 단독 | Row-Level Security만으로 격리 | 구현 단순, 애플리케이션 레이어 취약점 노출 |

## 보안 함의

<!-- 작성 예정: 보안 함의 분석 -->

- 서비스 레이어에서 JWT `tenant_id` 클레임 검증 필수
- 데이터베이스 쿼리는 반드시 테넌트 스키마 범위 내에서 실행
- 커넥션 풀링과 테넌트 컨텍스트 관리 간의 상호작용 주의

## 컴플라이언스 매핑

| 요구사항 범주 | 관련 설계 결정 |
|--------------|--------------|
| 접근 제어 | 테넌트 인식 인가, 스키마 격리 |
| 데이터 보호 | PHI 격리, 크로스 테넌트 접근 방지 |

## 미해결 질문

- 커넥션 풀링(RDS Proxy 등)과 schema-per-tenant 설계 간의 호환성
- 대규모 테넌트 확장 시 스키마 관리 자동화 방법
- 비상 접근(break-glass) 시나리오에서의 테넌트 격리 처리
