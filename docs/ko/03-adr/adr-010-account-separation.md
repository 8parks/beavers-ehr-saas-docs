# ADR-010. 관리 계정과 애플리케이션 계정 분리

## 상태

제안됨

## 컨텍스트

ADR-002에서 멀티 계정 전략을 개략적으로 결정하였습니다. 이 ADR은 관리(Administrator) 계정과 멤버(Member/Application) 계정의 역할 분리와 상호 운영 방식을 구체적으로 설계하는 데 초점을 맞춥니다.

<!-- 작성 예정: ADR-002와의 관계, 계정 간 신뢰 관계 설계 -->

## 결정

<!-- 작성 예정: 관리 계정 vs 멤버 계정 역할 분리 상세 -->

**초안:**
- 관리 계정: 보안 서비스(Config, Security Hub), 자동화 오케스트레이션, 중앙 로그 집계
- 멤버 계정: 애플리케이션 리소스, 대응 런북, 테넌트 데이터
- 계정 간 IAM 역할(Cross-account Role)로 관리 계정이 멤버 계정의 대응 역할을 위임받아 실행

## 고려된 대안

| 대안 | 설명 |
|------|------|
| 단일 계정 | 관리/운영 분리 없음 |
| 느슨한 계정 분리 | 동일 계정 내 IAM 역할로만 분리 |
| 엄격한 멀티 계정 분리 (권장) | 관리 계정 + 멤버 계정 + 로그 아카이브 계정 |

## 근거

<!-- 작성 예정: 근거 상세 -->

플랫폼 관리자가 애플리케이션 PHI에 직접 접근하지 않도록 계정 수준에서 경계를 설정합니다.

## 보안 함의

- 관리 계정의 보안 서비스가 멤버 계정을 모니터링하는 단방향 신뢰 관계 설계
- 멤버 계정의 대응 IAM 역할은 최소 필요 권한만 허용
- 계정 간 역할 위임 시 외부 ID(External ID) 또는 조건 키 사용 고려
- 관리 계정 자체의 보안 강화 (MFA, 루트 계정 잠금 등)

## 컴플라이언스 매핑

| 요구사항 범주 | 관련 설계 결정 |
|--------------|--------------|
| 접근 제어 | 관리자와 애플리케이션 운영자 계정 분리 |
| 감사 추적성 | 계정별 CloudTrail, 중앙 집계 |
| 최소 권한 원칙 | 계정 간 IAM 역할 최소 권한 |

## 미해결 질문

- AWS Organizations SCP로 멤버 계정에 적용할 제약 정책 설계
- 관리 계정과 멤버 계정 간 네트워크 연결 필요 여부
- 계정 수가 증가할 때(예: 테넌트별 계정) 자동화된 계정 온보딩 전략
