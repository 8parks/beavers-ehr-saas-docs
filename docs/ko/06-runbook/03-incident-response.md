# 3. 사고 대응

본 섹션은 위협 모델링의 고위험 시나리오를 실제 대응 절차로 변환한 것입니다.  
세 가지 절차 모두 공통으로 다음 순서를 따릅니다.

1. 탐지
2. 차단
3. 증적 보존
4. 영향 범위 산정
5. 복구
6. 사후 개선

## Cross-Tenant 접근

### 목적

한 테넌트 사용자가 다른 테넌트의 PHI, 첨부파일, 연구 산출물에 접근했거나 접근했을 가능성이 있는 상황을 즉시 차단합니다.

### 즉시 조치

- 관련 토큰과 세션을 무효화합니다.
- 필요 시 해당 tenant 또는 API 경로를 일시 중지합니다.
- presigned URL이 관련된 경우 즉시 폐기 또는 만료 처리합니다.

### 1시간 내 조치

1. 영향 tenant와 자산 범위를 1차 산정합니다.
2. 관련 요청 로그, 앱 감사 로그, 데이터 접근 로그를 보존합니다.
3. `tenant_registry`, tenant context, RLS, prefix 검증 상태를 역추적합니다.
4. 동일 경로에서 추가 시도가 있는지 확인합니다.

### 후속 조치

- 원인이 claim 검증, tenant mapping, DB context, object key 제약 중 어디에 있는지 분류합니다.
- 수정 전까지 동일한 유형의 기능 배포를 중단합니다.
- 재현 테스트와 음성 테스트를 보강합니다.

### 필수 증빙

- 사고 식별 시각
- 차단 시각
- 영향 tenant 및 자산 목록
- 로그 보존 위치
- 재현 및 수정 검증 결과

## PHI 유출

### 목적

PHI가 외부로 유출되었거나 유출 정황이 확인된 경우 추가 반출을 막고, 영향 범위와 통지 판단을 지원합니다.

### 15분 내 조치

1. 추가 반출 경로를 차단합니다.
2. 관련 세션과 토큰을 폐기합니다.
3. 필요 시 tenant를 `SUSPENDED` 상태로 전환합니다.
4. 문제 URL, export, dataset delivery 경로를 폐기합니다.

### 60분 내 조치

1. CloudTrail, 앱 감사 로그, SSM 세션 로그, KMS 사용 로그, S3 data event를 보존합니다.
2. 영향 tenant, 환자 수, 자산 유형을 1차 산정합니다.
3. 유출 경로를 분류합니다.
   - token 재사용
   - dataset 오제공
   - presigned URL 남용
   - break-glass 오남용
   - secret 노출
4. 통지 필요 여부를 판단할 수 있도록 사고 기록을 정리합니다.

### 사후 조치

- 통지 기준과 승인 절차에 따라 대내외 보고를 수행합니다.
- 원인 통제와 재발 방지 작업을 별도 추적합니다.
- tabletop 또는 사후 훈련에서 동일 시나리오를 재검증합니다.

### 필수 증빙

- 초기 차단 기록
- 증적 확보 체크리스트
- 영향 범위 산정 문서
- 통지 판단 기록
- 재발 방지 작업 추적표

## 자격증명 유출

### 목적

DB 비밀번호, secret, 운영자 예외 자격증명, 키 접근 권한이 유출되었을 때 데이터 평면 우회를 차단합니다.

### 즉시 조치

- 관련 자격증명 또는 권한을 비활성화합니다.
- 활성 세션과 연결된 임시 자격증명을 폐기합니다.
- 문제 경로를 사용하는 워크로드를 제한 또는 일시 중지합니다.

### 단계별 절차

1. 노출 유형을 분류합니다.
   - DB credential
   - 애플리케이션 secret
   - 운영자 예외 자격증명
   - 키 사용 권한
2. 비상 회전 절차를 실행합니다.
3. 노출 전후 사용 로그를 비교해 악용 여부를 판단합니다.
4. 해당 자격증명으로 접근 가능한 데이터 평면 범위를 산정합니다.
5. 회전 후 정상 기능을 검증합니다.

### 필수 증빙

- 노출 식별 기록
- 회전 수행 로그
- 사용 이력 비교 결과
- 영향 자산 목록
- 복구 검증 결과
