# 3. 사고 대응

본 섹션은 위협 모델링의 고위험 시나리오를 실제 대응 절차로 변환한 것입니다.  
세 가지 절차 모두 공통으로 다음 순서를 따릅니다.

1. 탐지
2. 차단
3. 증적 보존
4. 영향 범위 산정
5. 복구
6. 사후 개선

## 역할 및 에스컬레이션

| 역할 | 주요 책임 |
| --- | --- |
| 사고 대응 책임자 | 중요도 선언, 우선순위 조정, 최종 의사결정 |
| 보안 운영 팀 | 차단, 로그 보존, 조사 조정 |
| 서비스 오너 | 기능 영향 판단, 복구 승인, 수정 배포 조정 |
| 플랫폼 운영 담당자 | 계정/세션/네트워크/비밀정보 조치 실행 |
| 법무/컴플라이언스 담당자 | 통지 필요성 판단, 대외 보고 승인 |
| 고객 커뮤니케이션 담당자 | 테넌트 또는 외부 이해관계자 통지 수행 |

## 중요도 분류

| 중요도 | 정의 | 예시 | 기본 대응 기준 |
| --- | --- | --- | --- |
| 심각 | Cross-Tenant 접근 확인, PHI 대량 유출 의심, 데이터 평면 직접 노출 자격증명 유출 | 타 테넌트 환자정보 조회 확인, 데이터셋 오제공 확인 | 즉시 격리, 최고 우선순위, 승인자 실시간 호출 |
| 높음 | 영향 범위 미확정이지만 고위험 정황 존재 | presigned URL 대량 사용, 비정상 `Decrypt`, Break-glass 무승인 사용 | 1시간 내 범위 산정, 차단 우선 |
| 중간 | 시도 또는 경미한 노출 정황, 자동 탐지만 존재 | 실패한 토큰 재사용, 비활성 계정 접근 시도 | 영업시간 내 조사 착수, 재발 방지 조치 포함 |

## 공통 초기 대응 규칙

- 사고 기록을 생성하고, 첫 보고 시각과 보고자를 기록합니다.
- 차단 조치 전에 증적이 사라질 가능성이 높다면 최소한의 로그 보존부터 수행합니다.
- 재부팅, 임의 삭제, 정책 덮어쓰기처럼 증적을 훼손할 수 있는 작업은 승인 없이 수행하지 않습니다.
- PHI가 포함될 수 있는 원문 로그, 요청 본문, 파일 사본은 최소 인원만 접근하도록 분리합니다.
- 사고 대응 중 생성되는 임시 export, 스크린샷, 로그 묶음도 PHI 자산으로 취급합니다.

## 공통 증적 보존 체크리스트

| 항목 | 최소 조치 |
| --- | --- |
| 사고 식별 정보 | 사고 ID, 최초 탐지 시각, 최초 보고 채널 기록 |
| 로그 소스 | 애플리케이션 감사 로그, 운영 로그, 관리 평면 감사 로그, 객체 접근 로그 식별 |
| 체인 오브 커스터디 | 수집자, 수집 시각, 저장 위치, 전달 시각 기록 |
| 보존 조치 | 삭제/만료 방지, 보존 기간 설정, 접근 권한 축소 |
| 분석 사본 | 원본과 분리된 사본에서 분석 수행 |

로그 보존 순서는 사고 유형에 따라 달라질 수 있으나, 원칙적으로 관리 평면 감사 로그, 애플리케이션 감사 로그, 데이터 접근 로그, 운영 세션 로그 순으로 식별하고 보존합니다.

## Cross-Tenant 접근

### 목적

한 테넌트 사용자가 다른 테넌트의 PHI, 첨부파일, 연구 산출물에 접근했거나 접근했을 가능성이 있는 상황을 즉시 차단합니다.

### 즉시 조치

1. 관련 토큰과 세션을 무효화합니다.
2. 필요 시 해당 tenant 또는 API 경로를 일시 중지합니다.
3. presigned URL이 관련된 경우 즉시 폐기 또는 만료 처리합니다.
4. 사고 대응 책임자가 중요도를 선언하고, 조사 범위 확장을 승인합니다.

### 1시간 내 조치

1. 영향 tenant와 자산 범위를 1차 산정합니다.
2. 다음 로그 소스를 우선 보존합니다.
   - 애플리케이션 감사 로그
   - API 요청/응답 메타데이터
   - 객체 접근 로그 또는 데이터 이벤트 로그
   - 관리 평면 감사 로그
3. `tenant_registry`, tenant context, RLS, prefix 검증 상태를 역추적합니다.
4. 동일 경로에서 추가 시도가 있는지 확인합니다.
5. 확인된 접근이 읽기, 쓰기, 다운로드, export 중 무엇인지 구분합니다.
6. 영향 범위를 환자 수, 객체 수, 데이터셋 수 단위로 기록합니다.

### 24시간 내 조치

1. 재현 테스트와 음성 테스트를 수행해 실패 지점을 특정합니다.
2. 같은 결함 패턴이 다른 API 또는 배치 경로에도 있는지 확장 조사합니다.
3. 수정 적용 전까지 동일 유형 기능의 신규 배포를 중단합니다.
4. 영향 테넌트 통지 필요 여부를 내부 기준에 따라 판단합니다.

### 후속 조치

- 원인이 claim 검증, tenant mapping, DB context, object key 제약 중 어디에 있는지 분류합니다.
- 수정 전까지 동일한 유형의 기능 배포를 중단합니다.
- 재현 테스트와 음성 테스트를 보강합니다.
- 수정 후 24시간 이상 강화 모니터링을 유지합니다.

### 필수 증빙

- 사고 식별 시각
- 차단 시각
- 영향 tenant 및 자산 목록
- 로그 보존 위치
- 재현 및 수정 검증 결과
- 통지 또는 미통지 판단 기록

### 금지 사항

- 영향 범위를 확정하기 전에 로그 보존 없이 접근 경로만 차단하고 종료하지 않습니다.
- Cross-Tenant 쓰기 가능성이 있는 경우 데이터 무결성 검증을 생략하지 않습니다.

## PHI 유출

### 목적

PHI가 외부로 유출되었거나 유출 정황이 확인된 경우 추가 반출을 막고, 영향 범위와 통지 판단을 지원합니다.

### 15분 내 조치

1. 추가 반출 경로를 차단합니다.
2. 관련 세션과 토큰을 폐기합니다.
3. 필요 시 tenant를 `SUSPENDED` 상태로 전환합니다.
4. 문제 URL, export, dataset delivery 경로를 폐기합니다.
5. 사고 대응 책임자와 법무/컴플라이언스 담당자를 즉시 호출합니다.

### 60분 내 조치

1. CloudTrail, 앱 감사 로그, SSM 세션 로그, KMS 사용 로그, S3 data event를 보존합니다.
2. 영향 tenant, 환자 수, 자산 유형을 1차 산정합니다.
3. 유출 경로를 분류합니다.
   - token 재사용
   - dataset 오제공
   - presigned URL 남용
   - break-glass 오남용
   - secret 노출
4. 통지 필요 여부를 판단할 수 있도록 사고 기록을 정리합니다.

### 24시간 내 조치

1. 영향을 받은 자산을 환자 정보, 진료 기록, 첨부파일, 데이터셋으로 세분화합니다.
2. 유출이 읽기만 있었는지, 재가공 또는 재배포 가능성까지 있는지 판단합니다.
3. 통지 초안을 작성하되, 미확정 사실과 확정 사실을 구분합니다.
4. 추가 반출이 불가능하다는 복구 검증을 수행합니다.

### 사후 조치

- 통지 기준과 승인 절차에 따라 대내외 보고를 수행합니다.
- 원인 통제와 재발 방지 작업을 별도 추적합니다.
- tabletop 또는 사후 훈련에서 동일 시나리오를 재검증합니다.
- 사고 종결 전까지 관련 지표를 강화 모드로 모니터링합니다.

### 필수 증빙

- 초기 차단 기록
- 증적 확보 체크리스트
- 영향 범위 산정 문서
- 통지 판단 기록
- 재발 방지 작업 추적표
- 재반출 차단 검증 결과

### 금지 사항

- 영향 환자 수가 확정되지 않았다는 이유로 통지 판단 기록 자체를 남기지 않는 행위를 금지합니다.
- 조사 편의상 원문 PHI를 일반 티켓 시스템이나 공용 채널에 복제하지 않습니다.

## 자격증명 유출

### 목적

DB 비밀번호, secret, 운영자 예외 자격증명, 키 접근 권한이 유출되었을 때 데이터 평면 우회를 차단합니다.

### 즉시 조치

1. 관련 자격증명 또는 권한을 비활성화합니다.
2. 활성 세션과 연결된 임시 자격증명을 폐기합니다.
3. 문제 경로를 사용하는 워크로드를 제한 또는 일시 중지합니다.
4. 해당 자격증명이 데이터 평면, 관리 평면, 예외 경로 중 어디에 속하는지 분류합니다.

### 단계별 절차

1. 노출 유형을 분류합니다.
   - DB credential
   - 애플리케이션 secret
   - 운영자 예외 자격증명
   - 키 사용 권한
2. 비상 회전 절차를 실행합니다.
3. 다음 로그를 우선 수집합니다.
   - secret 조회 또는 변경 이력
   - 관리 평면 인증 이력
   - 데이터 접근 이력
   - 운영 세션 로그
4. 노출 전후 사용 로그를 비교해 악용 여부를 판단합니다.
5. 해당 자격증명으로 접근 가능한 데이터 평면 범위를 산정합니다.
6. 회전 후 정상 기능을 검증합니다.
7. 필요 시 관련 계정 또는 역할에 대한 추가 제한 정책을 임시 적용합니다.

### 필수 증빙

- 노출 식별 기록
- 회전 수행 로그
- 사용 이력 비교 결과
- 영향 자산 목록
- 복구 검증 결과

### 금지 사항

- 자격증명만 교체하고 그 자격증명으로 수행된 과거 접근 범위를 산정하지 않는 행위를 금지합니다.
- 회전 이후 정상 동작 검증 없이 서비스 재개를 선언하지 않습니다.
