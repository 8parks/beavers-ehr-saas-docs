# 사고 대응

이 장은 위협 모델링의 고위험 시나리오를 실제 운영 절차로 바꾼 문서입니다. 사고 대응의 목적은 단순히 서비스를 다시 살리는 것이 아니라, 추가 피해를 막고, 영향을 설명 가능한 수준으로 산정하고, 사후 재발 방지까지 연결하는 데 있습니다.

사고 유형이 다르더라도 대응 흐름은 크게 다르지 않습니다. 먼저 탐지하고, 차단하고, 기록을 보존한 뒤, 영향 범위를 계산하고, 통제를 복구하고, 마지막으로 재발 방지 작업을 남깁니다.

## 역할 및 에스컬레이션 {#roles-and-escalation}

| 역할 | 주요 책임 |
| --- | --- |
| 사고 대응 책임자 | 중요도 선언, 대응 우선순위 조정, 종료 승인 |
| 보안 운영 팀 | 차단 조치, 로그 보존, 조사 조정 |
| 서비스 오너 | 기능 영향 판단, 복구 시점 승인, 수정 배포 조정 |
| 플랫폼 운영 담당자 | 계정, 세션, 네트워크, 비밀정보 조치 실행 |
| 법무/컴플라이언스 담당자 | 통지 필요성 판단, 대외 보고 승인 |
| 고객 커뮤니케이션 담당자 | 고객·이해관계자 통지 수행 |

사고가 `심각` 또는 `높음`으로 분류되면 사고 대응 책임자와 서비스 오너는 실시간 협의를 유지해야 합니다. PHI 통지 판단이 필요한 경우 법무/컴플라이언스 담당자가 빠지는 상태에서 사고를 종결하지 않습니다.

## 중요도 분류 {#severity-classification}

| 중요도 | 정의 | 예시 | 기본 대응 수준 |
| --- | --- | --- | --- |
| 심각 | Cross-Tenant 접근 확인, PHI 대량 유출 정황, 데이터 평면 직접 노출 자격증명 유출 | 타 테넌트 환자 정보 열람 확인, 잘못된 데이터셋 제공 확인 | 즉시 격리, 실시간 승인 체계 호출 |
| 높음 | 영향 범위는 미확정이나 고위험 징후 존재 | 대량 데이터셋 다운로드 시도, 비정상 `Decrypt`, 무승인 Break-glass 사용 | 1시간 내 범위 산정, 차단 우선 |
| 중간 | 시도 또는 제한된 노출 정황, 자동 탐지만 존재 | 실패한 토큰 재사용, 비활성 계정 접근 시도 | 영업시간 내 조사 착수, 재발 방지 포함 |

중요도는 탐지 시점에 고정하지 않습니다. 조사 중 영향 범위가 커지거나 데이터 성격이 변경되면 즉시 상향 조정합니다.

## 초기 대응 원칙 {#initial-response-rules}

초기 대응은 “무엇을 먼저 끌 것인가”보다 “무엇을 먼저 보존할 것인가”를 함께 판단해야 합니다.

- 사고 기록을 생성하고, 최초 보고 시각과 보고자를 남깁니다.
- 차단 전에 기록이 사라질 위험이 크면 최소한의 로그 보존부터 수행합니다.
- 재부팅, 임의 삭제, 정책 덮어쓰기처럼 조사 기록을 훼손할 수 있는 조치는 승인 없이 수행하지 않습니다.
- PHI가 포함될 수 있는 로그, 요청 본문, 파일 사본은 조사 인력만 접근할 수 있는 별도 위치로 분리합니다.
- 조사 중 만들어진 스크린샷, export, 로그 묶음도 PHI 자산으로 취급합니다.

초기 대응의 목표는 완벽한 분석이 아니라, 추가 피해를 막을 수 있을 만큼 빠르게 상황을 안정시키는 데 있습니다.

## 기록 확보 및 보존 {#evidence-preservation-checklist}

기록은 많이 수집하는 것보다, 나중에 법적·운영적으로 설명 가능한 상태로 남기는 것이 중요합니다.

| 항목 | 최소 요구사항 |
| --- | --- |
| 사고 식별 정보 | 사고 ID, 최초 탐지 시각, 최초 보고 채널 기록 |
| 로그 소스 | 애플리케이션 감사 로그, 운영 로그, 관리 평면 감사 로그, 객체 접근 로그 식별 |
| 수집 이력 | 수집자, 수집 시각, 저장 위치, 전달 시각 기록 |
| 보존 조치 | 삭제 방지, 만료 연장, 접근 권한 축소 |
| 분석 사본 | 원본과 분리된 사본에서 분석 수행 |

로그 보존의 우선순위는 사고 유형에 따라 달라질 수 있으나, 원칙적으로 다음 순서를 따릅니다.

1. 관리 평면 감사 로그
2. 애플리케이션 감사 로그
3. 데이터 접근 로그
4. 운영 세션 로그

운영자는 조사 편의를 위해 원본 로그를 직접 수정하거나 일반 티켓 시스템으로 복사하지 않습니다.

## 복구 선언 기준 {#recovery-declaration-baseline}

복구 선언은 서비스 응답이 돌아왔다는 의미가 아닙니다. 우회 경로가 차단되었고, 동일 유형 재발 시도도 막힌 상태를 운영적으로 확인하는 절차입니다.

| 확인 항목 | 최소 기준 |
| --- | --- |
| 원인 통제 | 사고 원인 또는 우회 경로 차단 완료 |
| 기능 검증 | 핵심 사용자 흐름 정상 동작 확인 |
| 음성 테스트 | 동일 유형 재발 시도 차단 확인 |
| 기록 보존 | 조사에 필요한 로그와 기록 보존 완료 |
| 승인 | 사고 대응 책임자와 서비스 오너 승인 완료 |
| 후속 작업 | 미해결 위험과 재발 방지 작업 등록 완료 |

PHI 통지 판단이 필요한 사고는 법무/컴플라이언스 검토가 끝난 뒤에만 종결할 수 있습니다.

## Cross-Tenant 접근 {#cross-tenant-access}

### 대응 목표

한 테넌트 사용자가 다른 테넌트의 환자 정보, 첨부파일, 연구 산출물에 접근했거나 그 가능성이 확인된 경우, 추가 접근을 즉시 차단하고 손상 범위를 데이터 유형별로 재구성합니다.

### 초기 차단

1. 관련 토큰과 세션을 무효화합니다.
2. 필요 시 해당 API 경로나 tenant 상태를 일시 중지합니다.
3. 데이터셋 전달 경로나 객체 접근 권한이 관련된 경우 즉시 차단합니다.
4. 사고 대응 책임자가 중요도를 선언하고 조사 범위를 확정합니다.

### 조사와 영향 산정

사고 발생 후 1시간 내에는 어떤 데이터가 어떤 경로로 노출되었는지 1차 범위를 계산해야 합니다.

- 영향 tenant 식별
- 환자 수, 객체 수, 데이터셋 수 집계
- 읽기, 쓰기, 다운로드, export 중 영향 유형 분류
- `tenant_registry`, tenant context, RLS, object prefix 상태 역추적
- 같은 결함 패턴이 다른 API와 배치 경로에도 있는지 확인

### 수정과 재개

24시간 내에는 실패 지점을 특정하고, 동일 유형 기능의 신규 배포를 중단한 상태에서 수정 방향을 확정해야 합니다. 수정 후에는 같은 요청을 재현하는 시험과 타 테넌트 접근을 차단하는 음성 테스트를 함께 수행합니다.

### 종료 기록

- 사고 식별 시각
- 차단 시각
- 영향 tenant와 자산 목록
- 로그 보존 위치
- 재현 시험과 수정 검증 결과
- 통지 또는 미통지 판단 기록

### 금지 사항

- 영향 범위를 계산하기 전에 경로만 닫고 종료하지 않습니다.
- 쓰기 가능성이 확인된 사고에서 데이터 무결성 검증을 생략하지 않습니다.

## PHI 유출 {#phi-disclosure}

### 대응 목표

PHI가 외부로 반출되었거나 반출 정황이 있는 경우, 추가 반출을 차단하고 통지 판단에 필요한 사실관계를 빠르게 정리합니다.

### 초기 15분

이 단계의 목표는 반출 경로를 닫는 것입니다.

1. 문제 URL, export, dataset delivery 경로를 폐기합니다.
2. 관련 세션과 토큰을 무효화합니다.
3. 필요 시 tenant를 `SUSPENDED`로 전환합니다.
4. 사고 대응 책임자와 법무/컴플라이언스 담당자를 즉시 호출합니다.

### 초기 60분

이 단계에서는 통지 여부를 판단할 수 있을 만큼의 기초 자료를 모읍니다.

- CloudTrail
- 애플리케이션 감사 로그
- SSM 세션 로그
- KMS 사용 로그
- S3 data event

동시에 영향 tenant, 환자 수, 자산 유형을 1차 산정하고, 유출 경로가 token 재사용인지, 데이터셋 오제공인지, 전달 경로 오남용인지 분류합니다.

### 24시간 이내

영향 자산을 환자 정보, 진료 기록, 첨부파일, 데이터셋으로 세분화하고, 단순 열람인지 재가공 또는 재배포 가능성까지 있었는지 판단합니다. 통지 초안은 이 시점부터 준비하되, 확인된 사실과 추정 상태를 섞지 않습니다.

### 종료 기록

- 초기 차단 기록
- 기록 확보 내역
- 영향 범위 산정 문서
- 통지 판단 기록
- 재발 방지 작업 추적표
- 재반출 차단 검증 결과

### 금지 사항

- 영향 환자 수가 미확정이라는 이유로 통지 판단 기록 자체를 남기지 않는 행위를 금지합니다.
- 원문 PHI를 일반 티켓 시스템이나 공용 채널로 복제하지 않습니다.

## 자격증명 유출 {#credential-exposure}

### 대응 목표

DB 비밀번호, secret, 예외 접근용 자격증명, 키 사용 권한이 노출된 경우, 해당 자격증명으로 우회 가능한 데이터 평면과 관리 평면 경로를 차단하고 사용 범위를 재구성합니다.

### 초기 차단

1. 관련 자격증명 또는 권한을 비활성화합니다.
2. 활성 세션과 임시 자격증명을 폐기합니다.
3. 문제 경로를 사용하는 워크로드를 제한하거나 일시 중지합니다.
4. 자격증명이 데이터 평면, 관리 평면, 예외 경로 중 어디에 속하는지 분류합니다.

### 조사와 회전

비상 회전은 이 절차와 연결되어야 합니다. 운영자는 다음 자료를 우선 수집합니다.

- secret 조회 또는 변경 이력
- 관리 평면 인증 이력
- 데이터 접근 이력
- 운영 세션 로그

그 뒤 노출 전후 사용 로그를 비교해 악용 여부를 판단하고, 해당 자격증명으로 접근 가능한 자산 범위를 계산합니다. 회전 이후에는 읽기, 쓰기, 배치, 운영 경로가 모두 정상 동작하는지 따로 검증해야 합니다.

### 종료 기록

- 노출 식별 기록
- 회전 수행 로그
- 사용 이력 비교 결과
- 영향 자산 목록
- 복구 검증 결과

### 금지 사항

- 자격증명만 교체하고 과거 접근 범위 산정을 생략하지 않습니다.
- 기능이 살아 있다는 이유만으로 검증 없이 서비스 재개를 선언하지 않습니다.
