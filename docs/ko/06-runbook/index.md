# 운영 가이드 (Runbook)

설계 문서와 위협 모델링은 통제가 왜 필요한지를 설명합니다. 운영 가이드는 그 통제를 누가, 어떤 순서로, 어떤 기록을 남기며 유지할지를 정의합니다. 이 장의 목적은 운영 작업을 짧은 점검 항목으로 축약하는 것이 아니라, 멀티테넌트 EHR SaaS를 실제로 운영할 때 반복 사용 가능한 절차와 판단 기준을 제시하는 데 있습니다.

::: warning
이 문서는 가상의 EHR SaaS를 기준으로 작성한 운영 가이드입니다. 실제 환경에 적용할 때는 병원별 계약, 법정 보존 의무, 통지 체계, 조직의 승인 구조를 반영해 역할명과 세부 절차를 보완해야 합니다.
:::

## 문서의 역할

이 장은 다음과 같은 상황에서 참조하는 기준서입니다.

- 신규 병원이나 연구기관을 서비스에 등록할 때
- 계약 종료 또는 보안 사고로 테넌트 상태를 변경할 때
- 비밀정보, 키, 예외 접근 경로를 회전하거나 폐기할 때
- Cross-Tenant 접근, PHI 유출, 자격증명 노출과 같은 사고가 발생했을 때
- 운영 드리프트, 로그 누락, 권한 누적, 잔존 자산을 정기적으로 점검할 때

운영자는 이 문서를 읽고 바로 작업에 들어가는 것이 아니라, 작업의 시작 조건과 종료 기준을 먼저 확인한 뒤 필요한 승인과 기록 준비를 마친 상태에서 절차를 수행해야 합니다.

## 이 문서를 사용하는 방식

모든 민감 작업은 같은 흐름을 따릅니다.

1. 작업 또는 사고를 식별하고, 티켓과 식별자를 발급합니다.
2. 요청자, 승인자, 실행자, 검증자를 분리해 지정합니다.
3. 대상 자산, 영향 범위, 중단 가능성, 되돌림 경로를 확인합니다.
4. 실행 전에 보존이 필요한 로그와 기록을 먼저 확보합니다.
5. 절차를 수행한 뒤 기능 검증과 음성 테스트를 함께 실행합니다.
6. 결과, 미해결 위험, 후속 조치를 운영 기록에 남기고 종료합니다.

작업의 종료 시점은 명령이 끝난 시점이 아니라, 통제 상태가 다시 설명 가능해진 시점입니다. 예를 들어 계정을 비활성화했더라도 세션, 데이터셋 전달 권한, 배치 작업, 데이터셋 경로가 살아 있으면 작업은 끝난 것으로 보지 않습니다.

## 역할 및 승인 체계

| 작업 유형 | 요청 주체 | 승인 주체 | 실행 주체 | 종료 확인 |
| --- | --- | --- | --- | --- |
| 테넌트 생성, 상태 변경 | 영업/운영 담당, 서비스 오너 | 운영 승인자, 보안 승인자 | 플랫폼 운영 담당 | 보안 운영 팀 |
| 데이터 반환, 파기, 보존 전환 | 고객 담당, 서비스 오너 | 계약 책임자, 보안 승인자 | 플랫폼 운영 담당 | 보안 운영 팀, 감사 담당 |
| 자격증명 및 키 회전 | 서비스 오너, 보안 운영 팀 | 운영 승인자, 보안 승인자 | 플랫폼 운영 담당 | 서비스 오너, 보안 운영 팀 |
| Break-glass 접근 | 장애 대응 책임자 | 2인 승인 체계, 사후 검토 책임자 | 승인된 운영자 | 보안 운영 팀, 감사 담당 |
| 사고 통지 및 외부 보고 | 사고 대응 책임자 | 법무/컴플라이언스 승인자, 경영 승인자 | 대외 커뮤니케이션 담당 | 사고 대응 책임자 |

조직 규모가 작아서 한 사람이 여러 역할을 맡더라도, 승인과 종료 확인까지 같은 사람이 수행하는 구조는 허용하지 않습니다. 운영 효율보다 직무 분리 원칙을 우선합니다.

## 운영 절차 설계 기준

이 장의 세부 절차는 다음 기준을 전제로 작성합니다.

- 운영 절차는 평상시 경로와 예외 경로를 구분해야 합니다.
- PHI, 연구 데이터셋, 테넌트 매핑 정보, 키와 비밀정보는 서로 다른 자산군으로 다뤄야 합니다.
- 수동 작업이 포함되더라도 사후에 작업 맥락을 재구성할 수 있어야 합니다.
- 모든 민감 작업은 요청, 승인, 실행, 검증, 기록 보존의 다섯 단계를 빠뜨리지 않아야 합니다.
- 절차는 위협 모델링의 `T-01`, `T-07`, `T-09`, `T-12`, `T-14`와 충돌하지 않아야 합니다.

실제 운영에서는 서비스 연속성과 기록 보존이 충돌하는 상황이 생깁니다. 이 경우 우선순위는 사고 유형에 따라 달라질 수 있지만, 조사 기록을 훼손하는 조치는 반드시 명시적인 승인 아래에서만 수행해야 합니다.

## 운영 기록 관리

운영 가이드는 실행 자체보다 기록 관리가 중요합니다. 동일한 작업을 두 달 뒤 다시 검토했을 때, 누가 어떤 판단을 했고 어떤 범위에 영향을 주었는지 설명할 수 있어야 합니다.

| 기록 유형 | 반드시 남겨야 하는 내용 | 저장 원칙 |
| --- | --- | --- |
| 작업 티켓 | 식별자, 요청 사유, 대상 자산, 영향 범위, 승인 이력 | 운영 티켓 시스템 또는 변경 관리 시스템 |
| 감사 기록 | 로그 원본 위치, 수집 시각, 수집자, 전달 경로 | 무결성 검증 가능한 저장소 또는 수정 불가 저장소 |
| 검증 결과 | 양성 테스트, 음성 테스트, 실패 항목, 되돌림 여부 | 티켓과 상호 참조 가능한 저장소 |
| 예외 승인 기록 | 예외 사유, 승인자, 만료 시점, 후속 검토 일정 | 예외 관리 등록부 |
| 종료 기록 | 종료 판단 시각, 종료 승인자, 미해결 위험, 후속 조치 | 운영 보고서 또는 사고 보고서 |

기록 보존은 사고 대응에서만 요구되는 절차가 아닙니다. 오프보딩, 키 회전, Break-glass, 데이터 반환처럼 사후 분쟁 가능성이 있는 작업에도 같은 수준의 기록 관리가 필요합니다.

## 변경 관리와 예외 승인

운영 절차가 실제 환경에서 흔들리는 지점은 대개 예외입니다. 긴급 장애, 외부 규제 대응, 레거시 연동 문제처럼 즉시 표준 절차를 따르기 어려운 경우에도 예외는 기록 가능한 형태로 남아야 합니다.

- 예외는 구두 승인으로 종료하지 않습니다.
- 예외 승인에는 사유, 만료 시점, 대체 통제, 후속 검토 일정이 포함되어야 합니다.
- 표준 절차를 생략한 경우에는 생략한 절차와 그로 인한 위험을 명시해야 합니다.
- 예외가 반복되면 운영 절차가 아니라 설계 또는 조직 체계의 문제로 보고 재검토해야 합니다.
