# 운영 가이드 (Runbook)

설계와 위협 모델링이 정의한 통제는 실제 운영 절차로 내려오지 않으면 유지되지 않습니다. 본 런북은 멀티테넌트 EHR SaaS 환경에서 반복적으로 수행되는 운영 작업과 사고 대응 절차를 표준화하기 위한 문서입니다.  
본 문서는 기술 팀, 보안 운영 팀, 테넌트 운영 담당자, 승인권자가 공통으로 참조하는 실행 기준으로 사용합니다.

::: warning
본 런북은 가상의 EHR SaaS를 기준으로 작성된 운영 가이드입니다. 실제 운영에 적용할 때는 병원별 계약, 보존 의무, 통지 의무, 조직 승인 체계에 맞춰 세부 절차를 보완해야 합니다.
:::

## 목적

본 섹션은 운영 절차를 단순 체크리스트가 아니라 배포 이후에도 지속적으로 유지되어야 하는 통제 기준으로 정의합니다. 따라서 각 절차는 요청, 승인, 실행, 검증, 종료, 증빙 보존의 순서를 따르며, 예외 경로를 포함해 사후 재구성이 가능해야 합니다.

## 적용 원칙

본 런북의 절차는 다음 원칙을 전제로 적용합니다.

- 민감 작업은 요청자, 승인자, 실행자가 동일인이 되지 않도록 설계합니다.
- 서비스 연속성과 증적 보존이 충돌할 때는 사고 유형별 사전 정의된 우선순위에 따라 판단합니다.
- 운영 절차는 애플리케이션 로그, 감사 로그, 관리 평면 로그를 함께 사용해 재구성 가능해야 합니다.
- 로그, 키, 자격증명, 백업, 데이터셋 아티팩트는 각각 별도 자산으로 취급합니다.
- 작업 종료 기준은 “명령 수행”이 아니라 “통제 상태 검증 완료”로 정의합니다.

## 역할 및 승인 체계

| 작업 유형 | 요청 권한 | 승인 권한 | 실행 책임 | 사후 검증 |
| --- | --- | --- | --- | --- |
| Tenant 생성 및 상태 변경 | 영업/운영 담당자, 서비스 오너 | 운영 승인자, 보안 승인자 | 플랫폼 운영 담당자 | 보안 운영 팀 |
| 데이터 반환 및 파기 | 고객 담당자, 서비스 오너 | 계약 책임자, 보안 승인자 | 플랫폼 운영 담당자 | 보안 운영 팀, 감사 담당자 |
| 자격증명 및 키 회전 | 서비스 오너, 보안 운영 팀 | 운영 승인자, 보안 승인자 | 플랫폼 운영 담당자 | 서비스 오너, 보안 운영 팀 |
| Break-glass 접근 | 장애 대응 책임자 | 2인 승인 체계, 사후 검토 책임자 | 승인된 운영자 | 보안 운영 팀, 감사 담당자 |
| 사고 통지 및 외부 보고 | 사고 대응 책임자 | 법무/컴플라이언스 승인자, 경영 승인자 | 지정 대외 커뮤니케이션 담당자 | 사고 대응 책임자 |

위 표의 역할 명칭은 조직별로 다를 수 있으나, 직무 분리 원칙은 변경하지 않습니다. 소규모 조직에서 동일 인력이 복수 역할을 겸하는 경우에도 승인과 사후 검증은 독립적으로 남아야 합니다.

## 운영 원칙

- 평상시 운영 경로와 예외 경로를 구분합니다.
- PHI, 연구 데이터셋, 테넌트 매핑 정보는 동일한 운영 절차로 취급하지 않습니다.
- 모든 민감 작업은 요청, 승인, 실행, 검증, 종료, 증빙 보존의 순서를 따릅니다.
- 운영 절차는 위협 모델링의 `T-01`, `T-07`, `T-09`, `T-12`, `T-14` 통제와 일관되어야 합니다.
- 수동 작업이 필요한 경우에도 작업 결과를 사후에 재구성할 수 있는 수준의 증빙을 남겨야 합니다.

## 공통 실행 순서

모든 운영 절차는 다음 순서를 기본으로 따릅니다.

1. 작업 티켓 또는 사고 기록을 생성합니다.
2. 변경 창구, 승인자, 실행자, 검증자를 명시합니다.
3. 대상 자산, 영향 범위, 복구 또는 백아웃 계획을 확정합니다.
4. 실행 전 증적 보존이 필요한 자산을 먼저 확보합니다.
5. 통제를 실행하고, 성공 여부를 기능 검증과 음성 테스트로 확인합니다.
6. 로그와 증빙을 보존하고, 후속 조치 또는 미해결 리스크를 기록합니다.

## 증적 보존 기본 요건

| 항목 | 최소 요구사항 |
| --- | --- |
| 사고/작업 식별자 | 고유 ID 부여, 관련 티켓과 상호 참조 유지 |
| 시간 정보 | UTC 또는 조직 표준 시간대 사용, 시작/종료/핸드오프 시각 기록 |
| 작업 주체 | 요청자, 승인자, 실행자, 검증자 식별 가능 상태 유지 |
| 원본 로그 | 수정 불가 저장소 또는 무결성 검증 가능한 저장소에 보존 |
| 체인 오브 커스터디 | 증적 수집자, 전달자, 저장 위치, 전달 시각 기록 |
| 보존 기간 | 법규, 계약, 조사 목적에 맞는 최소 보존 기간 정의 |

증적 보존은 사고 대응에서만 필요한 절차가 아닙니다. 오프보딩, 키 회전, Break-glass, 데이터 반환처럼 사후 분쟁 가능성이 있는 작업에도 동일 기준을 적용합니다.
