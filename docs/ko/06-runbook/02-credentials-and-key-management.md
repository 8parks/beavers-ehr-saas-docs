# 2. 자격증명 및 키 관리

본 섹션은 비밀정보와 키를 설정값이 아니라 운영 자산으로 다룹니다.  
정기 회전과 비상 회전 절차는 분리해야 하며, 회전 이후에도 서비스 기능과 검증 기능이 유지되어야 합니다.

## 공통 원칙

- 자격증명 회전과 암호화 키 회전은 동일 절차로 취급하지 않습니다.
- 회전 계획에는 백아웃 기준과 기능 검증 기준이 포함되어 있어야 합니다.
- 키 또는 secret을 교체한 후에는 구 버전의 사용 중단과 참조 제거를 별도 검증합니다.
- 회전 이벤트는 서비스 중단 방지보다 장기 노출 위험 감소를 우선하되, 안전한 전환 절차를 포함해야 합니다.
- 비상 회전은 정기 회전의 단축판이 아니라, 악용 차단과 증적 보존을 함께 수행하는 별도 사고 절차입니다.

## 역할 분리

| 역할 | 책임 |
| --- | --- |
| 요청자 | 회전 필요성 제기, 영향 범위 제출 |
| 승인자 | 회전 시점, 서비스 영향, 예외 승인 검토 |
| 실행자 | 자격증명 또는 키 전환 수행 |
| 검증자 | 기능 검증, 구 버전 사용 종료, 로그 점검 |

## 키 교체

### 목적

정기적인 키 및 자격증명 회전을 통해 장기 노출 위험을 낮추고, 최소 권한 상태를 유지합니다.

### 적용 대상

- DB 접속 자격증명
- 애플리케이션 secret
- 연구 데이터셋 처리 경로 secret
- 서명 검증에 사용되는 키 참조 정보
- KMS customer managed key의 회전 정책

### 회전 유형 구분

| 회전 유형 | 주요 대상 | 핵심 검증 포인트 |
| --- | --- | --- |
| 자격증명 회전 | DB 비밀번호, 애플리케이션 secret, 운영용 secret | 애플리케이션 재연결 성공, 구 자격증명 차단 확인 |
| 암호화 키 회전 | 저장 데이터 보호용 KMS 키 참조 | 신규 암호화 성공, 기존 데이터 복호화 유지 확인 |
| 서명 키 회전 | 진료 기록 서명 또는 검증 체계 | 신규 서명 생성 성공, 기존 서명 검증 유지 확인 |

회전 유형별로 실패 방식이 다르므로, 하나의 변경 창구에서 실행하더라도 검증 기준은 분리해 관리해야 합니다.

### 절차

1. 회전 대상과 회전 주기를 확인합니다.
2. 영향 워크로드를 식별합니다.
   - 환자 조회
   - 진료 기록 조회/작성
   - 데이터셋 생성
   - 운영자 예외 접근
3. 회전 전 백아웃 계획과 검증 계획을 준비합니다.
4. 새로운 자격증명 또는 키 버전을 준비합니다.
5. 전환 순서를 결정합니다.
   - 읽기 경로와 쓰기 경로의 동시 영향 여부 확인
   - 롤링 전환 가능 여부 확인
   - 서비스 중단 창구 필요 여부 확인
6. 애플리케이션이 새 값으로 정상 동작하는지 단계적으로 검증합니다.
7. 구 버전 사용을 중단합니다.
8. 구 버전 참조가 남아 있지 않은지 로그와 설정을 다시 점검합니다.
9. 회전 후 일정 기간 동안 비정상 접근, 오류율 증가, 검증 실패가 없는지 모니터링합니다.

### 자격증명 회전 추가 요구사항

- 구 자격증명의 동시 유효 기간은 최소화합니다.
- 회전 직후 실패한 연결 시도가 구 자격증명을 계속 재사용하지 않는지 확인합니다.
- CI/CD, 배치 작업, 운영 스크립트 등 비대화형 경로까지 모두 새 자격증명을 사용해야 합니다.

### 암호화 키 회전 추가 요구사항

- 신규로 생성되는 데이터부터 새 키 참조를 사용해야 합니다.
- 기존 데이터의 복호화 가능성은 유지되어야 합니다.
- 키 정책 또는 권한 변경이 회전과 동시에 일어나는 경우, 기능 검증과 권한 검증을 분리해 수행합니다.

### 서명 키 회전 시 추가 원칙

- 새로 생성되는 진료 기록은 새 서명 체계를 사용합니다.
- 기존 기록의 검증 가능성은 유지되어야 합니다.
- 회전 직후 일정 기간은 구 버전 검증 정보가 남아 있어야 합니다.
- 검증 실패가 발생할 경우 구 키 복원 여부가 아니라 검증 체계 손상 여부를 우선 확인합니다.

### 필수 증빙

- 회전 계획서
- 영향 분석 기록
- 회전 수행 로그
- 회전 후 기능 검증 결과
- 구 버전 폐기 또는 비활성화 기록

### 완료 기준

- 회전 이후 모든 워크로드가 정상 동작합니다.
- 구 버전 자격증명 또는 키 사용이 종료되었습니다.
- 예외적으로 남겨둔 구 버전이 있다면 사유와 만료 시점이 명시되어 있습니다.
- 기능 검증, 권한 검증, 로그 점검이 모두 완료되었습니다.

## 비상 회전 절차

### 목적

유출이 의심되거나 실제 노출이 확인된 비밀정보와 키를 즉시 교체하고 추가 악용을 차단합니다.

### 트리거

- secret 노출 스캔 탐지
- 로그 또는 코드 저장소 내 비밀정보 유출
- 운영자 계정 탈취 의심
- 비정상적인 KMS 사용 탐지

### 초기 분류 기준

| 분류 | 예시 | 기본 대응 수준 |
| --- | --- | --- |
| 데이터 평면 직접 노출 | DB 자격증명, 데이터셋 처리 secret 노출 | 즉시 차단 및 IR 최고 우선순위 적용 |
| 관리 평면 노출 | 운영용 secret, 자동화 계정 자격증명 노출 | 즉시 차단, 관련 권한 범위 확인 |
| 키 사용 권한 오남용 | 비정상 `Decrypt`, 비정상 서명 사용 | 권한 차단, 로그 보존, 영향 범위 재산정 |

AWS KMS의 키 material은 일반적으로 직접 추출되지 않으므로, 많은 경우 실제 위험은 키 자체의 유출보다 키 사용 권한의 오남용에 있습니다. 따라서 비상 회전은 키 material 노출과 권한 남용을 구분해 판단해야 합니다.

### 절차

1. 노출 범위를 1차 분류합니다.
   - DB 자격증명 노출
   - 애플리케이션 secret 노출
   - 운영자 예외 접근 secret 노출
   - 키 사용 권한 오남용
2. 문제 자격증명 또는 키의 신규 사용을 즉시 차단합니다.
3. 관련 세션과 연결된 임시 자격증명을 폐기합니다.
4. 증적 보존이 필요한 로그 소스를 우선 잠급니다.
   - 관리 평면 감사 로그
   - 애플리케이션 감사 로그
   - 데이터 접근 로그
   - 운영 세션 로그
5. 새 자격증명 또는 대체 키를 발급합니다.
6. 영향을 받는 워크로드 설정을 갱신합니다.
7. 회전 전후 사용 로그를 비교해 추가 악용 여부를 확인합니다.
8. 정상 동작과 추가 악용 차단 여부를 검증합니다.
9. 사고 티켓에 회전 시각, 대상, 영향 범위, 후속 점검 결과를 기록합니다.

### 필수 검증 항목

| 검증 대상 | 판정 기준 |
| --- | --- |
| 구 자격증명 차단 | 구 버전으로 신규 인증 또는 연결 불가 상태 |
| 새 자격증명 반영 | 주요 워크로드 정상 동작 상태 |
| 키 사용 로그 | 승인되지 않은 principal 또는 경로의 추가 사용 없음 |
| 파생 영향 | 배치 작업, 운영 자동화, 예외 접근 경로 정상성 확인 |

### 필수 증빙

- 사고 티켓
- 회전 실행 로그
- 세션 폐기 기록
- 회전 후 검증 결과
- 관련 로그 보존 기록

### 완료 기준

- 노출된 자격증명 또는 키로 더 이상 접근할 수 없습니다.
- 영향 워크로드가 복구되었습니다.
- 회전 원인과 재발 방지 조치가 등록되었습니다.
- 증적 보존과 사후 검토 일정이 사고 기록에 연결되어 있습니다.
