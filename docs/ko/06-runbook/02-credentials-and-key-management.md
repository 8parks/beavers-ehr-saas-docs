# 자격증명 및 키 관리

비밀정보와 키는 설정값이 아니라 운영 자산입니다. 따라서 발급, 사용, 교체, 폐기, 예외 승인까지 전 과정을 운영 기록으로 남겨야 합니다. 이 장은 정기 회전과 비상 회전을 분리해 설명하고, 회전 이후에도 서비스 기능과 검증 기능이 모두 유지되도록 운영 기준을 정의합니다.

## 관리 원칙 {#shared-principles}

- 자격증명 회전과 암호화 키 회전은 동일한 변경으로 취급하지 않습니다.
- 회전 계획에는 되돌림 경로와 검증 시나리오가 반드시 포함되어야 합니다.
- 새 값 적용 확인보다 구 값 사용 중단 확인을 더 중요하게 봅니다.
- 예외 접근용 비밀정보는 일반 서비스 비밀정보보다 짧은 주기와 강한 승인 절차를 적용합니다.
- 사고성 회전은 정기 회전의 빠른 버전이 아니라, 차단과 기록 보존을 포함한 별도 절차로 운영합니다.

## 기준 주기 {#rotation-baseline}

다음 표는 기본 운영 기준입니다. 조직 정책이나 규제 요구가 더 엄격하면 더 짧은 주기를 적용합니다.

| 자산 유형 | 기본 최대 주기 | 예외 승인 조건 |
| --- | --- | --- |
| DB 접속 자격증명 | 90일 | 레거시 연동, 단계적 이전 필요, 공급자 제약 |
| 애플리케이션 secret | 90일 | 외부 시스템 의존성, 무중단 전환 불가 |
| 예외 접근용 secret | 30일 | 미사용 유지 확인, 대체 경로 구축 중 |
| 서비스 계정·자동화 자격증명 | 90일 | 교체 시 운영 중단 위험 존재 |
| 서명 키 참조 정보 | 180일 | 기존 검증 체계와의 호환성 유지 필요 |
| 저장 데이터 보호용 KMS 키 참조 | 조직 키 관리 정책 준수, 최소 연 1회 재검토 | 자동 회전 미지원, 외부 의존성 존재 |

이 주기는 권장값이 아니라 운영 기준입니다. 예외를 승인할 때는 기간 연장 사유와 종료 계획을 함께 남겨야 합니다.

## 역할 분리 {#duty-separation}

| 역할 | 책임 |
| --- | --- |
| 요청자 | 회전 필요성 제기, 대상 자산과 영향 범위 제출 |
| 승인자 | 회전 시점, 서비스 영향, 예외 여부 검토 |
| 실행자 | 비밀정보·키 전환 수행 |
| 검증자 | 기능 확인, 구 버전 사용 종료 확인, 로그 점검 |

회전 작업은 변경 승인 없이 수행하지 않습니다. 다만 비상 회전은 사후 승인으로 전환될 수 있으므로, 사고 기록과 연계된 승인 이력이 별도로 남아야 합니다.

## 키 교체 {#key-rotation}

### 적용 범위

- DB 접속 자격증명
- 애플리케이션 secret
- 연구 데이터셋 처리 경로 secret
- 서명 검증에 사용하는 키 참조 정보
- 저장 데이터 보호용 KMS 키 참조

### 준비 단계

정기 회전은 일정표를 따라 수행하더라도 사전 검토 없이 자동 실행하지 않습니다. 운영자는 회전 전에 다음 질문에 답할 수 있어야 합니다.

- 어떤 워크로드가 영향을 받는가
- 읽기 경로와 쓰기 경로를 동시에 건드리는가
- 재시작 없이 롤링 전환이 가능한가
- 장애 발생 시 구 버전으로 되돌릴 수 있는가
- 회전 이후 어떤 로그를 봐야 실패를 빨리 알아차릴 수 있는가

영향 워크로드는 최소한 다음 네 가지 경로를 포함해 검토합니다.

- 환자 조회
- 진료 기록 조회 및 작성
- 데이터셋 생성 및 다운로드
- 운영자 예외 접근 경로

### 회전 유형 구분

| 회전 유형 | 주요 대상 | 반드시 확인할 사항 |
| --- | --- | --- |
| 자격증명 회전 | DB 비밀번호, 애플리케이션 secret, 운영용 secret | 애플리케이션 재연결 성공, 구 자격증명 차단 상태 |
| 암호화 키 참조 전환 | 저장 데이터 보호용 KMS 키 참조 | 신규 암호화 성공, 기존 데이터 복호화 유지 |
| 서명 체계 전환 | 진료 기록 서명·검증 경로 | 신규 서명 생성 성공, 기존 서명 검증 유지 |

실패 양상이 서로 다르기 때문에 한 번의 변경 창구에서 묶어 수행하더라도, 검증 항목은 유형별로 분리해 기록해야 합니다.

### 자격증명 회전 절차

1. 새 자격증명을 발급합니다.
2. 참조 대상을 단계적으로 바꿉니다.
3. 읽기 경로와 쓰기 경로를 개별 검증합니다.
4. 배치 작업과 비대화형 경로가 새 값을 사용하도록 갱신합니다.
5. 구 자격증명을 비활성화합니다.
6. 실패한 연결이 구 값을 재사용하지 않는지 로그로 확인합니다.

자격증명 회전에서 가장 흔한 실패는 값이 바뀐 뒤 일부 작업이 이전 값을 계속 들고 있는 상태입니다. 따라서 배포 성공보다 재시도 로그와 실패율 변화를 먼저 살펴봐야 합니다.

### 암호화 키 참조 전환 절차

암호화 키를 바꾼다는 것은 기존 데이터를 다시 암호화하는 작업과 새 데이터에 적용할 참조를 바꾸는 작업을 구분해야 한다는 뜻입니다. 이 가이드라인의 기준은 다음과 같습니다.

- 새로 생성되는 데이터는 새 키 참조를 사용해야 합니다.
- 기존 데이터는 복호화 가능 상태를 유지해야 합니다.
- 키 정책 변경과 키 참조 변경이 동시에 일어나는 경우, 권한 검증과 기능 검증을 분리해 수행해야 합니다.

운영자는 전환 직후 환자 조회, 진료 기록 저장, 데이터셋 생성 경로에서 암호화 실패가 없는지 확인해야 합니다.

### 서명 체계 전환 절차

서명 체계는 단순 암호화와 다르게 기존 기록의 검증 가능성이 유지되어야 합니다. 따라서 다음 원칙을 지킵니다.

- 새로 생성되는 진료 기록은 새 서명 체계를 사용합니다.
- 기존 기록은 이전 검증 정보로 계속 검증 가능해야 합니다.
- 전환 직후 일정 기간은 구 버전 검증 정보가 남아 있어야 합니다.
- 검증 실패가 발생하면 키 복구보다 검증 체계 손상 여부를 먼저 확인합니다.

### 종료 확인

회전 작업은 다음 네 가지가 모두 충족될 때만 종료합니다.

| 확인 항목 | 종료 기준 |
| --- | --- |
| 기능 상태 | 주요 워크로드 정상 동작 |
| 구 버전 사용 | 신규 사용 없음 |
| 예외 경로 | 배치, 자동화, 운영 스크립트까지 반영 완료 |
| 기록 상태 | 회전 계획, 검증 결과, 예외 기록 보존 완료 |

### 금지 사항

- 회전 계획 없이 운영 중 바로 secret 값을 덮어쓰지 않습니다.
- 구 자격증명 비활성화 없이 새 값만 배포하고 종료하지 않습니다.
- 서명 체계 전환에서 기존 기록 검증 시험을 생략하지 않습니다.

## 비상 회전 절차 {#emergency-rotation}

### 개시 조건

비상 회전은 다음과 같은 상황에서 시작합니다.

- 비밀정보 스캔에서 실제 노출이 확인된 경우
- 코드 저장소나 로그에 자격증명이 평문으로 남은 경우
- 운영자 계정 탈취 또는 세션 하이재킹이 의심되는 경우
- 비정상적인 `Decrypt`, 서명 요청, secret 조회가 감지된 경우

### 초기 판단

비상 회전은 먼저 노출 유형을 분류하는 일부터 시작합니다.

| 분류 | 예시 | 우선 조치 |
| --- | --- | --- |
| 데이터 평면 직접 노출 | DB 자격증명, 데이터셋 처리 secret 노출 | 즉시 차단, 서비스 영향 범위 산정 |
| 관리 평면 노출 | 운영용 secret, 자동화 계정 자격증명 노출 | 권한 차단, 사용 이력 확인 |
| 키 사용 권한 오남용 | 비정상 `Decrypt`, 비정상 서명 요청 | 권한 제한, 로그 보존, 추가 악용 확인 |

AWS KMS의 키 material은 일반적으로 직접 추출되지 않습니다. 실제 위험은 키 자체보다 키를 사용할 수 있는 권한이 잘못 쓰이는 데서 발생하는 경우가 많습니다. 따라서 비상 회전은 `키 유출`과 `키 사용 권한 오남용`을 구분해 다뤄야 합니다.

### 차단 단계

1. 노출된 자격증명 또는 권한의 신규 사용을 차단합니다.
2. 관련 세션과 임시 자격증명을 폐기합니다.
3. 서비스 영향이 큰 경우 해당 경로를 일시 중지합니다.
4. 관리 평면 감사 로그, 애플리케이션 감사 로그, 데이터 접근 로그, 운영 세션 로그를 우선 보존합니다.

이 단계에서 가장 중요한 것은 “새 값을 넣는 것”보다 “노출된 값으로 더 이상 접근하지 못하게 하는 것”입니다.

### 전환 단계

1. 새 자격증명 또는 대체 키 참조를 발급합니다.
2. 영향을 받는 워크로드 설정을 갱신합니다.
3. 읽기, 쓰기, 배치, 운영 경로를 각각 시험합니다.
4. 회전 전후 로그를 비교해 추가 악용 징후를 확인합니다.
5. 필요 시 관련 principal에 임시 제한 정책을 추가합니다.

### 종료 확인

| 확인 항목 | 종료 기준 |
| --- | --- |
| 구 값 차단 | 노출된 값으로 신규 인증 또는 연결 불가 |
| 새 값 반영 | 주요 워크로드 정상 동작 |
| 추가 악용 여부 | 승인되지 않은 사용 흔적 없음 |
| 후속 조치 | 재발 방지 작업과 사후 검토 일정 등록 완료 |

### 필수 기록

- 사고 티켓
- 노출 유형 분류 결과
- 회전 수행 로그
- 세션 폐기 기록
- 사용 이력 비교 결과
- 후속 점검 일정

### 금지 사항

- 노출된 값을 차단하지 않은 상태에서 새 값 배포만 진행하지 않습니다.
- 회전 이후 과거 접근 범위 산정을 생략하지 않습니다.
- 서비스가 살아 있다는 이유만으로 검증을 생략하고 종료하지 않습니다.
