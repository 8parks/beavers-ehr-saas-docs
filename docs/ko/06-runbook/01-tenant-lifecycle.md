# Tenant 라이프사이클

Tenant 라이프사이클 절차는 단순 계정 관리가 아니라 멀티테넌트 격리의 생성, 유지, 종료를 다루는 운영 절차입니다.  
특히 온보딩과 오프보딩은 `tenant_registry`, 계정 상태, 데이터 저장 위치, 키 접근 범위를 함께 다뤄야 합니다.

## 상태 전이 규칙

| 상태 | 의미 | 허용 작업 | 금지 작업 |
| --- | --- | --- | --- |
| `PROVISIONING` | 초기 생성 및 검증 단계 | 자산 생성, 테스트 계정 검증, 음성 테스트 | 실제 고객 데이터 적재, 외부 사용자 일반 접근 |
| `ACTIVE` | 정상 운영 상태 | 일반 서비스 제공, 승인된 운영 변경 | 승인 없는 예외 접근, 무기한 임시 권한 유지 |
| `SUSPENDED` | 중지 또는 조사 상태 | 제한된 조사, 보존 조치, 복구 준비 | 신규 일반 접근, 데이터 파기 없는 임의 해제 |
| `OFFBOARDED` | 계약 종료 및 운영 종료 상태 | 보존 기록 조회, 감사 대응 | 서비스 재개, 일반 사용자 접근 재활성화 |

상태 전이는 반드시 `tenant_registry`, 계정 상태, 접근 경로 상태가 함께 일치해야 합니다. 문서상 상태와 실제 접근 가능 상태가 다르면 통제가 실패한 것으로 간주합니다.

## 온보딩

### 목적

신규 병원 또는 기관을 기존 테넌트와 격리된 상태로 등록하고, 초기 관리자 접근을 안전하게 제공합니다.

### 트리거

- 신규 고객 계약 체결
- 파일럿 테넌트 생성 승인
- 기존 테넌트의 조직 분리 또는 신규 기관 추가

### 사전 조건

- 고객 식별자, 계약 범위, 데이터 처리 범위가 확정되어 있어야 합니다.
- 테넌트 유형이 병원, 연구기관, SaaS 운영용 중 무엇인지 정의되어 있어야 합니다.
- 초기 관리자 역할, 승인자, 운영 책임자가 지정되어 있어야 합니다.
- 네트워크 격리, 데이터 저장 구조, 접근 경계가 표준 템플릿으로 준비되어 있어야 합니다.
- 검증에 사용할 양성 테스트 및 음성 테스트 시나리오가 준비되어 있어야 합니다.

### 역할 분리

| 역할 | 책임 |
| --- | --- |
| 요청자 | 계약 정보와 서비스 범위 제출 |
| 승인자 | 계약 범위, 데이터 처리 범위, 테넌트 유형 검토 |
| 실행자 | 테넌트 생성, 계정 발급, 저장 구조 구성 |
| 검증자 | 격리 검증, 음성 테스트, 증빙 검토 |

### 절차

1. 테넌트 신청서, 승인 기록, 계약상 데이터 처리 범위를 확인합니다.
2. `tenant_id`를 발급하고 중복 여부를 검증합니다.
3. 작업 티켓에 요청자, 승인자, 실행자, 검증자를 명시합니다.
4. `tenant_registry`에 테넌트 상태를 `PROVISIONING`으로 등록합니다.
5. 데이터 저장 구조를 생성합니다.
   - 데이터베이스 스키마 또는 해당하는 테넌트 논리 구획 생성
   - 임상 첨부파일 저장 경로 생성
   - 연구 데이터셋 저장 경로가 필요한 경우 별도 생성
6. 테넌트별 메타데이터를 등록합니다.
   - schema 이름
   - 임상 파일 prefix
   - tenant 유형
   - 생성 시각
   - 초기 상태
7. 초기 관리자 계정을 생성합니다.
   - Cognito 사용자 생성
   - 역할과 그룹 검증
   - MFA 요구 여부 확인
8. 초기 자격증명을 안전한 채널로 전달합니다.
   - 이메일 본문에 평문 비밀번호를 넣지 않습니다.
   - 임시 비밀번호 또는 초대 링크는 만료 시간을 가진 방식으로 전달합니다.
9. 교차 검증을 수행합니다.
   - 신규 관리자 토큰으로 자기 테넌트 접근이 가능한지 확인
   - 다른 테넌트 리소스 접근이 거부되는지 음성 테스트 수행
   - 환자 조회, 기록 작성, 첨부파일 접근, 데이터셋 요청 경로를 개별 검증
10. 이상이 없으면 `tenant_registry` 상태를 `ACTIVE`로 변경합니다.
11. 온보딩 완료 증빙을 보존합니다.

### 필수 검증 항목

| 검증 대상 | 판정 기준 |
| --- | --- |
| 인증 및 로그인 | 초기 관리자만 승인된 역할로 로그인 가능 상태 |
| 데이터 격리 | 타 테넌트 자산 접근 거부 확인 |
| 저장 경로 | 임상 파일 및 데이터셋 경로가 올바른 prefix 또는 구획 사용 상태 |
| 로그 수집 | 감사 로그와 운영 로그가 신규 테넌트 활동을 식별 가능 상태 |
| 회수 가능성 | 계정 비활성화, 세션 무효화, 상태 전환 절차 재현 가능 상태 |

### 필수 증빙

- 온보딩 승인 기록
- `tenant_id`와 매핑 정보 등록 기록
- 초기 관리자 계정 생성 기록
- MFA 및 역할 검증 결과
- cross-tenant 음성 테스트 결과
- `PROVISIONING -> ACTIVE` 상태 전이 기록

### 완료 기준

- 신규 테넌트가 자기 테넌트 내에서만 정상 접근 가능합니다.
- 기존 테넌트와의 데이터 격리 검증이 완료되었습니다.
- 초기 관리자 자격증명 전달이 완료되었고, 평문 비밀정보가 남아 있지 않습니다.

### 금지 사항

- `tenant_id`, schema, prefix를 수동 메모나 코드 상수로만 관리하지 않습니다.
- 관리자 계정을 공유 계정으로 발급하지 않습니다.
- cross-tenant 음성 테스트 없이 `ACTIVE` 상태로 전환하지 않습니다.
- 로그 수집이 확인되지 않은 상태에서 실제 고객 데이터 적재를 허용하지 않습니다.

## 오프보딩

### 목적

계약 종료 또는 서비스 종료 시 테넌트의 접근 경로를 차단하고, 데이터 반환 또는 파기 절차를 검증 가능한 방식으로 완료합니다.

### 트리거

- 계약 종료
- 고객의 서비스 해지 요청
- 장기 중단 후 최종 파기 승인
- 법적 분쟁 종료 후 파기 전환

### 사전 조건

- 데이터 반환, 보존, 파기 중 어떤 정책이 적용되는지 확정되어 있어야 합니다.
- 법정 보존 의무 또는 계약상 보존 조항이 검토되어 있어야 합니다.
- 고객 통지 담당자와 내부 승인자가 지정되어 있어야 합니다.
- 백업, 복제본, 감사 로그, 법적 보존 사본의 처리 원칙이 합의되어 있어야 합니다.

### 역할 분리

| 역할 | 책임 |
| --- | --- |
| 요청자 | 해지 또는 종료 요청 제출 |
| 승인자 | 계약 및 법적 보존 요구사항 승인 |
| 실행자 | 접근 차단, 반환, 파기, 상태 전환 수행 |
| 검증자 | 잔존 자산 점검, 음성 테스트, 완료 판정 |

### 절차

1. 오프보딩 요청과 승인 문서를 확인합니다.
2. `tenant_registry` 상태를 `SUSPENDED`로 변경합니다.
3. 모든 사용자 세션과 신규 접근을 차단합니다.
   - Cognito 사용자 비활성화
   - 활성 세션 무효화
   - 미사용 토큰/예외 권한 재검토
4. 데이터 반환이 필요한 경우 범위를 확정하고 export를 수행합니다.
   - export 대상 데이터 종류
   - 제공 형식
   - 전달 채널
   - 만료 시점
5. 파기 범위를 확정합니다.
   - 데이터베이스 스키마
   - 임상 첨부파일 객체
   - 연구 데이터셋 아티팩트
   - cache 또는 임시 export
   - 미만료 presigned URL
   - 관련 비밀정보 및 접근 권한
   - 백업 스냅샷과 복제본
   - DR 보관본
6. 반환 대상과 파기 대상을 분리해 기록합니다.
7. 서비스 제공자 보존 증적을 별도 식별합니다.
   - 애플리케이션 감사 로그 보존 위치
   - 관리 평면 감사 로그 보존 위치
   - 접근 통지, 파기 승인, 분쟁 대응 기록
8. 데이터 파기 또는 격리를 수행합니다.
9. 키 접근과 secret 접근을 차단합니다.
10. 오프보딩 후 음성 테스트를 수행합니다.
   - 기존 사용자 계정 접근
   - 기존 토큰 접근
   - 기존 URL 접근
   - 연구 데이터셋 재다운로드 시도
   - 예약된 배치 작업 또는 비대화형 경로 실행 시도
11. 법정 보존 대상이 남아 있다면 보존 사유, 만료 시점, 접근 제한 조건을 별도 기록합니다.
12. `tenant_registry` 상태를 `OFFBOARDED`로 전환합니다.

### 잔존 자산 등록부 최소 항목

| 항목 | 최소 기록 내용 |
| --- | --- |
| 자산 유형 | 운영 데이터, 백업, 로그, export, 임시 파일, 예외 권한 |
| 현재 상태 | 반환 완료, 파기 완료, 보존 중, 삭제 대기 |
| 보존 사유 | 법정 보존, 계약상 요구, 조사 목적, 복구 필요성 |
| 만료 시점 | 삭제 예정일 또는 재검토 시점 |
| 접근 제한 | 허용 역할, 승인 방식, 접근 로그 요구사항 |

### 서비스 제공자 보존 증적 최소 항목

| 항목 | 최소 기록 내용 |
| --- | --- |
| 증적 유형 | 감사 로그, 승인 기록, 통지 기록, 조사 기록 |
| 소유 주체 | 서비스 제공자 측 보안 운영 또는 감사 담당 |
| 보존 근거 | 규제 요구, 계약상 분쟁 대응, 조사 목적 |
| tenant 반환/삭제 대상 여부 | 비대상 표시 |
| 접근 제한 | 승인자, 조회 목적, 조회 로그 요구사항 |

### 필수 증빙

- 오프보딩 승인 기록
- 세션 무효화 및 계정 비활성화 기록
- export 제공 기록 또는 파기 기록
- S3 / DB / dataset artifact 삭제 결과
- 백업 및 복제본 처리 기록
- 잔존 자산 등록부
- 서비스 제공자 보존 증적 등록부
- post-offboarding 음성 테스트 결과
- `SUSPENDED -> OFFBOARDED` 상태 전이 기록

### 완료 기준

- 오프보딩 대상 테넌트는 더 이상 API, 데이터셋, 첨부파일, 예외 경로로 접근할 수 없습니다.
- 보존 또는 파기 상태가 자산 단위로 설명 가능합니다.
- 남아 있는 예외 보유 자산이 있다면 사유와 만료 시점이 문서화되어 있습니다.
- 백업, 복제본, 로그 보존본까지 포함한 잔존 자산 현황이 설명 가능합니다.

### 금지 사항

- 계정 비활성화만 하고 세션 무효화를 생략하지 않습니다.
- dataset artifact, cache, 미만료 presigned URL을 파기 범위에서 제외하지 않습니다.
- 보존 사유가 없는 잔존 데이터를 남겨두지 않습니다.
- 백업 및 복제본을 “운영 경로가 아님”이라는 이유로 파기 또는 보존 검토 범위에서 제외하지 않습니다.
- 서비스 제공자 보존 증적을 tenant 반환 또는 삭제 대상과 혼합 기록하지 않습니다.

## Tenant 중지

### 목적

분쟁, 미납, 보안 사고, 조사 필요성 등으로 인해 테넌트를 즉시 중지해야 할 때 신규 접근을 차단하고 데이터 상태를 보존합니다.

### 트리거

- Cross-Tenant 접근 의심
- PHI 유출 의심
- 법적 분쟁
- 미납 또는 계약 위반

### 사전 조건

- 중지 사유가 사고 대응, 계약 이행, 법적 요구 중 무엇인지 분류되어 있어야 합니다.
- 중지 해제 권한자와 오프보딩 전환 권한자가 지정되어 있어야 합니다.
- 중지 기간 동안 허용되는 지원 범위와 금지 행위가 정의되어 있어야 합니다.

### 절차

1. 중지 사유와 승인자를 확인합니다.
2. `tenant_registry` 상태를 `SUSPENDED`로 전환합니다.
3. 신규 로그인과 기존 세션을 차단합니다.
4. 비대화형 접근 경로를 중지합니다.
   - 예약된 배치 작업
   - 데이터셋 생성 또는 export 작업
   - 비동기 파일 처리 작업
   - 자동화 토큰 또는 서비스 계정 기반 호출
5. 데이터는 파기하지 않고 보존 상태로 유지합니다.
6. 중지 중 허용되는 운영 행위를 명시합니다.
   - 고객 지원 연락
   - 조사 목적 제한 조회
   - 법적 보존 조치
7. 중지 해제 또는 오프보딩 전환 조건을 문서화합니다.

### 중지 해제 전 필수 확인

| 확인 항목 | 판정 기준 |
| --- | --- |
| 중지 원인 해소 | 사고 원인 또는 계약상 분쟁 상태 해소 확인 |
| 접근 경로 검증 | 계정, 세션, 예외 권한, URL, export 경로 재검증 완료 |
| 비대화형 경로 검증 | 배치 작업, 서비스 계정, 비동기 작업 재활성화 승인 완료 |
| 고객 통지 | 필요 시 해제 시점과 범위 통지 완료 |
| 후속 상태 결정 | `ACTIVE` 복귀 또는 `OFFBOARDED` 전환 명시 완료 |

### 필수 증빙

- 중지 승인 기록
- 상태 전이 기록
- 세션 차단 기록
- 비대화형 경로 중지 또는 해제 기록
- 중지 해제 또는 오프보딩 전환 판단 기록

### 완료 기준

- 테넌트는 신규 접근 불가 상태여야 합니다.
- 데이터는 조사 및 보존 목적에 맞게 유지되어야 합니다.
- 비대화형 경로는 승인 없이 실행되지 않는 상태여야 합니다.
