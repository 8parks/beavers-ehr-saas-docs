# Tenant 라이프사이클

테넌트 라이프사이클은 계정 생성과 삭제만 다루는 절차가 아닙니다. 멀티테넌트 서비스에서 격리를 언제 만들고, 어떤 상태로 유지하며, 어떤 조건 아래 종료할지를 정의하는 운영 절차입니다. 특히 온보딩과 오프보딩은 `tenant_registry`, 계정 상태, 저장 경로, 권한 범위, 로그 수집 상태가 함께 맞물려야 합니다.

## 운영 상태 모델 {#state-transition-rules}

| 상태 | 의미 | 허용 행위 | 차단 대상 | 종료 판단 기준 |
| --- | --- | --- | --- | --- |
| `PROVISIONING` | 신규 구성 및 검증 단계 | 자산 생성, 시험 계정 검증, 음성 테스트 | 고객 데이터 적재, 일반 사용자 접근 | 격리 검증과 로그 수집 확인 완료 |
| `ACTIVE` | 정상 제공 상태 | 일반 서비스 제공, 승인된 변경 수행 | 무승인 예외 접근, 무기한 임시 권한 | 운영 정책에 따른 정상 사용 유지 |
| `SUSPENDED` | 중지, 분쟁, 조사 단계 | 제한된 조사, 보존 조치, 재개 준비 | 신규 일반 접근, 무승인 재개, 데이터 파기 | 재개 또는 오프보딩 경로 결정 완료 |
| `OFFBOARDED` | 계약 종료 및 운영 종료 상태 | 보존 기록 조회, 감사 대응 | 서비스 재개, 사용자 접근 재활성화 | 반환·파기·보존 상태 설명 가능 |

상태 값만 바뀌고 접근 경로가 그대로 열려 있으면 상태 전환은 실패한 것으로 봅니다. 운영자는 `tenant_registry`, 계정 상태, 세션 상태, 저장 경로, 비동기 작업 상태를 함께 확인해야 합니다.

## 온보딩 {#tenant-onboarding}

### 운영 목적

신규 병원 또는 기관을 기존 테넌트와 분리된 상태로 등록하고, 초기 운영자가 안전하게 첫 로그인과 첫 검증을 수행할 수 있도록 준비합니다.

### 전제 조건

- 계약 범위와 데이터 처리 범위가 확정된 상태여야 합니다.
- 테넌트 유형이 병원, 연구기관, 운영 전용 중 어디에 속하는지 정의되어 있어야 합니다.
- 초기 관리자 역할, 승인자, 운영 책임자가 문서로 지정되어 있어야 합니다.
- 격리 검증에 사용할 양성 테스트와 음성 테스트 시나리오가 준비되어 있어야 합니다.

### 준비 단계

온보딩 작업은 먼저 테넌트의 운영 경계를 명확히 하는 데서 시작합니다. 운영자는 요청서와 계약 문서를 검토하고, 다음 항목이 모두 채워졌는지 확인합니다.

- `tenant_id` 발급 기준
- 데이터 저장 구조와 네이밍 규칙
- 초기 관리자 계정 수와 역할 범위
- MFA 적용 여부
- 임상 파일 경로, 데이터셋 경로, 로그 식별 체계
- 실패 시 되돌림 방식

준비 단계가 끝나기 전에는 고객 데이터 적재를 허용하지 않습니다.

### 실행 단계

1. 작업 티켓을 만들고 요청자, 승인자, 실행자, 검증자를 지정합니다.
2. `tenant_id`를 발급하고 중복 여부를 확인합니다.
3. `tenant_registry`에 상태를 `PROVISIONING`으로 등록합니다.
4. 데이터베이스 스키마 또는 동등한 논리 구획을 생성합니다.
5. 임상 파일 저장 경로와 연구 데이터셋 경로를 생성합니다.
6. 테넌트 메타데이터를 등록합니다.
   - schema 이름
   - 임상 파일 prefix
   - tenant 유형
   - 생성 시각
   - 초기 상태
7. 초기 관리자 계정을 생성하고 역할과 MFA 정책을 적용합니다.
8. 자격증명을 안전한 전달 채널로 발급합니다.

이 단계에서 중요한 것은 자산을 만드는 일보다, 나중에 오프보딩과 감사에서 다시 찾아갈 수 있는 식별자를 일관되게 부여하는 일입니다.

### 검증 단계

온보딩은 계정이 발급되었다고 끝나지 않습니다. 신규 테넌트의 격리가 실제로 동작하는지, 서비스 관점에서 확인해야 합니다.

| 검증 대상 | 확인 방식 | 판정 기준 |
| --- | --- | --- |
| 로그인 경로 | 초기 관리자 직접 로그인 | 승인된 역할만 로그인 가능 상태 |
| 데이터 격리 | 자기 테넌트 조회와 타 테넌트 음성 테스트 | 타 테넌트 자산 접근 거부 확인 |
| 저장 경로 | 첨부파일, 데이터셋 경로 확인 | 올바른 prefix 또는 논리 구획 사용 상태 |
| 로그 수집 | 감사 로그와 운영 로그 확인 | 신규 활동을 `tenant_id` 단위로 식별 가능 상태 |
| 회수 가능성 | 계정 비활성화·세션 무효화 시험 | 상태 전환과 접근 차단 재현 가능 상태 |

검증자는 환자 조회, 기록 작성, 첨부파일 접근, 데이터셋 요청을 각각 독립 경로로 시험해야 합니다. 하나의 기능만 성공했다고 `ACTIVE` 상태로 전환하지 않습니다.

### 실패 시 처리

온보딩 도중 격리 실패나 로그 수집 실패가 발견되면 고객 데이터 적재를 중단하고, 상태를 `PROVISIONING`에 유지한 채 원인을 제거해야 합니다. 이미 생성한 계정과 경로가 있다면 삭제 또는 격리 상태로 되돌린 뒤 재검증합니다.

### 금지 사항

- `tenant_id`, schema, prefix를 수기 메모나 코드 상수에만 남기지 않습니다.
- 공유 관리자 계정을 발급하지 않습니다.
- 음성 테스트 없이 운영 상태로 전환하지 않습니다.
- 로그 수집이 확인되지 않은 상태에서 고객 데이터를 적재하지 않습니다.

## 오프보딩 {#tenant-offboarding}

### 운영 목적

계약 종료 또는 서비스 종료 시 테넌트의 접근 경로를 닫고, 데이터 반환·보존·파기 상태를 자산 단위로 설명할 수 있도록 정리합니다.

### 전제 조건

- 반환, 보존, 파기 중 어떤 정책을 적용할지 확정되어 있어야 합니다.
- 법정 보존 의무와 계약상 보존 조항이 검토되어 있어야 합니다.
- 고객 통지 담당자와 내부 승인자가 지정되어 있어야 합니다.
- 백업, 복제본, 감사 로그, 법적 보존 사본에 대한 처리 원칙이 정리되어 있어야 합니다.

### 계획 수립

오프보딩은 먼저 무엇을 고객에게 돌려주고, 무엇을 서비스 제공자가 규제 대응을 위해 별도로 보관할지 구분하는 작업입니다. 운영자는 다음 두 범주를 분리해 기록합니다.

- 고객 반환 또는 파기 대상 자산
- 서비스 제공자 보관 기록

이 구분이 선행되지 않으면, 고객 데이터 삭제와 감사 기록 보관이 서로 충돌하게 됩니다.

### 실행 단계

1. 오프보딩 요청과 승인 문서를 확인합니다.
2. `tenant_registry` 상태를 `SUSPENDED`로 전환합니다.
3. 사용자 계정, 활성 세션, 예외 권한을 차단합니다.
4. 필요한 경우 데이터 export 범위와 전달 채널을 확정해 반환 작업을 수행합니다.
5. 파기 범위를 자산 단위로 확정합니다.
   - 데이터베이스 스키마
   - 임상 첨부파일 객체
   - 연구 데이터셋 산출물
   - 임시 export와 cache
   - 활성화된 데이터셋 전달 권한
   - 관련 secret과 권한
   - 백업 스냅샷, 복제본, DR 보관본
6. 파기 또는 장기 보존 전환을 수행합니다.
7. 키 접근과 secret 접근을 차단합니다.

### 잔존 자산 관리

오프보딩 완료 여부는 “운영 경로에서 삭제됨”만으로 판단하지 않습니다. 다음과 같이 잔존 자산 등록부를 유지해야 합니다.

| 항목 | 반드시 기록할 내용 |
| --- | --- |
| 자산 유형 | 운영 데이터, 백업, 로그, export, 임시 파일, 예외 권한 |
| 현재 상태 | 반환 완료, 파기 완료, 보존 중, 삭제 대기 |
| 보존 사유 | 법정 보존, 계약상 요구, 조사 목적, 복구 필요성 |
| 재검토 시점 | 삭제 예정일 또는 보존 재승인 시점 |
| 접근 조건 | 허용 역할, 승인 방식, 접근 로그 요구사항 |

서비스 제공자가 별도로 보관하는 기록도 구분해야 합니다.

| 기록 유형 | 소유 주체 | 보존 근거 | 고객 반환/삭제 대상 여부 |
| --- | --- | --- | --- |
| 감사 로그 | 보안 운영, 감사 담당 | 규제 요구, 분쟁 대응, 조사 목적 | 비대상 |
| 승인 기록 | 운영 승인자, 계약 책임자 | 변경 통제, 감사 대응 | 비대상 |
| 통지 및 조사 기록 | 사고 대응 책임자, 법무/컴플라이언스 | 사고 대응, 대외 보고 | 비대상 |

### 검증 단계

오프보딩 후에는 접근이 정말 닫혔는지 시험해야 합니다.

- 기존 사용자 계정 로그인 시도
- 기존 토큰 재사용 시도
- 기존 데이터셋 전달 경로 접근 시도
- 연구 데이터셋 재다운로드 시도
- 예약된 배치 작업과 비대화형 경로 재실행 시도

검증 결과가 모두 차단 상태임이 확인되어야 `OFFBOARDED` 상태로 전환할 수 있습니다.

### 금지 사항

- 계정 비활성화만 하고 세션 무효화를 생략하지 않습니다.
- dataset artifact, cache, 활성화된 데이터셋 전달 권한을 파기 범위에서 제외하지 않습니다.
- 보존 사유가 없는 잔존 데이터를 남기지 않습니다.
- 백업과 복제본을 운영 경로가 아니라는 이유로 검토 대상에서 빼지 않습니다.
- 서비스 제공자 보관 기록을 고객 반환 또는 삭제 대상 자산과 섞어 관리하지 않습니다.

## Tenant 중지 {#tenant-suspension}

### 운영 목적

분쟁, 미납, 조사, 보안 사고처럼 즉시 접근을 제한해야 하는 상황에서 데이터는 보존한 채 서비스만 중지합니다. 중지는 종료가 아니라 보류 상태이며, 재개 또는 오프보딩으로 이어질 수 있어야 합니다.

### 전제 조건

- 중지 사유가 계약상 사유인지 사고 대응인지 분류되어 있어야 합니다.
- 중지 해제 승인자와 오프보딩 전환 승인자가 지정되어 있어야 합니다.
- 중지 기간 동안 허용할 운영 행위와 금지 행위가 정의되어 있어야 합니다.

### 실행 단계

1. 중지 사유와 승인자를 확인합니다.
2. `tenant_registry` 상태를 `SUSPENDED`로 전환합니다.
3. 신규 로그인과 기존 세션을 차단합니다.
4. 비대화형 접근 경로를 중지합니다.
   - 예약된 배치 작업
   - 데이터셋 생성 및 export 작업
   - 비동기 파일 처리
   - 서비스 계정 기반 호출
5. 데이터는 파기하지 않고 보존 상태로 유지합니다.

### 중지 상태에서 허용되는 행위

중지 상태는 모든 행위를 멈추는 상태가 아니라, 조사와 보존을 위한 최소 행위만 허용하는 상태입니다.

- 고객 지원 연락
- 조사 목적의 제한적 조회
- 법적 보존 조치
- 복구 또는 오프보딩 계획 수립

### 재개 판단

중지를 해제하기 전에는 원인이 제거되었는지와 접근 경로가 다시 열려도 되는지를 따로 확인해야 합니다.

| 확인 항목 | 판정 기준 |
| --- | --- |
| 중지 사유 해소 | 사고 원인 또는 계약상 분쟁 상태 해소 확인 |
| 접근 경로 상태 | 계정, 세션, 예외 권한, URL, export 경로 재검증 완료 |
| 비대화형 경로 상태 | 배치 작업, 서비스 계정, 비동기 작업 재활성화 승인 완료 |
| 통지 필요성 | 고객 또는 내부 이해관계자 통지 여부 판단 완료 |
| 다음 상태 | `ACTIVE` 복귀 또는 `OFFBOARDED` 전환 결정 완료 |

### 금지 사항

- 중지 상태에서 고객 데이터 파기를 시작하지 않습니다.
- 중지 해제 전에 배치 작업과 서비스 계정 경로를 먼저 열지 않습니다.
- 조사 목적 예외 접근을 일반 운영으로 확대하지 않습니다.
