# 정기 점검 및 검증

정기 점검은 보안 통제가 설계대로 남아 있는지를 확인하는 절차입니다.  
월간 점검은 운영 드리프트를 찾는 데 집중하고, 취약점 스캐닝은 노출면 점검에, 침투 테스트는 설계 가정 검증에 집중합니다.

## 점검 책임 {#review-ownership}

| 활동 | 주관 조직 | 검토 조직 | 최소 산출물 |
| --- | --- | --- | --- |
| 월간 점검 | 보안 운영 팀 | 서비스 오너, 플랫폼 운영 담당자 | 월간 운영 보고서 |
| 취약점 스캐닝 | 보안 운영 팀 또는 지정 스캐닝 담당자 | 서비스 오너 | 스캔 결과 및 시정 조치 목록 |
| 침투 테스트 | 독립 검증 조직 또는 외부 수행자 | 보안 운영 팀, 서비스 오너 | 결과 보고서 및 재테스트 기록 |

## 월간 점검 {#monthly-review}

### 점검 항목

- `tenant_registry` 정합성
- 비활성 tenant 접근 시도 존재 여부
- break-glass 사용 빈도와 승인 기록
- secret 접근 이벤트 이상치
- KMS 사용 패턴 이상치
- 앱 감사 로그와 CloudTrail 수집 상태
- presigned URL 발급 정책 드리프트 여부
- 오프보딩 잔존 자산 존재 여부

### 수행 절차

1. 월간 점검 체크리스트를 생성합니다.
2. 테넌트 상태, 계정 상태, 로그 수집 상태를 확인합니다.
3. 예외 항목을 심각도별로 분류합니다.
4. 즉시 수정 항목과 다음 릴리스 보완 항목을 분리합니다.
5. 점검 결과를 월간 운영 보고서로 남깁니다.

### 최소 판정 기준

| 점검 항목 | 정상 기준 | 즉시 에스컬레이션 기준 |
| --- | --- | --- |
| 비활성 tenant 접근 시도 | 승인된 테스트 또는 조사 목적 외 발생 없음 | 승인 기록 없는 접근 1건 이상 |
| Break-glass 사용 | 모든 사용 건에 승인 및 사후 검토 완료 | 승인 누락 또는 미종결 사용 1건 이상 |
| secret 접근 이벤트 | 승인된 운영 창구 또는 자동화 경로에 한정 | 미승인 principal 또는 비정상 시간대 사용 |
| KMS 사용 패턴 | 정의된 서비스 경로와 승인된 역할에 한정 | 비승인 principal 사용, 급격한 사용 급증, 설명 불가 사용 |
| 로그 수집 상태 | 앱 감사 로그, 운영 로그, 관리 평면 감사 로그 모두 정상 수집 | 로그 누락, 무결성 검증 실패, 보존 정책 드리프트 |
| 오프보딩 잔존 자산 | 등록부와 실제 상태 일치 | 등록 누락 자산 1건 이상, 만료 초과 자산 존재 |

위 표에서 정량 기준이 필요한 항목은 환경별 기준선을 별도로 정의해야 합니다. 다만 승인 누락, 미종결 예외 사용, 로그 누락, 잔존 자산 누락은 건수와 관계없이 즉시 시정 대상으로 분류합니다.

### 기준선 예시

| 항목 | 권장 기준선 예시 | 운영 해석 |
| --- | --- | --- |
| Break-glass 사용 | 월 0건 유지, 1건 발생 시 사후 검토 필수 | 반복 발생 시 상시 운영 경로 오남용 의심 |
| secret 조회 이벤트 | 승인된 자동화 경로 및 지정 운영 창구에 한정 | 신규 principal 등장 시 즉시 조사 |
| KMS 사용 패턴 | 정의된 서비스 경로 외 신규 사용 0건 | 야간 급증, 비정상 지역, 설명 불가 principal 사용 시 조사 |
| 비활성 tenant 접근 시도 | 승인된 테스트 외 0건 | 1건 이상 발생 시 즉시 에스컬레이션 |
| 오프보딩 잔존 자산 | 만료 초과 0건 | 1건 이상 존재 시 시정 조치 대상 |

위 예시는 기본선이며, 실제 환경에서는 tenant 수, 배치 처리량, 운영 시간대에 맞춰 수치와 허용 범위를 재정의해야 합니다. 다만 신규 principal 사용, 승인 누락, 로그 누락은 기준선 조정 대상이 아니라 항상 이상 징후로 처리합니다.

### 산출물

- 월간 점검 체크리스트
- 시정 조치 목록
- 미해결 리스크 목록
- 기준선 변경 또는 예외 승인 기록

## 취약점 스캐닝 {#vulnerability-scanning}

### 목적

공개 노출면, 애플리케이션 종속성, 운영 구성의 취약 상태를 조기에 탐지합니다.

### 범위

- 웹 진입점
- API 진입점
- 컨테이너 또는 빌드 아티팩트
- 코드 저장소 내 secret 노출 여부
- IAM / 네트워크 구성 드리프트

### 수행 원칙

- 프로덕션과 유사한 환경에서 수행합니다.
- PHI 원본을 사용하는 스캔은 금지합니다.
- 스캔 결과는 심각도와 악용 가능성으로 구분해 처리합니다.
- false positive는 별도 검토 기록 없이 무시하지 않습니다.

### 처리 기준

| 결과 등급 | 처리 원칙 |
| --- | --- |
| 높음 | 즉시 수정 또는 예외 승인 없이는 운영 유지 근거로 사용하지 않음 |
| 중간 | 수정 계획과 목표 일정 수립 필요 |
| 낮음 | 누적 추세 모니터링 및 차기 점검 시 재평가 |

### 주기 권고

- 코드 및 종속성 스캔은 릴리스 파이프라인과 정기 배치 모두에서 수행합니다.
- 공개 진입점에 대한 외부 노출면 스캔은 최소 월 1회 수행합니다.
- IAM, 네트워크, 스토리지 구성 드리프트 점검은 월간 점검과 연결합니다.

### 필수 증빙

- 스캔 시점과 범위
- 탐지 결과
- 조치 여부
- 재검증 결과
- false positive 판정 근거

## 침투 테스트 {#penetration-testing}

### 목적

문서화된 보안 가정이 실제로 유지되는지 독립적으로 검증합니다.

### 우선 검증 범위

- Cross-Tenant 접근 시도
- 동일 테넌트 내부 권한 상승
- presigned URL 재사용 및 객체 경로 오남용
- 연구 데이터셋 재식별 가능성
- break-glass 경로 남용 가능성
- 사고 대응 절차의 실행 가능성

### 수행 주기

- 주요 아키텍처 변경 전후
- 최소 연 1회
- 고위험 사고 또는 중대 결함 이후 재수행

### 사전 조건

- 테스트 범위, 허용 시간대, 금지 행위가 문서화되어 있어야 합니다.
- 실제 PHI 사용 여부와 대체 데이터 사용 원칙이 정의되어 있어야 합니다.
- 사고 대응 연락망과 비상 중단 절차가 테스트 전에 준비되어 있어야 합니다.

### 결과 처리 원칙

- `높음` 위험 결과는 수정 또는 예외 승인 없이 운영 유지 근거로 사용하지 않습니다.
- 위협 모델링 문서와 충돌하는 결과가 나오면 위협 시나리오를 갱신합니다.
- 침투 테스트 결과는 런북, 체크리스트, 운영 절차에 반영되어야 합니다.
- 재테스트 없이 “완료”로 닫는 항목은 예외 승인 없이는 허용하지 않습니다.

### 필수 증빙

- 테스트 범위와 전제
- 발견 사항 목록
- 재현 단계
- 수정 상태
- 재테스트 결과
- 테스트 중단 또는 예외 승인 기록
