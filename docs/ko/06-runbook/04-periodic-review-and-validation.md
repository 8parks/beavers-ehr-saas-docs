# 정기 점검 및 검증

정기 점검은 보안 통제가 한 번 설계된 뒤 그대로 남아 있는지를 확인하는 절차입니다. 운영 드리프트는 대개 작은 예외와 편의 조치에서 시작하므로, 월간 점검·취약점 스캐닝·침투 테스트를 별도 활동으로 나누고 각각 다른 질문을 던져야 합니다.

## 점검 책임 {#review-ownership}

| 활동 | 주관 조직 | 검토 조직 | 최소 산출물 |
| --- | --- | --- | --- |
| 월간 점검 | 보안 운영 팀 | 서비스 오너, 플랫폼 운영 담당자 | 월간 운영 보고서 |
| 취약점 스캐닝 | 보안 운영 팀 또는 지정 스캐닝 담당자 | 서비스 오너 | 스캔 결과, 시정 조치 목록 |
| 침투 테스트 | 독립 검증 조직 또는 외부 수행자 | 보안 운영 팀, 서비스 오너 | 결과 보고서, 재테스트 기록 |

점검 활동은 수행자와 결과 승인자를 분리합니다. 특히 침투 테스트 결과는 직접 테스트를 수행한 조직이 아닌 서비스 오너와 보안 운영 팀이 함께 시정 계획을 확정해야 합니다.

## 월간 점검 {#monthly-review}

### 점검 목표

월간 점검의 목적은 위협 모델링과 운영 가이드에서 정의한 통제가 실제 환경에 남아 있는지를 확인하는 데 있습니다. 즉, 설계가 맞았는지를 다시 검증하는 것이 아니라, 설계한 통제가 운영 과정에서 약해지지 않았는지를 살펴봅니다.

### 준비

월간 점검을 시작하기 전에 다음 자료를 준비합니다.

- 현재 tenant 상태 목록
- 최근 한 달간 예외 승인 기록
- Break-glass 사용 기록
- 주요 secret 접근 로그
- KMS 사용 로그
- 감사 로그 수집 상태
- 오프보딩 잔존 자산 등록부

준비 단계에서 자료가 누락되면 점검을 진행한 것으로 보지 않습니다. 자료를 확보하지 못한 사실 자체가 운영 통제 실패일 수 있기 때문입니다.

### 검토 절차

1. 테넌트 상태와 계정 상태를 대조합니다.
2. 비활성 tenant에 대한 접근 시도가 있었는지 확인합니다.
3. Break-glass 사용 건이 모두 승인과 사후 검토를 마쳤는지 확인합니다.
4. secret 접근 이벤트와 KMS 사용 패턴에서 신규 principal 또는 설명 불가 사용이 있는지 확인합니다.
5. 앱 감사 로그, 운영 로그, 관리 평면 감사 로그가 모두 정상 수집되는지 확인합니다.
6. 데이터셋 전달 경로 정책과 S3·KMS·DynamoDB VPC endpoint 정책이 실제 접근 경로와 어긋나지 않는지 확인합니다.
7. 오프보딩 자산 등록부와 실제 자산 상태가 일치하는지 확인합니다.

### 판정 기준

| 점검 항목 | 정상 기준 | 즉시 시정 기준 |
| --- | --- | --- |
| 비활성 tenant 접근 시도 | 승인된 시험 또는 조사 목적 외 발생 없음 | 승인 기록 없는 접근 1건 이상 |
| Break-glass 사용 | 모든 사용 건에 승인과 사후 검토 완료 | 승인 누락 또는 미종결 사용 1건 이상 |
| secret 접근 이벤트 | 승인된 자동화 경로와 지정 운영 경로에 한정 | 신규 principal 등장, 비정상 시간대 사용 |
| KMS 사용 패턴 | 정의된 서비스 경로와 승인된 역할에 한정 | 비승인 principal 사용, 설명 불가 급증 |
| 로그 수집 상태 | 앱 감사 로그, 운영 로그, 관리 평면 감사 로그 모두 정상 수집 | 로그 누락, 무결성 검증 실패, 보존 정책 드리프트 |
| 오프보딩 잔존 자산 | 등록부와 실제 자산 상태 일치 | 등록 누락 자산 존재, 만료 초과 자산 존재 |

### 결과 처리

점검 결과는 단순히 “이상 없음” 또는 “보완 필요”로 끝내지 않습니다. 발견 사항은 다음 세 범주로 나눕니다.

- 즉시 시정: 권한 누락, 로그 누락, 승인 없는 예외 사용처럼 바로 조치가 필요한 항목
- 계획 시정: 다음 변경 창구에서 수정 가능한 항목
- 구조 개선: 반복 예외나 설계와 운영의 간극이 드러난 항목

### 산출물

- 월간 점검 보고서
- 시정 조치 목록
- 미해결 위험 목록
- 예외 승인 또는 기준선 조정 기록

## 취약점 스캐닝 {#vulnerability-scanning}

### 운영 목적

취약점 스캐닝은 인터넷 노출면, 빌드 산출물, 종속성, 구성 드리프트에서 조기에 이상 징후를 찾기 위한 활동입니다. 침투 테스트처럼 깊은 공격 재현이 목적이 아니라, 넓은 범위를 주기적으로 훑어 취약 상태를 빨리 발견하는 데 초점을 둡니다.

### 범위 관리

- 웹 진입점
- API 진입점
- 빌드 산출물 또는 컨테이너 이미지
- 코드 저장소의 secret 노출 여부
- IAM, 네트워크, 스토리지 구성 드리프트

PHI 원본을 사용하는 스캔은 허용하지 않습니다. 테스트 데이터 또는 비식별 데이터만 사용합니다.

### 수행 방식

스캐닝은 릴리스 파이프라인과 정기 점검을 분리해 운용합니다. 릴리스 파이프라인에서는 배포 전 차단 목적의 스캔을 수행하고, 정기 점검에서는 이미 운영 중인 환경의 drift와 신규 노출면을 확인합니다.

### 결과 처리 기준

| 결과 등급 | 초기 검토 시점 | 처리 원칙 |
| --- | --- | --- |
| 높음 | 즉시 | 수정 또는 예외 승인 없이는 운영 유지 근거로 사용하지 않음 |
| 중간 | 영업일 기준 5일 이내 | 수정 계획과 목표 일정 수립 필요 |
| 낮음 | 다음 정기 점검 전 | 누적 추세 확인, 재평가 수행 |

false positive는 별도 근거 없이 닫지 않습니다. 왜 false positive로 판단했는지, 어떤 환경 차이 때문에 해당되지 않는지 기록해야 합니다.

### 기록과 근거 자료

- 스캔 시점과 범위
- 탐지 결과
- 조치 여부
- 재검증 결과
- false positive 판정 근거

## 침투 테스트 {#penetration-testing}

### 운영 목적

침투 테스트는 설계 문서가 가정한 통제가 실제 환경에서 유지되는지를 공격 관점에서 검증하는 활동입니다. 문서에 적힌 통제 문장보다, 우회가 실제로 가능한지 여부를 보는 절차라고 이해해야 합니다.

### 우선 검증 범위

- Cross-Tenant 접근 시도
- 동일 테넌트 내부 권한 상승
- 데이터셋 전달 경로 우회 및 객체 경로 오남용
- 연구 데이터셋 재식별 가능성
- Break-glass 경로 남용 가능성
- 사고 대응 절차의 실행 가능성

### 수행 준비

테스트를 시작하기 전에 다음 조건이 문서화되어 있어야 합니다.

- 범위와 제외 범위
- 허용 시간대
- 금지 행위
- 실제 PHI 사용 여부와 대체 데이터 원칙
- 비상 중단 절차와 연락망

준비가 부족한 상태에서 침투 테스트를 시작하면 테스트가 아니라 서비스 장애를 유발할 수 있으므로, 운영 통제 관점에서도 사전 합의가 필수입니다.

### 결과 반영

침투 테스트는 보고서 발행으로 끝나지 않습니다. 결과는 다음 세 문서에 반영되어야 합니다.

- 위협 모델링
- 운영 가이드
- 시정 조치 추적표

특히 위협 모델링의 가정과 충돌하는 결과가 나오면, 위협 시나리오를 먼저 갱신하고 런북 절차도 같이 손봐야 합니다.

### 재검증

`높음` 위험 결과는 수정 또는 예외 승인 없이 운영 유지 근거로 사용할 수 없습니다. 또한 재테스트 없이 “완료”로 닫는 항목은 허용하지 않습니다. 수정한 뒤 같은 경로에서 다시 공격이 실패하는지까지 확인해야 테스트가 끝납니다.

### 기록과 근거 자료

- 테스트 범위와 전제
- 발견 사항 목록
- 재현 단계
- 수정 상태
- 재테스트 결과
- 테스트 중단 또는 예외 승인 기록
