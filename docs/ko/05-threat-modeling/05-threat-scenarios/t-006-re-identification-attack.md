# T-06 재식별 공격

## 위협 개요

| 항목 | 내용 |
|------|------|
| 위험도 | 높음 |
| 관련 워크로드 | `S3` |
| 관련 자산 | `A-01`, `A-03`, `A-05`, `A-09` |
| 관련 경계 | `TB-06`, `TB-08` |
| 대표 행위자 | 연구기관 사용자, 외부 데이터 보유자, 승인 프로세스를 우회하려는 내부자 |

재식별 공격은 직접 식별자를 제거한 뒤에도 dataset이 개인을 식별할 수 있게 되는 상황을 의미합니다. 본 가이드라인에서 이는 단순 데이터 품질 문제가 아니라 별도의 고위험 Privacy 위협입니다.

이 위협의 핵심은 “이름과 주민번호를 제거했는가”가 아닙니다. 연구자가 받는 결과물이 외부 정보와 결합되거나 동일 과제의 반복 추출과 결합될 때 특정 개인, 특정 환자군, 특정 사건을 다시 식별할 수 있는지가 판단 기준입니다.

## 검토 대상

- 데이터셋 요청 API와 요청 처리 함수
- 데이터셋 승인 API와 승인 처리 함수
- 데이터셋 생성 API와 생성 처리 함수
- 데이터셋 요청·승인 상태 저장소의 `PENDING`, `APPROVED`, `REJECTED`, `READY` 상태 전이
- 연구 데이터셋 버킷과 dataset object 메타데이터
- field allowlist, cohort threshold, suppression rule, 반복 추출 검토 절차

## 검토 대상 준식별자

본 가이드라인에서 특히 주의할 준식별자 범주는 다음과 같습니다.

- 날짜 정보: 진단일, 검사일, 입원·퇴원일, 추적 관찰일
- 인구통계 정보: 연령, 성별, 거주권역, 보험 유형
- 기관 맥락 정보: 병원 규모, 진료과, 병동, 지역
- 임상 맥락 정보: 희귀 진단명, 검사 조합, 치료 시퀀스, 시술 여부

이들 항목은 각각 단독으로는 무해해 보일 수 있으나, 희귀 질환이나 소규모 cohort에서는 조합만으로 재식별 위험이 높아질 수 있습니다.

## 공격 성립 조건

- requested field allowlist가 과도하게 넓습니다.
- small cell suppression, generalization, threshold 기준이 없습니다.
- 동일 연구 과제 또는 동일 연구자에 대한 반복 추출 이력이 누적 위험으로 관리되지 않습니다.
- 가명처리 품질 검토가 형식적인 승인 절차로 대체됩니다.

추가로, dataset 수명주기와 승인 상태 정의가 느슨할 때 위험이 커집니다. 본 구조에서는 `PENDING`, `APPROVED`, `REJECTED`, `READY` 상태가 존재하므로, 가명처리와 위험 검토는 `APPROVED`와 `READY` 사이에서 수행되어야 합니다.

## 위협 전개

1. 연구기관은 승인 가능한 목적과 범위를 근거로 dataset을 요청합니다.
2. 생성 파이프라인이 직접 식별자 제거만 수행하고 준식별자 조합 위험을 충분히 평가하지 않습니다.
3. 제공된 dataset은 외부 공개 데이터, 내부 보유 데이터, 반복 제공된 과거 dataset과 결합됩니다.
4. 희귀 질환, 세부 날짜, 기관 규모, 부서, 검사 조합을 통해 특정 개인이 재식별됩니다.

특히 반복 제공 시에는 한 번의 추출에서 제거한 정보가 다른 회차 dataset과 결합되며 다시 드러날 수 있습니다. 따라서 단일 요청 기준의 안전성만으로 충분하지 않습니다.

## 필수 예방 통제

- dataset 요청 단계에서 목적, 필드, 기간, cohort 크기를 함께 검토해야 합니다.
- 생성 단계에서 generalization, suppression, threshold 검사를 수행해야 합니다.
- 반복 추출 이력과 누적 위험을 승인 단계에서 다시 검토해야 합니다.
- 연구자는 원본 DB와 clinical file bucket에 직접 접근할 수 없어야 합니다.

필수 통제는 다음 두 층으로 구분해 설계하는 것이 바람직합니다.

- 요청·승인 통제: 목적 적합성, 최소 제공 원칙, 반복 추출 이력 검토
- 생성 통제: 직접 식별자 제거, 준식별자 일반화, small cell suppression, 산출물 검토

## 필수 탐지 통제

- 작은 코호트 또는 희귀 질환 중심 요청을 별도 검토 대상으로 분류합니다.
- 동일 연구자에 대한 반복 요청 패턴을 모니터링합니다.
- 승인된 범위를 벗어난 필드 조합이 생성 요청에 포함되면 경보를 발생시킵니다.

## 대응 요구사항

- 이미 제공된 dataset의 추가 배포를 중단합니다.
- 필요 시 dataset를 폐기하고 재생성합니다.
- 승인 기록과 가명처리 기준을 재검토하고 예외 승인 절차를 수정합니다.

## 검증 기준

- 직접 식별자 제거 외의 재식별 위험 저감 기준이 문서화되어 있어야 합니다.
- 반복 추출 시 누적 위험을 재평가하는 절차가 존재해야 합니다.
- 재식별 우려가 있는 요청은 승인 거부 또는 재설계가 가능해야 합니다.

추가로, 다음 산출물이 남아야 합니다.

- 요청 당시 승인된 필드 목록과 목적
- 적용된 가명처리 규칙 또는 일반화 규칙
- 예외 승인 여부와 그 근거

## 요구 증빙 및 검증 주기

| 증빙 항목 | 최소 내용 | 주기 |
|----------|----------|------|
| dataset 제공 기준서 | 허용 필드, 금지 필드, cohort threshold, suppression 기준 | 요청 유형 변경 시마다 |
| 상태 전이 검증 | `APPROVED` 이전 생성 차단, `READY` 이전 다운로드 차단 결과 | 매 릴리스 |
| 반복 추출 검토 기록 | 동일 연구자 또는 동일 연구 과제의 누적 요청 검토 결과 | 월간 |
| 산출물 표본 검토 | 생성된 dataset 표본에 대한 재식별 위험 검토 기록 | 월간 또는 배치 변경 시마다 |

## 배포 차단 조건

- 가명처리 기준이 “이름 제거” 수준에 머물면 배포를 중단합니다.
- 승인된 dataset이 원본 PHI와 사실상 동등한 상세도를 가지면 배포를 승인하지 않습니다.
