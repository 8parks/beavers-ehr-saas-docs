# T-09 Break-glass 남용

## 위협 개요

| 항목 | 내용 |
|------|------|
| 위험도 | 높음 |
| 관련 워크로드 | `S6`, `S7` |
| 관련 자산 | `A-07`, `A-08`, `A-09`, `A-10` |
| 관련 경계 | `TB-07`, `TB-09` |
| 대표 행위자 | SaaS 운영자, 장애 대응자, 임시 권한 사용자 |

Break-glass는 예외 경로입니다. 따라서 이 경로가 일상 운영 수단이 되면 설계상 최소 권한 원칙은 무의미해집니다. 본 위협은 악의적 내부자뿐 아니라 선의의 운영자 실수까지 포함합니다.

## 검토 대상

- 운영자 SSM 접근 역할
- 관리용 bastion EC2
- Session Manager 연결에 필요한 interface endpoint 집합
- CloudTrail 세션 시작 이벤트와 Session Manager 세션 로그
- 운영자 승인 티켓, 임시 권한 발급, 만료 확인 절차

## 공격 성립 조건

- 2인 승인 없이 SSM 세션 또는 admin EC2 접근이 가능함.
- standing privilege 또는 장기 비밀정보가 남아 있음.
- 세션 시작, 포트포워딩, DB 질의, secret 조회, key 사용 기록이 서로 연결되지 않음.
- 사용 종료 후 권한 자동 회수와 사후 검토가 없음.

## 위협 시나리오

1. 야간 장애나 데이터 이상 상황에서 운영자가 break-glass 접근을 요청하거나 직접 세션을 시작함.
2. 승인 절차, 만료 시간, 세션 연계 기록이 느슨하면 운영자는 SSM 터널과 관리용 bastion을 통해 데이터 평면에 도달함.
3. 세션 중 PHI 조회, 수동 SQL 실행, secret 조회, KMS 사용이 발생하더라도 티켓 번호와 세션 기록이 연결되지 않으면 사용 목적을 사후에 입증하기 어려움.
4. 이러한 경로가 반복되면 break-glass는 예외 경로가 아니라 일상 운영 수단으로 변질되고, 최소 권한 원칙이 무력화됨.

## 필수 예방 사항

- 신청, 승인, 발급, 사용, 종료, 사후 검토를 하나의 절차로 정의합니다.
- JIT access와 짧은 만료 시간을 사용합니다.
- 공유 계정 사용을 금지하고 개인 식별 가능한 주체만 사용합니다.
- 평상시 운영에는 break-glass 경로를 사용하지 않도록 별도 운영 절차를 둡니다.

## 필수 탐지 사항

- 세션 시작 이벤트, port forwarding, 관리용 bastion 접근을 별도 경보 대상으로 분류합니다.
- 세션 중 secret read, DB access, KMS usage가 발생하면 상관관계 분석이 가능해야 합니다.
- 운영 시간 외 break-glass 접근은 강화된 검토 대상으로 처리합니다.

## 대응 요구사항

- 세션을 종료하고 임시 자격증명을 즉시 폐기합니다.
- 해당 세션의 승인 티켓, 세션 시작 이벤트, 포트 포워딩 기록, 관련 DB 작업 증적을 즉시 보존합니다.
- 사용 목적, 수행 명령, 접근 데이터 범위, 승인자 정보를 사후 검토합니다.
- 동일 경로의 반복 사용 원인을 분석해 평상시 운영 절차를 개선합니다.

## 검증 기준

- break-glass는 승인 없이 개시될 수 없어야 합니다.
- 세션 로그, 명령 로그, 관련 secret/key usage 기록이 연결 가능해야 합니다.
- 운영 종료 후 권한이 자동 회수되어야 합니다.

## 운영 점검 항목 및 주기

| 증빙 항목 | 최소 내용 | 주기 |
|----------|----------|------|
| 승인 기록 | 요청자, 승인자 2인, 목적, 허용 시간, 접근 범위 | 사용 시마다 |
| 세션 증빙 | 세션 시작 이벤트, 세션 로그, port forwarding 여부, 대상 인스턴스 정보 | 사용 시마다 |
| 사후 검토 기록 | 수행 작업, 접근 데이터 범위, 개선 조치 여부 | 사용 시마다 |
| 정기 운영 검토 | break-glass 사용 빈도, 상시화 징후, 예외 승인 추세 | 월간 |
