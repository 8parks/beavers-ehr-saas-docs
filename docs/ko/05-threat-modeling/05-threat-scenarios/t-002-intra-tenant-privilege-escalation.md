# T-02 동일 테넌트 내부 권한 상승

## 위협 개요

| 항목 | 내용 |
|------|------|
| 위험도 | 높음 |
| 관련 워크로드 | `S1`, `S2`, `S4`, `S6` |
| 관련 자산 | `A-01`, `A-05`, `A-06`, `A-10` |
| 관련 경계 | `TB-02`, `TB-06`, `TB-07` |
| 대표 행위자 | 병원 관리자, 간호사, 연구기관 사용자, 운영자 |

동일 테넌트 내부 권한 상승은 cross-tenant 사고와 달리 외형상 정상 테넌트 내부 활동처럼 보이기 때문에 탐지가 어렵습니다. 본 가이드라인에서는 병원 관리자, 간호사, 연구자, 운영자 각각의 권한 범위를 명시적으로 분리해야 합니다.

## 검토 대상

- Cognito claim의 tenant, role, tenant type, group 속성
- OAuth scope 기반 API 권한 분리
- 환자 조회, 진료 기록 조회, 진료 기록 생성 애플리케이션 함수
- 데이터셋 요청, 승인, 생성 애플리케이션 함수
- 병원 관리자, 간호사, 연구자, SaaS 운영자에 대한 역할 매트릭스와 route-scope 매핑 표

## 공격 성립 조건

- 역할 검증이 단순 RBAC에 머무르고 patient relation 또는 업무 근거를 확인하지 않음.
- hospital-admin 기본 권한에 차트 조회 또는 진료 기록 작성이 포함됨.
- 연구자 또는 승인자가 `dataset.generate` 같은 내부 경로를 직접 호출할 수 있음.
- 운영 경로와 업무 경로가 동일한 IAM 역할 또는 동일 API 경로를 공유함.

## 위협 시나리오

1. 병원 관리자, 간호사, 연구자, 운영자 중 한 사용자가 정상 계정으로 로그인함.
2. 프런트엔드 호출 기록, 공개 API 명세, 과거 운영 요청을 통해 상세 차트 조회, 최종 처방 저장, 데이터셋 승인, 데이터셋 생성 경로를 확인함.
3. 서버가 `tenant_id`와 기본 role만 확인하고 환자 관계, 업무 근거, 승인자 여부, tenant type을 충분히 검증하지 않으면 요청이 정상 처리됨.
4. 사용자는 본래 허용되지 않아야 하는 차트 열람, 기록 작성, 승인, 생성 행위를 수행하고, 로그 상으로는 같은 병원 내부의 정상 활동처럼 보이게 됨.

## 필수 예방 사항

- 역할별 허용 행위를 문서화한 권한 매트릭스를 기준으로 서버 측 인가를 구현합니다.
- chart read/write에는 RBAC 외에 patient-level ABAC 또는 care-team 근거를 추가합니다.
- dataset 요청, 승인, 생성 경로는 서로 다른 역할과 서로 다른 API scope를 사용해야 합니다.
- break-glass 경로는 일반 운영 경로와 IAM 수준에서 분리합니다.

## 필수 탐지 사항

- hospital-admin, nurse, researcher, saas-operator의 차트 접근 이벤트를 별도 분류합니다.
- 승인자 역할이 dataset 생성 또는 다운로드를 직접 수행하는지 탐지합니다.
- 동일 사용자가 짧은 시간 내 여러 환자 차트에 비정상 접근하는 패턴을 탐지합니다.

## 대응 요구사항

- 계정 비활성화 또는 세션 무효화를 즉시 수행합니다.
- 권한 부여 과정, Cognito group, custom claim, API scope를 재검토합니다.
- 이미 수행된 차트 열람, 수정, 생성, 다운로드 범위를 조사합니다.

## 검증 기준

- hospital-admin은 통계와 계정 관리만 가능하고 상세 진료 기록 기본 접근은 거부되어야 합니다.
- nurse는 제한된 조회와 보조 입력만 허용되고 진단 확정 또는 처방 최종 저장은 거부되어야 합니다.
- researcher는 승인 없이 생성 또는 원본 조회 경로를 호출할 수 없어야 합니다.

현재 구현 검토에서는 환자 조회, 진료 기록 조회, 진료 기록 생성 경로가 `hospital-admin`을 허용하는지 우선 확인해야 합니다. 병원 관리자에게 상세 차트 조회 또는 기록 작성 기본 권한이 남아 있으면, 이는 설계 예외가 아니라 인가 모델의 결함으로 판정합니다.

검증은 최소한 다음 세 계층 수준에서 차단 테스트가 필요합니다.

- 인증 계층: role, tenant type, group, scope가 불일치하는 token 조합
- API 인가 계층: hospital-admin, nurse, researcher가 금지된 route를 직접 호출하는 경우
- 업무 규칙 계층: 담당 환자가 아니거나 승인 권한이 없는 사용자가 업무 행위를 시도하는 경우

## 운영 점검 항목 및 주기

| 증빙 항목 | 최소 내용 | 주기 |
|----------|----------|------|
| 역할 매트릭스 | 역할별 허용 행위, 금지 행위, 예외 승인 조건 | 매 릴리스 |

| 이상 접근 탐지 규칙 | hospital-admin, nurse, researcher의 차트 접근 탐지 규칙과 경보 표본 | 월간 |
