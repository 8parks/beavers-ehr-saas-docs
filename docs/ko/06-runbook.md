# 운영 가이드 (Runbook)

설계와 위협 모델링이 정의한 통제는 실제 운영 절차로 내려오지 않으면 유지되지 않습니다. 본 런북은 멀티테넌트 EHR SaaS 환경에서 반복적으로 수행되는 운영 작업과 사고 대응 절차를 표준화하기 위한 문서입니다.  
본 문서는 기술 팀, 보안 운영 팀, 테넌트 운영 담당자, 승인권자가 공통으로 참조하는 실행 기준으로 사용합니다.

::: warning
본 런북은 가상의 EHR SaaS를 기준으로 작성된 운영 가이드입니다. 실제 운영에 적용할 때는 병원별 계약, 보존 의무, 통지 의무, 조직 승인 체계에 맞춰 세부 절차를 보완해야 합니다.
:::

## 운영 원칙

- 평상시 운영 경로와 예외 경로를 구분합니다.
- PHI, 연구 데이터셋, 테넌트 매핑 정보는 동일한 운영 절차로 취급하지 않습니다.
- 모든 민감 작업은 요청, 승인, 실행, 검증, 종료, 증빙 보존의 순서를 따릅니다.
- 운영 절차는 위협 모델링의 `T-01`, `T-07`, `T-09`, `T-12`, `T-14` 통제와 일관되어야 합니다.
- 수동 작업이 필요한 경우에도 작업 결과를 사후에 재구성할 수 있는 수준의 증빙을 남겨야 합니다.

## 1. Tenant 라이프사이클 {#tenant-라이프사이클}

Tenant 라이프사이클 절차는 단순 계정 관리가 아니라 멀티테넌트 격리의 생성, 유지, 종료를 다루는 운영 절차입니다.  
특히 온보딩과 오프보딩은 `tenant_registry`, 계정 상태, 데이터 저장 위치, 키 접근 범위를 함께 다뤄야 합니다.

### 온보딩

#### 목적

신규 병원 또는 기관을 기존 테넌트와 격리된 상태로 등록하고, 초기 관리자 접근을 안전하게 제공합니다.

#### 트리거

- 신규 고객 계약 체결
- 파일럿 테넌트 생성 승인
- 기존 테넌트의 조직 분리 또는 신규 기관 추가

#### 사전 조건

- 고객 식별자, 계약 범위, 데이터 처리 범위가 확정되어 있어야 합니다.
- 테넌트 유형이 병원, 연구기관, SaaS 운영용 중 무엇인지 정의되어 있어야 합니다.
- 초기 관리자 역할, 승인자, 운영 책임자가 지정되어 있어야 합니다.

#### 절차

1. 테넌트 신청서와 승인 기록을 확인합니다.
2. `tenant_id`를 발급하고 중복 여부를 검증합니다.
3. `tenant_registry`에 테넌트 상태를 `PROVISIONING`으로 등록합니다.
4. 데이터 저장 구조를 생성합니다.
   - 데이터베이스 스키마 또는 해당하는 테넌트 논리 구획 생성
   - 임상 첨부파일 저장 경로 생성
   - 연구 데이터셋 저장 경로가 필요한 경우 별도 생성
5. 테넌트별 메타데이터를 등록합니다.
   - schema 이름
   - 임상 파일 prefix
   - tenant 유형
   - 생성 시각
   - 초기 상태
6. 초기 관리자 계정을 생성합니다.
   - Cognito 사용자 생성
   - 역할과 그룹 검증
   - MFA 요구 여부 확인
7. 초기 자격증명을 안전한 채널로 전달합니다.
   - 이메일 본문에 평문 비밀번호를 넣지 않습니다.
   - 임시 비밀번호 또는 초대 링크는 만료 시간을 가진 방식으로 전달합니다.
8. 교차 검증을 수행합니다.
   - 신규 관리자 토큰으로 자기 테넌트 접근이 가능한지 확인
   - 다른 테넌트 리소스 접근이 거부되는지 음성 테스트 수행
9. 이상이 없으면 `tenant_registry` 상태를 `ACTIVE`로 변경합니다.
10. 온보딩 완료 증빙을 보존합니다.

#### 필수 증빙

- 온보딩 승인 기록
- `tenant_id`와 매핑 정보 등록 기록
- 초기 관리자 계정 생성 기록
- MFA 및 역할 검증 결과
- cross-tenant 음성 테스트 결과
- `PROVISIONING -> ACTIVE` 상태 전이 기록

#### 완료 기준

- 신규 테넌트가 자기 테넌트 내에서만 정상 접근 가능합니다.
- 기존 테넌트와의 데이터 격리 검증이 완료되었습니다.
- 초기 관리자 자격증명 전달이 완료되었고, 평문 비밀정보가 남아 있지 않습니다.

#### 금지 사항

- `tenant_id`, schema, prefix를 수동 메모나 코드 상수로만 관리하지 않습니다.
- 관리자 계정을 공유 계정으로 발급하지 않습니다.
- cross-tenant 음성 테스트 없이 `ACTIVE` 상태로 전환하지 않습니다.

### 오프보딩

#### 목적

계약 종료 또는 서비스 종료 시 테넌트의 접근 경로를 차단하고, 데이터 반환 또는 파기 절차를 검증 가능한 방식으로 완료합니다.

#### 트리거

- 계약 종료
- 고객의 서비스 해지 요청
- 장기 중단 후 최종 파기 승인
- 법적 분쟁 종료 후 파기 전환

#### 사전 조건

- 데이터 반환, 보존, 파기 중 어떤 정책이 적용되는지 확정되어 있어야 합니다.
- 법정 보존 의무 또는 계약상 보존 조항이 검토되어 있어야 합니다.
- 고객 통지 담당자와 내부 승인자가 지정되어 있어야 합니다.

#### 절차

1. 오프보딩 요청과 승인 문서를 확인합니다.
2. `tenant_registry` 상태를 `SUSPENDED`로 변경합니다.
3. 모든 사용자 세션과 신규 접근을 차단합니다.
   - Cognito 사용자 비활성화
   - 활성 세션 무효화
   - 미사용 토큰/예외 권한 재검토
4. 데이터 반환이 필요한 경우 범위를 확정하고 export를 수행합니다.
   - export 대상 데이터 종류
   - 제공 형식
   - 전달 채널
   - 만료 시점
5. 파기 범위를 확정합니다.
   - 데이터베이스 스키마
   - 임상 첨부파일 객체
   - 연구 데이터셋 아티팩트
   - cache 또는 임시 export
   - 미만료 presigned URL
   - 관련 비밀정보 및 접근 권한
6. 데이터 파기 또는 격리를 수행합니다.
7. 키 접근과 secret 접근을 차단합니다.
8. 오프보딩 후 음성 테스트를 수행합니다.
   - 기존 사용자 계정 접근
   - 기존 토큰 접근
   - 기존 URL 접근
   - 연구 데이터셋 재다운로드 시도
9. 법정 보존 대상이 남아 있다면 보존 사유, 만료 시점, 접근 제한 조건을 별도 기록합니다.
10. `tenant_registry` 상태를 `OFFBOARDED`로 전환합니다.

#### 필수 증빙

- 오프보딩 승인 기록
- 세션 무효화 및 계정 비활성화 기록
- export 제공 기록 또는 파기 기록
- S3 / DB / dataset artifact 삭제 결과
- post-offboarding 음성 테스트 결과
- `SUSPENDED -> OFFBOARDED` 상태 전이 기록

#### 완료 기준

- 오프보딩 대상 테넌트는 더 이상 API, 데이터셋, 첨부파일, 예외 경로로 접근할 수 없습니다.
- 보존 또는 파기 상태가 자산 단위로 설명 가능합니다.
- 남아 있는 예외 보유 자산이 있다면 사유와 만료 시점이 문서화되어 있습니다.

#### 금지 사항

- 계정 비활성화만 하고 세션 무효화를 생략하지 않습니다.
- dataset artifact, cache, 미만료 presigned URL을 파기 범위에서 제외하지 않습니다.
- 보존 사유가 없는 잔존 데이터를 남겨두지 않습니다.

### Tenant 중지

#### 목적

분쟁, 미납, 보안 사고, 조사 필요성 등으로 인해 테넌트를 즉시 중지해야 할 때 신규 접근을 차단하고 데이터 상태를 보존합니다.

#### 트리거

- Cross-Tenant 접근 의심
- PHI 유출 의심
- 법적 분쟁
- 미납 또는 계약 위반

#### 절차

1. 중지 사유와 승인자를 확인합니다.
2. `tenant_registry` 상태를 `SUSPENDED`로 전환합니다.
3. 신규 로그인과 기존 세션을 차단합니다.
4. 데이터는 파기하지 않고 보존 상태로 유지합니다.
5. 중지 중 허용되는 운영 행위를 명시합니다.
   - 고객 지원 연락
   - 조사 목적 제한 조회
   - 법적 보존 조치
6. 중지 해제 또는 오프보딩 전환 조건을 문서화합니다.

#### 필수 증빙

- 중지 승인 기록
- 상태 전이 기록
- 세션 차단 기록
- 중지 해제 또는 오프보딩 전환 판단 기록

#### 완료 기준

- 테넌트는 신규 접근 불가 상태여야 합니다.
- 데이터는 조사 및 보존 목적에 맞게 유지되어야 합니다.

## 2. 자격증명 및 키 관리 {#자격증명-및-키-관리}

이 섹션은 비밀정보와 키를 “설정값”이 아니라 운영 자산으로 다룹니다.  
정기 회전과 비상 회전 절차는 분리해야 하며, 회전 이후에도 서비스 기능과 검증 기능이 유지되어야 합니다.

### 키 교체

#### 목적

정기적인 키 및 자격증명 회전을 통해 장기 노출 위험을 낮추고, 최소 권한 상태를 유지합니다.

#### 적용 대상

- DB 접속 자격증명
- 애플리케이션 secret
- 연구 데이터셋 처리 경로 secret
- 서명 검증에 사용되는 키 참조 정보
- KMS customer managed key의 회전 정책

#### 절차

1. 회전 대상과 회전 주기를 확인합니다.
2. 영향 워크로드를 식별합니다.
   - 환자 조회
   - 진료 기록 조회/작성
   - 데이터셋 생성
   - 운영자 예외 접근
3. 회전 전 백아웃 계획과 검증 계획을 준비합니다.
4. 새로운 자격증명 또는 키 버전을 준비합니다.
5. 애플리케이션이 새 값으로 정상 동작하는지 단계적으로 검증합니다.
6. 구 버전 사용을 중단합니다.
7. 구 버전 참조가 남아 있지 않은지 로그와 설정을 다시 점검합니다.

#### 서명 키 회전 시 추가 원칙

- 새로 생성되는 진료 기록은 새 서명 체계를 사용합니다.
- 기존 기록의 검증 가능성은 유지되어야 합니다.
- 회전 직후 일정 기간은 구 버전 검증 정보가 남아 있어야 합니다.

#### 필수 증빙

- 회전 계획서
- 영향 분석 기록
- 회전 수행 로그
- 회전 후 기능 검증 결과
- 구 버전 폐기 또는 비활성화 기록

#### 완료 기준

- 회전 이후 모든 워크로드가 정상 동작합니다.
- 구 버전 자격증명 또는 키 사용이 종료되었습니다.
- 예외적으로 남겨둔 구 버전이 있다면 사유와 만료 시점이 명시되어 있습니다.

### 비상 회전 절차

#### 목적

유출이 의심되거나 실제 노출이 확인된 비밀정보와 키를 즉시 교체하고 추가 악용을 차단합니다.

#### 트리거

- secret 노출 스캔 탐지
- 로그 또는 코드 저장소 내 비밀정보 유출
- 운영자 계정 탈취 의심
- 비정상적인 KMS 사용 탐지

#### 절차

1. 노출 범위를 1차 분류합니다.
   - DB 자격증명 노출
   - 애플리케이션 secret 노출
   - 운영자 예외 접근 secret 노출
   - 키 사용 권한 오남용
2. 문제 자격증명 또는 키의 신규 사용을 즉시 차단합니다.
3. 관련 세션과 연결된 임시 자격증명을 폐기합니다.
4. 새 자격증명 또는 대체 키를 발급합니다.
5. 영향을 받는 워크로드 설정을 갱신합니다.
6. 정상 동작과 추가 악용 차단 여부를 검증합니다.
7. 사고 티켓에 회전 시각, 대상, 영향 범위, 후속 점검 결과를 기록합니다.

#### 필수 증빙

- 사고 티켓
- 회전 실행 로그
- 세션 폐기 기록
- 회전 후 검증 결과
- 관련 로그 보존 기록

#### 완료 기준

- 노출된 자격증명 또는 키로 더 이상 접근할 수 없습니다.
- 영향 워크로드가 복구되었습니다.
- 회전 원인과 재발 방지 조치가 등록되었습니다.

## 3. 사고 대응 {#사고-대응}

본 섹션은 위협 모델링의 고위험 시나리오를 실제 대응 절차로 변환한 것입니다.  
세 가지 절차 모두 공통으로 다음 순서를 따릅니다.

1. 탐지
2. 차단
3. 증적 보존
4. 영향 범위 산정
5. 복구
6. 사후 개선

### Cross-Tenant 접근

#### 목적

한 테넌트 사용자가 다른 테넌트의 PHI, 첨부파일, 연구 산출물에 접근했거나 접근했을 가능성이 있는 상황을 즉시 차단합니다.

#### 즉시 조치

- 관련 토큰과 세션을 무효화합니다.
- 필요 시 해당 tenant 또는 API 경로를 일시 중지합니다.
- presigned URL이 관련된 경우 즉시 폐기 또는 만료 처리합니다.

#### 1시간 내 조치

1. 영향 tenant와 자산 범위를 1차 산정합니다.
2. 관련 요청 로그, 앱 감사 로그, 데이터 접근 로그를 보존합니다.
3. `tenant_registry`, tenant context, RLS, prefix 검증 상태를 역추적합니다.
4. 동일 경로에서 추가 시도가 있는지 확인합니다.

#### 후속 조치

- 원인이 claim 검증, tenant mapping, DB context, object key 제약 중 어디에 있는지 분류합니다.
- 수정 전까지 동일한 유형의 기능 배포를 중단합니다.
- 재현 테스트와 음성 테스트를 보강합니다.

#### 필수 증빙

- 사고 식별 시각
- 차단 시각
- 영향 tenant 및 자산 목록
- 로그 보존 위치
- 재현 및 수정 검증 결과

### PHI 유출

#### 목적

PHI가 외부로 유출되었거나 유출 정황이 확인된 경우 추가 반출을 막고, 영향 범위와 통지 판단을 지원합니다.

#### 15분 내 조치

1. 추가 반출 경로를 차단합니다.
2. 관련 세션과 토큰을 폐기합니다.
3. 필요 시 tenant를 `SUSPENDED` 상태로 전환합니다.
4. 문제 URL, export, dataset delivery 경로를 폐기합니다.

#### 60분 내 조치

1. CloudTrail, 앱 감사 로그, SSM 세션 로그, KMS 사용 로그, S3 data event를 보존합니다.
2. 영향 tenant, 환자 수, 자산 유형을 1차 산정합니다.
3. 유출 경로를 분류합니다.
   - token 재사용
   - dataset 오제공
   - presigned URL 남용
   - break-glass 오남용
   - secret 노출
4. 통지 필요 여부를 판단할 수 있도록 사고 기록을 정리합니다.

#### 사후 조치

- 통지 기준과 승인 절차에 따라 대내외 보고를 수행합니다.
- 원인 통제와 재발 방지 작업을 별도 추적합니다.
- tabletop 또는 사후 훈련에서 동일 시나리오를 재검증합니다.

#### 필수 증빙

- 초기 차단 기록
- 증적 확보 체크리스트
- 영향 범위 산정 문서
- 통지 판단 기록
- 재발 방지 작업 추적표

### 자격증명 유출

#### 목적

DB 비밀번호, secret, 운영자 예외 자격증명, 키 접근 권한이 유출되었을 때 데이터 평면 우회를 차단합니다.

#### 즉시 조치

- 관련 자격증명 또는 권한을 비활성화합니다.
- 활성 세션과 연결된 임시 자격증명을 폐기합니다.
- 문제 경로를 사용하는 워크로드를 제한 또는 일시 중지합니다.

#### 단계별 절차

1. 노출 유형을 분류합니다.
   - DB credential
   - 애플리케이션 secret
   - 운영자 예외 자격증명
   - 키 사용 권한
2. 비상 회전 절차를 실행합니다.
3. 노출 전후 사용 로그를 비교해 악용 여부를 판단합니다.
4. 해당 자격증명으로 접근 가능한 데이터 평면 범위를 산정합니다.
5. 회전 후 정상 기능을 검증합니다.

#### 필수 증빙

- 노출 식별 기록
- 회전 수행 로그
- 사용 이력 비교 결과
- 영향 자산 목록
- 복구 검증 결과

## 4. 정기 점검 및 검증 {#정기-점검-및-검증}

정기 점검은 보안 통제가 설계대로 남아 있는지를 확인하는 절차입니다.  
월간 점검은 운영 드리프트를 찾는 데 집중하고, 취약점 스캐닝은 노출면 점검에, 침투 테스트는 설계 가정 검증에 집중합니다.

### 월간 점검

#### 점검 항목

- `tenant_registry` 정합성
- 비활성 tenant 접근 시도 존재 여부
- break-glass 사용 빈도와 승인 기록
- secret 접근 이벤트 이상치
- KMS 사용 패턴 이상치
- 앱 감사 로그와 CloudTrail 수집 상태
- presigned URL 발급 정책 드리프트 여부
- 오프보딩 잔존 자산 존재 여부

#### 수행 절차

1. 월간 점검 체크리스트를 생성합니다.
2. 테넌트 상태, 계정 상태, 로그 수집 상태를 확인합니다.
3. 예외 항목을 심각도별로 분류합니다.
4. 즉시 수정 항목과 다음 릴리스 보완 항목을 분리합니다.
5. 점검 결과를 월간 운영 보고서로 남깁니다.

#### 산출물

- 월간 점검 체크리스트
- 시정 조치 목록
- 미해결 리스크 목록

### 취약점 스캐닝

#### 목적

공개 노출면, 애플리케이션 종속성, 운영 구성의 취약 상태를 조기에 탐지합니다.

#### 범위

- 웹 진입점
- API 진입점
- 컨테이너 또는 빌드 아티팩트
- 코드 저장소 내 secret 노출 여부
- IAM / 네트워크 구성 드리프트

#### 수행 원칙

- 프로덕션과 유사한 환경에서 수행합니다.
- PHI 원본을 사용하는 스캔은 금지합니다.
- 스캔 결과는 심각도와 악용 가능성으로 구분해 처리합니다.
- false positive는 별도 검토 기록 없이 무시하지 않습니다.

#### 필수 증빙

- 스캔 시점과 범위
- 탐지 결과
- 조치 여부
- 재검증 결과

### 침투 테스트

#### 목적

문서화된 보안 가정이 실제로 유지되는지 독립적으로 검증합니다.

#### 우선 검증 범위

- Cross-Tenant 접근 시도
- 동일 테넌트 내부 권한 상승
- presigned URL 재사용 및 객체 경로 오남용
- 연구 데이터셋 재식별 가능성
- break-glass 경로 남용 가능성
- 사고 대응 절차의 실행 가능성

#### 수행 주기

- 주요 아키텍처 변경 전후
- 최소 연 1회
- 고위험 사고 또는 중대 결함 이후 재수행

#### 결과 처리 원칙

- `높음` 위험 결과는 수정 또는 예외 승인 없이 운영 유지 근거로 사용하지 않습니다.
- 위협 모델링 문서와 충돌하는 결과가 나오면 위협 시나리오를 갱신합니다.
- 침투 테스트 결과는 런북, 체크리스트, 운영 절차에 반영되어야 합니다.

#### 필수 증빙

- 테스트 범위와 전제
- 발견 사항 목록
- 재현 단계
- 수정 상태
- 재테스트 결과
