# T-03 의료 기록 무단 수정 및 전자서명 우회

## 위협 개요

| 항목 | 내용 |
|------|------|
| 위험도 | 높음 |
| 관련 워크로드 | `S2` |
| 관련 자산 | `A-01`, `A-02`, `A-08`, `A-09` |
| 관련 경계 | `TB-04`, `TB-05` |
| 대표 행위자 | 과도 권한 사용자, 손상된 애플리케이션 경로, 운영자 예외 접근 |

의료 기록은 법적 증적과 진료 연속성의 기반이다. 따라서 무단 수정이 가능하거나 서명 검증 경로가 형식적으로만 존재하면, 기밀성 이상으로 심각한 운영·법적 리스크가 발생한다.

## 검토 대상

- 진료 기록 생성 함수의 record 저장, attachment metadata 고정, KMS `Sign` 호출 경로
- 진료 기록 조회 함수의 조회 결과 반환 전 서명 검증 또는 검증 결과 처리 경로
- 기록 서명용 KMS 키
- 임상 파일 버킷의 `object_key`, `version_id`, `sha256_hash` 관리 방식
- Aurora record DB와 record change history 보존 경로

## 공격 성립 조건

- write 권한이 필요한 역할보다 넓게 배정되어 있다.
- record hash, attachment hash, KMS 서명이 동일한 canonical payload를 기준으로 계산되지 않는다.
- 조회 시 `Verify`를 수행하지 않고 저장된 서명 값만 존재한다.
- 변경 이력이 별도 보존되지 않아 overwrite가 가능하다.

## 위협 전개

1. 공격자는 record 본문, 첨부파일 메타데이터, 작성자 정보 중 하나를 수정한다.
2. 시스템이 hash 또는 signature를 다시 계산하지 않거나 검증하지 않으면 수정이 은폐된다.
3. 감사 로그에 변경 전후 차이와 변경 주체가 남지 않으면 사후 입증이 어려워진다.

## 필수 예방 통제

- 진료 기록 본문과 첨부파일 메타데이터는 canonical payload 기준으로 서명해야 한다.
- `record_id`, `tenant_id`, `patient_id`, 작성자 식별자, attachment metadata를 서명 대상에 포함한다.
- 첨부파일은 `object_key`, `version_id`, `sha256_hash`를 함께 고정한다.
- 변경 이력은 별도 테이블 또는 append-only 로그에 남겨 overwrite만으로는 증적이 사라지지 않게 한다.

## 필수 탐지 통제

- 조회 또는 감사 시 서명 검증 실패 이벤트를 별도 경보로 처리한다.
- 동일 사용자의 비정상 write 빈도, 대량 수정, 삭제 시도를 탐지한다.
- attachment hash 불일치와 version drift를 감시한다.

## 대응 요구사항

- 영향 record와 attachment 범위를 식별하고 읽기 경로를 제한한다.
- 마지막 정상 상태와 비교하여 복구 여부를 판단한다.
- 법적 보존 대상일 경우 원본 증적을 별도 보존한다.

## 검증 기준

- 저장 직후와 조회 시점 모두에서 서명 검증 경로가 존재해야 한다.
- attachment metadata 변경만으로도 무결성 검증이 실패해야 한다.
- 운영자 수동 수정이 필요한 경우에도 별도 승인과 추적 가능한 경로만 허용해야 한다.

구현 점검 시에는 저장 경로의 `Sign` 호출만 확인해서는 충분하지 않다. 조회 경로가 저장된 서명 값의 존재만 확인하는지, 또는 실제로 KMS `Verify` 또는 동등한 검증 절차를 수행하는지까지 확인해야 한다.

## 요구 증빙 및 검증 주기

| 증빙 항목 | 최소 내용 | 주기 |
|----------|----------|------|
| canonical payload 정의서 | `record_id`, `tenant_id`, `patient_id`, 작성자, attachment metadata 포함 여부 | 설계 변경 시마다 |
| 서명 검증 시험 결과 | 정상 조회, 본문 변조, attachment metadata 변조 시 검증 결과 | 매 릴리스 |
| 변경 이력 표본 | record 수정 전후와 행위자 식별이 가능한 change log 표본 | 월간 |
| key usage 로그 | 기록 서명용 KMS 키에 대한 `Sign`, `Verify` 사용 주체와 빈도 | 월간 |

## 배포 차단 조건

- 전자서명이 선택 기능으로만 존재하고 실제 검증에 사용되지 않으면 배포를 중단한다.
- record 본문과 attachment metadata가 서로 다른 기준으로 서명되면 배포를 승인하지 않는다.
