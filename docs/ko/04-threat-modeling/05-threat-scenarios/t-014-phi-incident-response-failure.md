# T-14 PHI 유출 사고 대응 실패

## 위협 개요

| 항목 | 내용 |
|------|------|
| 위험도 | 높음 |
| 관련 워크로드 | `S1`, `S2`, `S3`, `S6` |
| 관련 자산 | `A-01`, `A-02`, `A-03`, `A-07`, `A-09`, `A-10` |
| 관련 경계 | `TB-05`, `TB-06`, `TB-07`, `TB-08`, `TB-09` |
| 대표 행위자 | 모든 데이터 접근 주체, 사고 대응 조직, 운영자 |

PHI 유출 자체뿐 아니라, 유출 의심 상황에서 적시에 차단·조사·통지·재발방지를 수행하지 못하는 상태도 독립 위협으로 관리해야 한다.

## 검토 대상 구현

- CloudTrail, app audit, SSM 세션 로그, KMS usage log, S3 data event 수집 경로
- 세션 무효화, presigned URL 폐기, tenant 일시 중지, key/secret 회전 절차
- 사고 지휘 체계, 의사결정 권한, 통지 승인 절차, 증적 보존 절차
- dataset 제공 경로와 break-glass 경로에 대한 긴급 차단 절차

## 공격 성립 조건

- 사고 심각도 분류 체계와 의사결정 권한자가 정의되지 않는다.
- 세션 폐기, secret 회전, dataset URL 폐기, 고객 통지 순서가 준비되지 않는다.
- 기술 팀, 보안 팀, 운영 팀, 법무 또는 대외 대응 역할 분리가 없다.
- 조사 과정에서 필요한 감사 증적이 즉시 확보되지 않는다.

## 위협 전개

1. token 탈취, URL 남용, 운영자 오남용, 오제공 등으로 유출 징후가 발생한다.
2. 조사자는 영향 환자와 tenant를 즉시 특정하지 못한다.
3. 추가 반출 차단 조치가 지연되고, 통지와 보고 판단도 늦어진다.
4. 사건은 종료되더라도 구조적 원인이 남아 재발 가능성이 유지된다.

## 필수 예방 통제

- PHI 사고 대응 runbook을 기술 조치와 운영 조치로 분리해 문서화한다.
- 사고 심각도와 책임자, 승인자, 통지 기준을 미리 정의한다.
- 세션 회수, key/secret 회전, URL 폐기, tenant 알림 절차를 표준화한다.
- 훈련 가능한 tabletop 시나리오를 운영한다.

## 필수 탐지 통제

- 이상 다운로드, 이상 URL 사용, 이상 secret read, 이상 break-glass 접근을 사고 트리거로 정의한다.
- 조사 지연 또는 증적 누락 자체를 경보 대상으로 본다.
- 사고 대응 중 발생한 추가 운영 행위도 별도 감사 대상으로 기록한다.

## 대응 요구사항

- 사고 식별 후 15분 이내에 추가 반출 차단, 세션 회수, 문제 URL 폐기, 필요 시 tenant 일시 중지를 수행한다.
- 60분 이내에 CloudTrail, app audit, SSM, KMS, S3 관련 증적을 보존하고 영향 tenant와 자산 범위를 1차 산정한다.
- 영향 범위, 원인, 증적, 통지 대상, 개선 항목을 병렬로 정리한다.
- 사건 종료 후 재발 방지 조치를 등록하고 검증 일정까지 포함한다.

## 검증 기준

- PHI 유출 사고 대응 runbook이 기술 단계와 운영 단계로 나뉘어 있어야 한다.
- 사고 대응 도중 수행되는 break-glass, secret rotation, URL 폐기 이벤트가 모두 감사 가능해야 한다.
- 주기적인 모의훈련 결과가 문서화되어야 한다.

## 요구 증빙 및 검증 주기

| 증빙 항목 | 최소 내용 | 주기 |
|----------|----------|------|
| 사고 대응 절차서 | containment, evidence preservation, notification, recovery 단계와 책임자 | 분기별 검토 |
| tabletop 훈련 결과 | 시나리오, 의사결정 지연, 누락 통제, 개선 항목 | 분기별 |
| 증적 확보 체크리스트 | CloudTrail, app audit, SSM, KMS, S3, Cognito별 확보 순서와 보존 위치 | 분기별 검토 |
| 사후 개선 추적표 | 사고 또는 훈련 후 등록된 개선 조치와 완료 상태 | 사고 또는 훈련 시마다 |

## 배포 차단 조건

- PHI 유출 사고 대응 절차가 없다면 배포를 중단한다.
- 영향 범위 산정과 통지 기준을 설명할 수 없으면 배포를 승인하지 않는다.
