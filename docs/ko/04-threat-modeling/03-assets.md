# 자산 식별

## 자산 위험도 기준

자산 위험도는 자산이 손상되었을 때의 사업 영향, 환자 안전 영향, 규제 영향, 복구 난이도를 종합하여 `높음`, `중간`, `낮음`으로 구분한다.

위험도는 자산 자체의 중요도와 더불어, 해당 자산이 다른 통제의 전제가 되는지 여부를 함께 반영한다. 예를 들어 `tenant_registry`, Cognito claim, KMS key는 직접 PHI를 저장하지 않더라도, 이 자산이 오염되면 PHI 통제가 연쇄적으로 실패할 수 있으므로 높은 위험도로 분류한다.

|--------|------|---------------|--------|----------|
| `A-01` | 환자 PHI 및 e-PHI | Aurora 기반 환자/진료 기록 데이터베이스 | 높음 | cross-tenant 노출, 무단 수정, 비인가 반출 |
| `A-02` | 진료 첨부파일 | 임상 파일 저장 S3 버킷 | 높음 | 객체 경로 오남용, 공개 노출, 삭제 |
| `A-03` | 연구 데이터셋 | 연구 데이터셋 저장 S3 버킷 | 높음 | 재식별, 승인 우회, 과다 제공 |
| `A-04` | Tenant 컨텍스트 메타데이터 | tenant registry 데이터 저장소 | 높음 | 잘못된 schema 매핑, prefix 충돌, 상태 오염 |

| `A-06` | 인증 및 인가 컨텍스트 | Cognito JWT, authorizer context | 높음 | token 재사용, claim 불일치, 역할 오남용 |
| `A-07` | 데이터베이스 자격증명 및 비밀정보 | AWS Secrets Manager | 높음 | secret 유출, 비상 접근 오남용 |
| `A-08` | 암호화 및 전자서명 키 | AWS KMS | 높음 | 비인가 복호화, 가짜 서명, key policy 남용 |
| `A-09` | 감사 로그 및 조사 증적 | CloudTrail, CloudWatch, audit S3 | 높음 | 누락, 변조, 상관관계 상실 |
| `A-10` | 운영자 비상 접근 경로 | SSM Session Manager, admin EC2 | 중간 | 승인 우회, standing privilege, 목적 외 사용 |

## 자산 분류 해석

자산은 용도에 따라 세 개의 군으로 나뉜다.

- 데이터 자산: PHI, 첨부파일, 연구 데이터셋
- 통제 자산: `tenant_registry`, 인증/인가 컨텍스트, KMS key, secret
- 증적 자산: 감사 로그, 운영자 세션 로그, 승인 상태 기록

이 구분은 보호 수준을 달리하기 위한 것이 아니라, 설계 검토 시 “무엇이 직접 보호 대상인가”와 “무엇이 보호 체계의 전제인가”를 구분하기 위해 사용한다.

## 자산 보호 요구사항

### PHI 및 e-PHI

`A-01`, `A-02`는 본 가이드라인의 가장 높은 민감도 자산군이다. 저장, 전송, 조회, 수정, 반출, 복구 경로 모두에서 별도 통제가 필요하다.

### Tenant 컨텍스트 자산

`A-04`, `A-06`은 직접 PHI는 아니지만, 이 값이 오염되면 PHI 보호 통제가 무력화된다. 따라서 설계상 루트 오브 트러스트로 취급한다. 특히 `tenant_id`, `schema_name`, `clinical_file_prefix`, `tenant_type`, `group`, `scope`는 여러 방어 계층에서 재사용되므로 정합성 검증이 필수다.

### 운영 및 조사 자산

`A-07`, `A-08`, `A-09`, `A-10`은 데이터 평면을 직접 저장하지 않아도, 손상되면 전체 보안 체계가 붕괴할 수 있다. 운영자 접근, key usage, 감사 증적은 별도 자산군으로 분류해 관리한다. 특히 조사 증적의 연속성이 깨지면 기밀성 또는 무결성 사고의 범위 산정 자체가 불가능해진다.
