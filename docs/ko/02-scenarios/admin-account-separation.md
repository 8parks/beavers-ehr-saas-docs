# 시나리오: 관리 계정 & 앱 계정 분리 운영

## 1. 비즈니스 로직 정의

보안 서비스와 자동화 오케스트레이션을 담당하는 관리 계정과 실제 애플리케이션 리소스가 위치한 멤버 계정이 어떻게 협력하여 보안 이상에 대응하는지를 설명합니다.

## 2. 보안 목표

- 관리 계정과 멤버 계정은 명확한 책임 경계를 가집니다.
- 플랫폼 관리자는 테넌트 PHI에 직접 접근하지 않습니다.
- 계정 간 신뢰 관계는 최소 필요 권한으로 제한됩니다.
- 모든 계정 간 활동은 감사 가능합니다.

## 3. 계정 역할 분리

| 계정 | 포함 리소스 | 접근 가능 대상 |
|------|-----------|--------------|
| 관리 계정 | Config, Security Hub, EventBridge, Lambda, CloudTrail 집계 | 보안 결과, 메타데이터 |
| 멤버 계정 | 애플리케이션 리소스, PHI, 대응 런북 | 테넌트 PHI (애플리케이션 레이어만) |

## 6. 보안 컨트롤

| 컨트롤 | 구현 방법 | 관련 위협 |
|--------|---------|---------|
| 계정 경계 | AWS 멀티 계정 구조 | 관리자 PHI 직접 접근 |
| 최소 권한 역할 | 대응 IAM 역할 최소 권한 | T-009 대응 역할 남용 |
| 계정 간 감사 | 계정별 CloudTrail + 중앙 집계 | 추적성 |

## 7. 설계 체크리스트

- [ ] 관리 계정에서 멤버 계정 PHI에 직접 접근하는 경로가 없습니다.
- [ ] 계정 간 IAM 역할이 최소 권한으로 설정되어 있습니다.
- [ ] 모든 AssumeRole 이벤트가 CloudTrail에 기록됩니다.
- [ ] 멤버 계정의 대응 역할 사용이 모니터링됩니다.
- [ ] 비상 접근(break-glass) 절차가 문서화되어 있습니다.

## 8. 미해결 질문

- AWS Organizations SCP로 멤버 계정에 적용할 제약 설계
- 비상 접근 시나리오에서의 계정 경계 처리
- 테넌트 수가 증가할 때 계정 온보딩 자동화 방법
