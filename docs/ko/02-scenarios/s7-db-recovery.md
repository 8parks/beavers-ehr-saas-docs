# S7. 데이터베이스 장애 및 복구

RDS/Aurora에 장애가 발생했을 때 어떻게 복구하며, 복구 과정에서 테넌트 격리와 데이터 무결성을 어떻게 유지할 것인지 다룹니다.

## 비즈니스 흐름

데이터베이스 장애는 모든 테넌트에 영향을 줄 수 있습니다. 복구 과정에서 테넌트 간 데이터가 섞이거나, 일부 테넌트의 데이터만 복구되는 상황을 방지해야 합니다.

<!-- 작성 예정: 장애 유형별 복구 흐름 -->

## 보안 목표

- 복구 과정에서 테넌트 격리가 유지되어야 합니다.
- 복구된 데이터의 무결성이 검증되어야 합니다.
- 복구 활동이 감사 로그로 기록되어야 합니다.

## 장애 유형

<!-- 작성 예정 -->

| 유형 | 설명 | 복구 전략 |
|------|------|---------|
| 인스턴스 장애 | RDS 인스턴스 비정상 종료 | Aurora 자동 페일오버 |
| 데이터 손상 | 스키마 또는 데이터 손상 | 스냅샷 복구 |
| 논리적 삭제 | 실수로 인한 데이터 삭제 | Point-in-time 복구 |

## 보안 체크리스트

- [ ] Aurora Multi-AZ 또는 자동 페일오버가 설정되어 있습니다.
- [ ] 정기 스냅샷이 암호화된 상태로 보관됩니다.
- [ ] 복구 절차가 문서화되고 주기적으로 테스트됩니다.
- [ ] 복구 활동이 CloudTrail에 기록됩니다.

## 미해결 질문

- RTO(복구 목표 시간), RPO(복구 목표 시점) 요구사항 정의
- 멀티 리전 복구 전략 필요 여부
