## 2. 보안 목표

- 인가된 사용자만 해당 테넌트의 환자 기록에 접근할 수 있어야 합니다.
- 역할에 따라 접근 가능한 데이터 범위가 제한되어야 합니다.
- 조회 이벤트는 감사 로그로 기록되어야 합니다.
- 만료되거나 취소된 토큰으로는 접근이 불가능해야 합니다.

## 3. 기능 흐름

1. 사용자가 로그인합니다.
2. 사용자가 환자를 검색합니다.
3. 환자 기록 목록이 반환됩니다.
4. 사용자가 특정 기록을 선택하여 조회합니다.
5. 조회 이벤트가 기록됩니다.

## 6. AWS 서비스 및 책임

| AWS 서비스 | 역할 |
|-----------|------|
| Amazon Cognito | 인증 및 JWT 발급 |
| Amazon API Gateway | API 진입점 |
| AWS Lambda | 테넌트 검증 및 데이터 조회 |
| Amazon RDS/Aurora PostgreSQL | PHI 저장 및 조회 |
| AWS KMS | 저장 데이터 복호화 |
| AWS CloudTrail | 접근 이벤트 감사 |

## 7. 보안 컨트롤

| 컨트롤 | 구현 방법 | 관련 위협 |
|--------|---------|---------|
| 테넌트 격리 | JWT tenant_id 검증, 스키마 격리 | T-001 크로스 테넌트 접근 |
| 역할 기반 접근 제어 | JWT role 클레임 검증 | T-002 과도한 권한 접근 |
| 토큰 검증 | JWT 서명 및 만료 검증 | T-005 손상된 토큰 |

## 8. 컴플라이언스 매핑

| 요구사항 범주 | 설계 대응 |
|--------------|---------|
| 접근 제어 | 역할 및 테넌트 기반 인가 |
| 감사 추적성 | PHI 조회 이벤트 기록 |

## 9. 위협 및 완화

| 위협 | 완화 방법 |
|------|---------|
| T-001 크로스 테넌트 접근 | 테넌트 컨텍스트 검증, 스키마 격리 |
| T-005 손상된 토큰 | JWT 만료, 서명 검증 |

## 10. 설계 체크리스트

- [ ] JWT tenant_id 및 role 클레임 검증
- [ ] 올바른 테넌트 스키마에서만 데이터 조회
- [ ] 조회 이벤트 감사 로그 생성
- [ ] 응답에 불필요한 PHI 필드 미포함
