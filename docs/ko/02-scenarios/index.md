# 시나리오 카탈로그

설계가 실제로 동작하는지는 구체적인 흐름을 따라가봐야 알 수 있습니다. 각 시나리오는 하나의 비즈니스 또는 운영 상황을 중심으로, 아키텍처가 그 흐름을 어떻게 보호하는지를 설명합니다.

## 비즈니스 시나리오

병원 실사용자 관점의 핵심 흐름입니다.

| 시나리오 | 핵심 보안 목표 |
|---------|--------------|
| [S1. 환자 진료 기록 작성](./patient-record-creation) | 인가된 의사만 기록 작성, KMS 서명, 감사 로그 |
| [S2. 환자 진료 기록 조회](./patient-record-retrieval) | 테넌트 격리, 역할 기반 접근 제어, 감사 |
| [S3. 연구 데이터셋 생성](./research-dataset-generation) | 가명처리, 원본 PHI 접근 차단, 감사 |

## 운영 시나리오

SaaS 플랫폼 운영자 관점의 흐름입니다.

| 시나리오 | 핵심 보안 목표 |
|---------|--------------|
| [S4. Tenant 온보딩](./s4-tenant-onboarding) | 격리된 테넌트 환경 생성, 초기 자격증명 안전 전달 |
| [S5. Tenant 오프보딩](./s5-tenant-offboarding) | 접근 차단, 데이터 반환 또는 파기, 키 폐기 |

## 비상 / 장애 대응 시나리오

일반적인 운영 범위를 벗어난 예외 상황입니다.

| 시나리오 | 핵심 보안 목표 |
|---------|--------------|
| [S6. 운영자 비상 접근 (Break-glass)](./s6-break-glass) | 최소 권한, 즉시 기록, 사후 검토 |
| [S7. 데이터베이스 장애 및 복구](./s7-db-recovery) | 테넌트 격리 유지, 무결성 검증, 감사 |

## 시나리오 작성 구조

각 시나리오는 다음 구조를 따릅니다.

1. 비즈니스 흐름 — 사용자 관점에서 무슨 일이 일어나는가
2. 보안 목표 — 이 흐름에서 지켜야 할 것은 무엇인가
3. 주요 단계 — 아키텍처가 각 단계를 어떻게 처리하는가
4. AWS 서비스 및 역할
5. 보안 체크리스트
6. 미해결 질문
