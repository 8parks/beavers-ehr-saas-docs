# S2. 환자 진료 기록 작성 및 조회

의사가 진료 소견·진단·처방을 작성·확정하고 간호사가 바이탈 등 보조 정보를 입력하는 흐름입니다. 서비스 내에서 가장 민감한 데이터가 생성·수정되는 영역이며, 역할 분리와 기록 무결성이 핵심 보안 요구사항입니다. 진료 기록 조회와 작성이 Lambda 레벨에서 분리되어 있으며, 의료법 제23조에 따라 전자의무기록에는 반드시 전자서명이 기재되어야 합니다. 전자서명은 인증된 EMR 전자서명 솔루션을 NAT Gateway를 통해 외부 연동하는 방식으로 구현하며, 당사자가 서명했다는 사실을 증명할 수 있는 구조를 갖추는 것이 핵심입니다.

| 항목 | 내용 |
|---|---|
| 주 사용 역할 | 의사, 간호사 |
| 핵심 데이터 | 이름, 연락처, 생년월일, 환자 ID, 예약 정보, 진료 소견, 진단, 처방 약물명·용량·기간, 알레르기/금기 정보, 의사·간호사 메모 |


## 1. 비즈니스 흐름

의사는 환자 조회 후 진료 기록을 작성하고, 진단 및 처방을 확정합니다. 간호사는 환자의 상태 정보(바이탈, 메모 등)를 입력하거나 일부 보조 정보를 수정할 수 있습니다. 모든 진료 기록은 환자 ID와 `tenant_id`를 기반으로 저장되며, 이후 조회 시 동일한 조건으로 필터링됩니다.

**서비스 흐름**

의사 로그인 → 대시보드 접속 → 환자 검색/선택 → 진료 내용 입력 → 첨부파일이 있는 경우 S3 업로드 → 의사의 전자서명으로 진료기록 저장 → 감사 로그 기록

**세부 설명**

- 진료 기록 조회: 사용자 요청 → 인증(JWT) → role 확인(의사 여부) → 환자 존재 및 tenant 검증 → 진료 기록 정보 확인 → 역할별 필드 필터링 → 결과 반환
- 진료 기록 작성: 사용자 요청 → 인증(JWT) → role 확인(의사 여부) → 환자 존재 및 tenant 검증 → 전자서명 요청 및 확인 → 데이터 암호화 → 데이터 저장 → 변경 이력 기록
  - 환자 ID와 사용자 `tenant_id`가 불일치하면 반드시 차단되어야 함

## 2. 보안 목표

- 의사만 진단/처방 확정 가능, 간호사는 보조 입력 및 허용된 필드에 대한 제한적 수정만 가능
- API Gateway에서 JWT 검증 시 `tenant_id` 추출 및 `role = doctor/nurse` 확인
- 데이터 수정 이력 감사로그 기록됨
- 동일 Tenant 내에서도 `tenant_id` 검증으로 환자 단위 접근 통제 적용
- 진료 기록 작성 시 인증된 외부 전자서명 솔루션 연동, API 키 + SHA-256 해시 기반 서명 적용
- 전자서명 미기재 시 적법한 전자의무기록으로 인정되지 않으며 의료법 제90조 형사벌 대상
- 수정 시 원본 보존 필수(의료법 제22조 제2항), 접속기록 별도 보관(의료법 제23조 제4항)
- 간단한 진료 기록은 Aurora에 Schema별 분리 저장하고 조회는 Read Instance, 저장 로직은 Write Instance를 사용
- 진료 파일은 진료 S3에 Tenant별 Prefix로 분리 저장
- 로그는 로그 S3에 Tenant별 Prefix로 분리 저장


## 3. 보안 설계

**인증 및 역할 기반 쓰기 통제**

Amazon Cognito User Pool 기반 로그인 처리 후, Pre-token Lambda Trigger를 통해 JWT에 `custom:tenant_id`, `role` 클레임을 주입합니다. API Gateway에서 JWT를 검증하고 `role = doctor/nurse` 여부를 확인하여 진료 기록 작성 API 접근을 제한합니다. 진단, 처방 확정은 의사인 경우에만 허용하며, 간호사는 허용된 필드(바이탈, 메모)에만 쓰기 가능합니다.

**테넌트 격리**

JWT의 `custom:tenant_id` claim으로 API 레이어에서 1차 검증하고, Aurora의 Schema-per-Tenant 방식으로 DB 스키마를 분리하여 2중 방어를 구성합니다. RLS 정책(`app.record_id = row.record_id`)을 적용하여 애플리케이션 레이어 우회 시에도 DB 계층에서 격리를 강제합니다.

**전자서명**

의료법 제23조에 따라 전자의무기록에는 반드시 전자서명이 기재되어야 합니다. 전자서명 미기재 시 적법한 전자의무기록으로 인정되지 않으며, 의료법 제90조에 따른 형사벌(500만원 이하의 벌금) 대상이 됩니다. 본 시나리오에서는 보건복지부 EMR 인증제 기준을 충족하는 외부 전자서명 솔루션을 연동하는 방식을 채택합니다. `create-record` Lambda에서 SHA-256 기반 `record_hash`를 생성하고, Secrets Manager에서 API 키를 런타임에 조회한 뒤 NAT Gateway를 통해 외부 전자서명 솔루션에 서명을 요청합니다. 반환된 서명값, 서명 시각, 서명자 정보(`cognito:sub`, `tenant_id`, `role`)를 Aurora Write Instance에 함께 저장하여 당사자가 서명했다는 사실을 증명할 수 있는 구조를 갖춥니다.

**기록 무결성 및 원본 보존**

의료법 제22조 제2항에 따라 진료 기록 수정 시 수정 전 원본이 모두 보존되도록 버전 이력으로 관리합니다. 원본 데이터는 삭제하지 않으며, 모든 수정에 수정자 ID·수정 시각·변경 전후 값을 감사 로그에 기록합니다. 진료기록부 보존 의무는 10년이며, 접속기록은 의료법 제23조 제4항에 따라 별도 보관합니다.

**파일 저장 보안**

진료 파일은 S3 Gateway Endpoint를 통해 진료 전용 S3 버킷에 Tenant별 Prefix(Tenant A Prefix / Tenant B Prefix)로 분리 저장합니다. 버킷은 Public Access를 차단하며 KMS CMK로 암호화합니다.

**자격증명 및 암호화 관리**

Lambda는 DB 접속 정보와 외부 전자서명 솔루션 API 키를 Secrets Manager Interface Endpoint를 통해 런타임에 조회합니다. Aurora 저장 데이터는 KMS CMK(고객 관리 키)로 암호화하며, CMK는 주기적으로 교체합니다.

**네트워크 경계 보안**

애플리케이션 레이어는 Private App Subnet, DB 레이어는 Private DB Subnet에 배치하며, 각 서브넷에 NACL과 Security Group을 이중 적용합니다. 외부 전자서명 솔루션 연계는 NAT Gateway를 통해서만 허용하여 아웃바운드 트래픽 경로를 단일화합니다. RDS Proxy를 통해 Aurora로의 직접 연결을 차단합니다.

**감사 로그 무결성**

CloudWatch와 CloudTrail로 수집된 로그는 S3 Gateway Endpoint를 통해 로그 전용 S3 버킷에 Tenant별 Prefix(Tenant1 Prefix / Tenant2 Prefix)로 분리 저장합니다. 전자서명 요청, 응답 이벤트(서명 시각, 서명자 정보, `record_hash`)도 감사 로그 대상에 포함합니다.

## 5. 보안 체크리스트

- [ ] 진단·처방 확정은 `role = doctor`만 가능하도록 Lambda에서 검증합니다.
- [ ] 간호사는 허용된 필드(바이탈·메모)에만 쓰기가 가능합니다.
- [ ] `create-record` Lambda에서 SHA-256 기반 `record_hash`가 생성됩니다.
- [ ] 서명값·서명 시각·서명자 정보(`cognito:sub`, `tenant_id`, `role`)가 Aurora Write Instance에 함께 저장됩니다.
- [ ] 전자서명 솔루션이 보건복지부 EMR 인증제 기준을 충족합니다.
- [ ] 진료 기록 수정 시 원본이 보존되고 변경 전후 값이 감사 로그에 기록됩니다.
- [ ] 진료기록부 보존 기간(10년)이 충족됩니다.
- [ ] 접속기록이 별도 보관됩니다(의료법 제23조 제4항).
- [ ] S3 버킷이 Public Access를 차단하고 KMS CMK로 암호화되어 있습니다.
- [ ] 전자서명 요청·응답 이벤트가 감사 로그 대상에 포함됩니다.
