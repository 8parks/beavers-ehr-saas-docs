# 컴플라이언스

이 플랫폼이 다루는 데이터는 법적으로 보호되는 의료 정보입니다. 어떤 규제가 적용되는지, 그 규제가 아키텍처 결정에 어떻게 반영되었는지를 정리합니다.

본 문서는 국내 개인정보보호법과 HIPAA/HITECH 요구사항을 AWS 기반 EHR SaaS 아키텍처 통제 항목에 매핑한다. 개인정보보호법은 개별 조항별 의무가 비교적 명확하므로 조항 단위로 요구사항과 구현 방안을 정리하였다. 반면 HIPAA/HITECH은 Security Rule, Privacy Rule, HITECH Act의 여러 조항이 인증, 접근통제, 감사로그, 암호화, 책임 공유 등 동일한 보안 통제 영역에 중복 적용되므로, 본 문서에서는 관련 조항을 통제 영역별로 묶어 정리한다.

::: warning 면책 조항
이 가이드는 법률 자문이 아닙니다. 실제 컴플라이언스 여부는 전문가 법률 검토를 통해 확인해야 합니다.
:::

이 플랫폼은 한국 내 병원을 주요 테넌트로 가정합니다. 해외 연구기관과의 데이터 공유가 발생할 경우 추가 규제가 적용될 수 있습니다.

| 규제 / 표준 | 주요 관심사 | 우선순위 |

| 개인정보보호법 | 개인정보 처리, 가명처리, 동의 | 높음 |
| 의료법 | 의료 기록 보존, 접근 권한 | 높음 |
| HIPAA | PHI 보호, 감사 로그, 사고 대응 | 높음 |

## 책임 공유 및 계약 범위 {#책임-공유-및-계약-범위}

누가 무엇에 대해 책임지는지 명확히 해야 합니다. 책임이 불분명하면 사고 발생 시 대응이 어렵습니다.

| 주체 | 책임 범위 |
|---|---|
| SaaS 운영자 | 플랫폼 인프라 보안, 테넌트 격리, 보안 모니터링 |
| 병원 (테넌트) | 사용자 접근 관리, 내부 정책 준수, 데이터 입력 품질 |
| 연구기관 | 연구 데이터 활용 범위 준수, 재식별 금지 |
| 클라우드 제공자 (AWS) | 물리적 인프라, 기반 서비스 보안 |

| 규제 | 조항 | 요구사항 | 아키텍처 구현 |
|---|---|---|---|

### 관련 조항

- 개인정보보호법 제26조 (업무위탁에 따른 개인정보의 처리 제한)
- 의료법 제22조 제2항 (진료기록부 등 - 보존 의무)
- 의료법 시행규칙 제15조 (진료기록부 등의 보존 기간)
- HITECH Act Title XIII Subtitle D / §13400
- HITECH Act §13401
- 45 CFR §160.103
- 45 CFR §164.502
- 45 CFR §164.504(e)
- 45 CFR §164.308, 45 CFR §164.310, 45 CFR §164.312, 45 CFR §164.316
- ISO 27001 5.20 공급업체 계약 내 정보 보안 관련 설명 (Addressing Information Security Within Supplier Agreements)

### 조항 요지

**개인정보보호법 제26조 (업무위탁에 따른 개인정보의 처리 제한)**
병원(위탁자)이 SaaS(수탁자)에게 환자 데이터 처리를 위탁하는 구조는 개인정보보호법상 처리위탁에 해당합니다. 수탁자(SaaS 운영자)는 위탁받은 목적 외로 개인정보를 이용할 수 없고, 위탁자(병원)의 지시 없이 개인정보를 처리할 수 없으며, 안전성 확보 조치(제29조)를 이행해야 합니다. SaaS 운영자가 테넌트 의료 데이터에 접근하지 않는다는 것을 단순 정책 선언이 아닌 기술적으로 불가능한 구조로 증빙해야 합니다.

**의료법 제22조 제2항 / 시행규칙 제15조**
진료기록부등의 보존 의무를 규정하며, SaaS가 이 기능을 위탁받아 운영하는 경우 시행규칙 제15조의 기록 유형별 법정 보존 기간을 계약서에 명시하고 기술적으로 구현해야 합니다.

**HIPAA / HITECH**
EHR SaaS가 병원 등 Covered Entity를 대신하여 PHI를 생성, 수신, 유지, 전송하는 구조라면 SaaS 제공자는 Business Associate에 해당할 가능성이 높습니다. HITECH Act는 Business Associate에게도 HIPAA Security Rule의 행정적, 물리적, 기술적 보호조치와 문서화 요건이 직접 적용되도록 책임을 확장합니다.

**ISO 27001 5.20**
공급업체와 정보 보안 요구사항을 수립하고 합의하도록 요구합니다. 계약서 수준에서 보안 의무를 명시하고 정기 검토 절차를 수립해야 합니다.

### 요구사항

- 병원 tenant, SaaS 제공자, AWS 간 책임 범위 정의
- PHI 처리 경로 및 저장 위치 문서화
- AWS BAA 검토 및 수락
- 병원과 SaaS 제공자 간 계약 범위 정리
- 하위 서비스 제공자 및 서브프로세서 사용 여부 확인
- tenant별 데이터 처리 책임 분리
- 보안통제 적용 증빙 보관
- 정책, 절차, 로그, 점검 결과 문서화
- 위탁계약에 기록 유형별 법정 보존 기간 명시 (의료법 시행규칙 제15조)

### 아키텍처 구현

- AWS Artifact에서 AWS BAA 검토 및 수락
- 병원 tenant와 SaaS 제공자 간 BAA 또는 위탁계약 검토
- PHI가 흐르는 AWS 서비스와 경로를 데이터 흐름도로 표시
- PHI가 비대상 서비스 또는 외부 서비스로 흘러가지 않도록 통제
- 병원, SaaS 제공자, AWS 간 책임 매트릭스 작성
- IAM 정책, KMS key policy, 네트워크 구성도, 로그 보존 정책, 백업 정책 문서화
- AWS Config, Security Hub, GuardDuty, CloudTrail을 활용해 보안 설정 및 운영 증빙 확보
- 위탁계약서에 의료법 시행규칙 제15조 기반 기록 유형별 보존 기간 명시
- 보존 기간 만료 자동 파기 스케줄러 운영 및 파기 이력 감사로그 보관

## PHI에 대한 정의 및 보호 정리 {#phi에-대한-정의-및-보호-정리}

| 규제 | 조항 | 요구사항 | 아키텍처 구현 |
|---|---|---|---|
| **개인정보보호법** | 제17조 (개인정보의 제공) | - 수집 목적 범위 내 제공<br>- tenant별 데이터 소유권 및 접근 범위 구분<br>- PHI가 로그에 평문으로 남지 않도록 제한 | - API Gateway/Lambda Authorizer에서 `tenant_id`, `role`, `purpose` 검증<br>- 감사로그에 PHI 원문 대신 `user_id`, `tenant_id`, `record_id`, `action`, `timestamp` 기록 |
| **의료법** | 제22조 제1항 (진료기록부 등 - 작성 의무)<br>제22조 제2항 (진료기록부 등 - 보존 의무)<br>시행규칙 제15조 (보존 기간) | - 진료에 관한 기록 작성 의무<br>- 기록 유형별 법정 보존 기간 준수<br>- 추가기재, 수정 전 원본까지 모두 보존 | - `retention_class` + `expires_at` 메타데이터 저장<br>- S3 Object Lock으로 법정 보존 기간 동안 삭제, 변조 방지<br>- 전자의무기록 수정 시 overwrite 대신 versioning으로 원본 보존 |
| **HIPAA** | `§262 / 45 CFR §160.103`<br>Privacy Rule<br>`45 CFR §164.502`<br>`HITECH §13405(b)` | - PHI/ePHI 데이터 식별 및 분류<br>- PHI 저장 위치, 처리 흐름 파악<br>- 목적 기반 접근통제 적용<br>- 연구 목적 데이터는 비식별, 가명처리 후 제공<br>- PHI 사용, 공개 이력 기록 | - RDS/Aurora, S3, CloudWatch 등 PHI 포함 리소스 식별<br>- 리소스 태깅 (`data_classification=PHI`, `tenant_id`, `environment`)<br>- PHI 저장 RDS/Aurora → private subnet 배치<br>- S3 Block Public Access + SSE-KMS 적용 |
| **ISO 27001** | `5.34` 개인정보 및 개인식별정보 보호 (Privacy And Protection Of PII) | - 개인정보 처리 흐름 문서화<br>- Minimum Necessary 원칙 적용 | - 데이터 흐름도 작성 및 관리<br>- 접근 목적별 역할 분리 구현 |

### 관련 조항

- 개인정보보호법 제17조 (개인정보의 제공)
- 의료법 제22조 제1항 (진료기록부 등 - 작성 의무)
- 의료법 제22조 제2항 (진료기록부 등 - 보존 의무)
- 의료법 시행규칙 제15조 (진료기록부 등의 보존 기간)
- HIPAA §262 / Social Security Act §1171
- 45 CFR §160.103
- HIPAA Privacy Rule
- 45 CFR Part 164 Subpart E
- 45 CFR §164.502
- HITECH Act §13405(b)
- 45 CFR §164.502(b)
- ISO 27001 5.34 개인정보 및 개인식별정보 보호 (Privacy And Protection Of PII)

### 조항 요지

**HIPAA**
보호대상 건강정보(PHI)를 개인을 식별할 수 있는 건강정보로 정의합니다. 여기에는 과거, 현재, 미래의 신체적 또는 정신적 건강상태, 의료서비스 제공, 의료비 지불과 관련된 정보가 포함됩니다. EHR SaaS에서는 환자 기록, 진단 정보, 처방 정보, 검사 결과, 의료 문서, 보험 및 진료비 정보 등을 PHI로 분류해야 합니다. PHI는 저장만 보호하는 것이 아니라 사용, 공개 목적과 권한에 따라 접근 범위를 제한해야 하며, Minimum Necessary 원칙에 따라 목적 달성에 필요한 최소 범위로 제한되어야 합니다.

**의료법 제22조 / 시행규칙 제15조**
의료인이 진료기록부, 조산기록부, 간호기록부 등 진료에 관한 기록을 작성, 보존하도록 의무화합니다. 전자의무기록(제23조 제1항)을 포함하며, 추가기재, 수정된 경우 수정 전 원본까지 모두 보존해야 합니다. 시행규칙 제15조는 기록 유형별 법정 보존 기간을 규정합니다.

| 기록 유형 | 보존 기간 |
|---|---|
| 진료기록부, 수술기록 | 10년 |
| 환자명부, 검사소견기록, 방사선사진 및 소견서, 간호기록부 | 5년 |
| 진단서 등의 부본 | 3년 |
| 처방전 | 2년 |

**ISO 27001 5.34**
개인정보 및 개인식별정보를 보호하기 위한 처리 흐름 문서화와 Minimum Necessary 원칙 적용을 요구합니다.

### 요구사항

- PHI/ePHI 데이터 식별 및 분류
- PHI 저장 위치와 처리 흐름 파악
- 목적 기반 접근통제 적용
- tenant별 데이터 소유권 및 접근 범위 구분
- 필요한 데이터만 제공
- PHI가 로그에 평문으로 남지 않도록 제한
- 연구 목적 데이터는 비식별 또는 가명처리 후 제공
- PHI 사용, 공개 이력 기록
- 진료기록 유형별 법정 보존 기간 준수 (의료법 시행규칙 제15조)
- 추가기재, 수정 전 원본 포함 보존 (의료법 제22조 제2항)

### 아키텍처 구현

- RDS/Aurora PostgreSQL의 환자 테이블, S3 의료문서, CloudWatch Logs 등 PHI 포함 가능 리소스 식별
- 리소스 태깅 적용
  - `data_classification=PHI`
  - `tenant_id=hospital_a`
  - `environment=prod`
- PHI 저장 RDS/Aurora는 private subnet에 배치
- S3 의료문서는 Block Public Access와 SSE-KMS 적용
- API Gateway/Lambda Authorizer에서 JWT의 `tenant_id`, `role`, `purpose` 검증
- 의사, 병원 관리자, 연구자별 접근 범위를 애플리케이션 계층에서 제한
- 감사로그에는 PHI 원문 대신 `user_id`, `tenant_id`, `record_id`, `action`, `timestamp` 중심으로 기록
- 데이터 생성 시 `retention_class` + `expires_at` 메타데이터 저장

  | `retention_class` | `expires_at` 기준 |
  |---|---|
  | `MEDICAL_RECORD` | 생성일 + 10년 |
  | `PATIENT_LIST` / `NURSING_RECORD` | 생성일 + 5년 |
  | `DIAGNOSIS_COPY` | 생성일 + 3년 |
  | `PRESCRIPTION` | 생성일 + 2년 |

- S3 Object Lock으로 법정 보존 기간 동안 기록 삭제, 변조 방지

**개인정보보호법 제28조의4 (가명정보에 대한 안전조치의무 등)**

**ISO 27001 5.34**
가명정보 처리 흐름을 문서화하고 재식별 방지 통제를 구현하도록 요구합니다.

수집 목적 외 정보 제공을 금지하며, 특히 멀티테넌트 SaaS 환경에서 병원 A의 데이터가 병원 B로 노출되는 Cross-Tenant 접근을 원천 차단할 것을 요구합니다.

**의료법 제23조 제3, 4항 (전자의무기록)**
제3항은 누구든지 정당한 사유 없이 전자의무기록에 저장된 개인정보를 탐지하거나 누출, 변조 또는 훼손하여서는 안 된다고 규정합니다. 제4항(2018. 3. 27. 신설)은 전자의무기록에 추가기재, 수정을 한 경우 보건복지부령으로 정하는 바에 따라 접속기록을 별도로 보관해야 한다고 의무화합니다.

**HIPAA**
총 4개 조항을 통해 인증을 규율합니다. ePHI에 접근하는 사용자가 실제 허가된 사용자임을 확인할 수 있어야 하며, 사용자 유형별(의사, 병원 관리자, 연구자, 환자) 인증 체계를 분리하고 민감 작업에는 추가 인증을 적용해야 합니다.

**ISO 27001 5.17, 8.2, 8.3**
인증 정보의 안전한 관리, 특권 계정에 대한 별도 통제, 역할별 정보 접근 제한을 요구합니다. 특히 SaaS 운영자(플랫폼 팀)가 고객의 의료데이터에 직접 접근하지 못하도록 기술적 분리를 요구합니다.

> **공통 핵심**: ePHI에 접근하는 사용자가 실제 허가된 사용자임을 항상 확인해야 하며, 플랫폼 운영자 자신도 고객 데이터에 무제한 접근해서는 안 되고, 모든 접속은 추적 가능해야 합니다.

### 조항 요지

**개인정보보호법 제17조 (개인정보의 제공)**
개인정보를 수집한 목적 범위에서만 제공할 수 있습니다. 의료진이 다른 tenant 환자 데이터에 접근하는 것은 제17조 제1항 위반이므로 API 레이어뿐 아니라 DB 레이어에서도 이중으로 차단해야 합니다. 연구기관 제공은 수집 목적(진료) 범위 밖이므로 가명처리(제28조의2) 근거를 명시한 별도 경로를 통해서만 허용됩니다.

**개인정보보호법 제21조 (개인정보의 파기)**
보존 의무가 있는 데이터라도 보존 기간 종료 후에는 파기해야 하므로 보존 기간 관리 구조는 반드시 설계에 포함되어야 합니다. 백업 데이터에 해당 레코드가 남아 있으면 법적으로 완전 파기로 인정되지 않으므로 백업에서도 복원이 불가능한 구조가 필요합니다.

**개인정보보호법 제26조 (업무위탁에 따른 개인정보의 처리 제한)**
SaaS 운영자가 테넌트 의료 데이터에 접근하지 않는다는 것을 기술적으로 증빙해야 합니다. 단순 정책 선언이 아닌 구조적 불가능 또는 접근 시 반드시 감사 기록이 남는 구조여야 합니다.

**의료법 제21조 / 제21조의2 / 제22조 제3항**
제21조는 환자 본인이 자신의 기록에 대한 열람, 사본 발급을 요청할 권리를 갖고, 환자 아닌 제3자에게 기록을 열람하게 하는 것은 법이 정한 예외 사유(제21조 제3항 각 호) 외에는 금지됩니다. 제21조의2는 의료기관 간 진료기록 전송 시 환자 동의 또는 법적 근거가 필요합니다. 제22조 제3항은 의료인이 진료기록부등을 거짓으로 작성하거나 고의로 사실과 다르게 추가기재, 수정하는 것을 금지합니다.

**HIPAA**
PHI의 기밀성과 무결성을 보장하고 미승인 접근이나 공개를 방지하도록 요구합니다. EHR SaaS에서는 단순 로그인 여부가 아니라 사용자의 `tenant`, `role`, `purpose`에 따라 접근 가능한 데이터 범위를 제한해야 합니다.

**ISO 27001 5.15, 5.16, 5.18, 8.12, 8.20, 8.22**
역할별 정보 접근 제한, 아이덴티티 관리, 데이터 유출 방지, 네트워크 보안 및 분리를 요구합니다.

### 아키텍처 구현

**① Cross-tenant 차단: 이중 검증 구조**

- 공통 Authorizer 레이어에서 모든 요청에 대해 `tenant_id` 강제 검증 (개별 API 누락 방지)
- DB 계층에서 schema-per-tenant 또는 PostgreSQL RLS로 이중 차단

**② 연구기관 제공: 단방향 파이프라인 + 네트워크 격리**

```
운영 DB (VPC A) → 가명처리 → Research Zone DB (VPC B)
```

- Research Zone → 운영 DB 역방향 접근을 Security Group / Network ACL로 차단
- 연구원 계정은 Research Zone의 read-only 엔드포인트에만 접근 권한 부여

**③ SaaS 운영자 접근 통제**

- 테넌트별 KMS CMK로 데이터 암호화
- 운영자 IAM Role에 CMK Decrypt 권한 미부여 → DB 물리 접근해도 평문 데이터 조회 불가
- CloudTrail로 CMK 접근 시도 자동 기록 → 미접근 증빙

**④ 파기 구조**

- 데이터 생성 시 `retention_class` + `expires_at` 메타데이터 저장, 주기적 만료 파기 처리
- 레코드별 KMS DEK 개별 관리 → 파기 시 DEK 삭제로 백업 포함 복원 불가 처리

**⑤ 진료기록 열람 및 무결성**

- 기록 열람 API에 요청자 신원 및 법정 권한 검증 로직 구현 (의료법 제21조 제3항 각 호 기반)
- 의료기관 간 전송 API에 환자 동의 여부 검증 단계 포함 (의료법 제21조의2)
- 진료기록 수정 시 overwrite 금지, versioning + hash로 무결성 보장 (의료법 제22조 제3항)

**개인정보보호법 제18조 (개인정보의 목적 외 이용, 제공 제한)**
수집 목적(진료) 외 용도로 원본 데이터를 사용하는 모든 경로를 차단해야 합니다. 연구 목적은 별도 법적 근거(제28조의2)를 적용하고, 원본 DB가 아닌 분리된 분석 환경에서만 처리해야 합니다.

진료기록부등(전자의무기록 포함, 추가기재, 수정 전 원본 포함)을 보건복지부령에 따라 보존하도록 의무화합니다. 제23조 제1항은 전자의무기록이 전자서명법에 따른 전자서명이 기재된 전자문서여야 함을 요건으로 규정합니다.

**의료법 제23조 제2항 (관리, 보존 시설, 장비 구비)**

**ISO 27001 8.11, 8.24**

### 관련 조항

- 의료법 제22조 제2항 (진료기록부 등 - 보존 의무)
- 의료법 제23조 제2항 (전자의무기록 관리, 보존 시설 및 장비 구비 의무)
- 의료법 시행규칙 제15조 (진료기록부 등의 보존 기간)
- HIPAA Security Rule
- 45 CFR §164.306, 45 CFR §164.308, 45 CFR §164.312
- ISO 27001 5.24 정보 보안 사고 관리 계획 및 준비 (Information Security Incident Management Planning and Preparation)
- ISO 27001 5.26 정보 보안 사고 대응 (Response To Information Security Incidents)
- ISO 27001 5.29 중단 상황에서의 정보 보안 (Information Security During Disruption)
- ISO 27001 5.30 업무 연속성을 위한 ICT 준비 (ICT Readiness For Business Continuity)
- ISO 27001 8.13 정보 백업 (Information Backup)
- ISO 27001 8.14 정보 처리 시설의 이중화 (Redundancy of information processing facilities)

### 조항 요지

**의료법 제22조 제2항 / 제23조 제2항 / 시행규칙 제15조**
진료기록부등을 법정 보존 기간 동안 보존해야 하므로 백업, 재해복구 체계가 법정 보존 기간과 연계되어야 합니다. 장애, 재해 발생 시에도 법정 보존 기간 중인 기록에 접근 가능해야 하며, 제23조 제2항은 전자의무기록의 안전한 관리, 보존을 위한 시설과 장비 구비를 의무화합니다.

**HIPAA Security Rule**
ePHI의 기밀성, 무결성뿐 아니라 가용성도 보호하도록 요구합니다. EHR SaaS는 장애나 사고 발생 시에도 필요한 시점에 의료정보에 접근할 수 있도록 백업과 복구 체계를 갖추어야 합니다.

**ISO 27001 5.24, 5.26, 5.29, 5.30, 8.13, 8.14**
사고 관리 계획 수립, 정보 보안 사고 대응 절차, 중단 상황 대응 방안, 업무 연속성을 위한 ICT 준비, 정보 백업 체계 구축, 정보 처리 시설 이중화를 요구합니다.

### 요구사항

- ePHI 저장소 백업 및 장애 복구 절차
- 백업 데이터 암호화 및 복구 테스트
- RPO/RTO 목표 정의 및 재해복구 Runbook 작성
- 법정 보존 기간 중인 기록의 가용성 보장 (의료법 제22조 제2항, 시행규칙 제15조)
- 전자의무기록 관리, 보존을 위한 시설, 장비 구비 (의료법 제23조 제2항)
- 백업 데이터도 법정 보존 기간에 맞춰 보존

### 아키텍처 구현

- RDS/Aurora Multi-AZ 구성, automated backup 및 snapshot 설정
- S3 versioning 및 lifecycle policy 검토
- 백업 데이터 SSE-KMS 암호화
- CloudWatch Alarm으로 장애 탐지
- 복구 절차 Runbook 문서화, RPO/RTO는 비용과 아키텍처 수준에 따라 목표값으로 제시
- 법정 보존 기간이 남아있는 기록은 백업에서도 S3 Object Lock 적용 (의료법 제22조 제2항)
- 진료기록 유형별 백업 보존 기간을 법정 기간에 맞춰 설정

  | 기록 유형 | 백업 보존 기간 |
  |---|---|
  | 진료기록부, 수술기록 | 10년 |
  | 환자명부, 검사소견기록, 방사선사진, 간호기록부 | 5년 |
  | 진단서 등의 부본 | 3년 |
  | 처방전 | 2년 |

- 복구 테스트 결과를 감사로그로 보관

## NPRM 선제대응 및 향후 고도화 계획 {#nprm-선제대응-및-향후-고도화-계획}

본 프로젝트는 현행 HIPAA Security Rule 및 HITECH 요구사항을 기준으로 AWS 기반 EHR SaaS 보안 아키텍처를 설계합니다. 다만 향후 Security Rule 개정 가능성을 고려하여, NPRM에서 제안된 강화 항목 중 아키텍처에 선제적으로 반영 가능한 부분과 향후 고도화로 추가 검토가 필요한 부분을 구분합니다.

NPRM 선제대응 항목은 현재 법적 의무로 확정된 요구사항이라기보다, 향후 규제 강화에 대비하기 위한 보안 고도화 방향으로 해석합니다.

| NPRM 제안 사항 | 현행 규제 | 본 프로젝트 대응 |
|---|---|---|
| Required/Addressable 구분 폐지 → 전 항목 의무화 | R/A 구분 존재 | 핵심 암호화, 접근통제, 감사로그, 백업 통제는 기본 반영. 운영 절차 문서화와 정기 검토 체계 보완 필요 |
| 취약점 스캔 의무화 | 명시적 주기 없음 | AWS Inspector 또는 외부 스캐너 도입 검토. Phase 2에서 6개월 주기 취약점 스캔 절차 수립 |
| 침투 테스트 의무화 | 명시적 주기 없음 | 연 1회 외부 펜테스트 위탁 및 결과 보고서 보관 절차 수립 필요 |

| BA 컨틴전시 플랜 발동 시 24시간 내 통지 | 침해 통지 중심 | EventBridge → SNS/Slack 자동 알림 워크플로우를 Phase 2 운영 고도화 항목으로 반영 |
| 자산 인벤토리 및 네트워크 맵 작성, 관리 의무화 | 명시적 자동화 요구 없음 | AWS Config, Resource Tagging, 네트워크 다이어그램 버전관리로 대응 가능. 정기 검토 절차 필요 |
| MFA 의무화 | Addressable 또는 권고 성격 | Cognito MFA 필수화로 선제 반영 |
| 연 1회 컴플라이언스 감사 의무화 | Risk Analysis 중심 | AWS Audit Manager 도입 검토 및 연 1회 내부 점검 절차 수립 |
