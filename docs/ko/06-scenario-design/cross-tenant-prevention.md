# 시나리오: 크로스 테넌트 접근 방지

## 1. 비즈니스 로직 정의

한 병원(테넌트)의 사용자가 다른 병원의 PHI에 접근하려 시도하는 상황에서 이를 방지하는 메커니즘입니다.

## 2. 보안 목표

- 테넌트 격리는 서비스 레이어와 데이터베이스 레이어 모두에서 시행되어야 합니다.
- 크로스 테넌트 접근 시도는 탐지되고 로그로 기록되어야 합니다.
- 격리 메커니즘은 단일 레이어에 의존하지 않아야 합니다 (심층 방어).

## 3. 격리 레이어

| 레이어 | 메커니즘 | 설명 |
|--------|---------|------|
| 인증 레이어 | Cognito + JWT | tenant_id 클레임 발급 |
| API 레이어 | API Gateway + Lambda Authorizer | JWT 검증 |
| 서비스 레이어 | Lambda 비즈니스 로직 | tenant_id 기반 쿼리 범위 제한 |
| 데이터베이스 레이어 | Schema-per-Tenant | 스키마 수준 격리 |
| 심층 방어 | Row-Level Security | 추가 데이터베이스 레이어 검증 |

## 7. 컴플라이언스 매핑

| 요구사항 범주 | 설계 대응 |
|--------------|---------|
| 접근 제어 | 다층 테넌트 격리 |
| 데이터 보호 | 테넌트 간 PHI 격리 |

## 8. 위협 및 완화

| 위협 | 완화 방법 |
|------|---------|
| T-001 크로스 테넌트 PHI 접근 | 다층 격리 (JWT + 서비스 레이어 + DB 스키마 + RLS) |

## 9. 설계 체크리스트

- [ ] 모든 API 엔드포인트에서 JWT tenant_id 검증
- [ ] Lambda 비즈니스 로직이 tenant_id로 쿼리 범위 제한
- [ ] 테넌트 스키마에서만 데이터 조회
- [ ] RLS 정책이 활성화되어 있음
- [ ] 크로스 테넌트 접근 시도 시 오류 및 로그 생성
- [ ] 이상 접근 패턴 탐지 알람 설정

## 10. 미해결 질문

- 서비스 레이어의 테넌트 검증 우회 가능한 시나리오 식별
- RLS 정책의 성능 영향 측정
