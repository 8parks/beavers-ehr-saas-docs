# 시나리오: 연구 데이터셋 생성

## 1. 비즈니스 로직 정의

연구자가 PHI에서 파생된 가명처리된 데이터셋을 요청하고 생성하는 흐름입니다.

## 2. 보안 목표

- 연구 데이터셋은 충분히 가명처리되어야 합니다.
- 데이터셋 생성 과정이 완전히 감사 가능해야 합니다.
- 원본 PHI와 가명처리된 데이터 간의 매핑 정보는 별도로 안전하게 보관되어야 합니다.
- 연구자는 원본 PHI에 직접 접근할 수 없어야 합니다.

## 3. 기능 흐름

1. 연구자가 데이터셋 생성을 요청합니다.
2. 시스템이 요청을 검토하고 승인합니다.
3. 가명처리 파이프라인이 실행됩니다.
4. 가명처리된 데이터셋이 생성됩니다.
5. 연구자가 데이터셋에 접근합니다.
6. 전체 과정이 감사 로그로 기록됩니다.

## 6. AWS 서비스 및 책임

| AWS 서비스 | 역할 |
|-----------|------|
| AWS Lambda | 가명처리 파이프라인 실행 |
| AWS KMS | 가명 식별자 해시 생성 |
| Amazon DynamoDB | 가명 매핑 테이블 저장 |
| Amazon S3 | 데이터셋 저장 |
| AWS CloudTrail | 생성 이벤트 감사 |

## 7. 보안 컨트롤

| 컨트롤 | 구현 방법 | 관련 위협 |
|--------|---------|---------|
| 가명처리 | KMS 기반 해시, 매핑 테이블 분리 | T-006 불충분한 가명처리 |
| 접근 제어 | 연구자 역할 검증, 승인 워크플로우 | 무단 데이터 접근 |
| 감사 | 전체 생성 과정 기록 | 추적성 부재 |

## 8. 컴플라이언스 매핑

| 요구사항 범주 | 설계 대응 |
|--------------|---------|
| 데이터 보호 | 연구 데이터 가명처리 |
| 감사 추적성 | 데이터셋 생성 이력 기록 |
| 제3자 공유 | 가명처리 기준 충족 후 공유 |

## 9. 위협 및 완화

| 위협 | 완화 방법 |
|------|---------|
| T-006 불충분한 가명처리 | 다단계 가명처리, 재식별 위험 평가 |

## 10. 설계 체크리스트

- [ ] 모든 PHI 식별자가 가명처리됩니다.
- [ ] 가명 매핑 테이블이 원본 PHI와 동일 수준으로 보호됩니다.
- [ ] 연구자는 매핑 테이블에 접근할 수 없습니다.
- [ ] 데이터셋 생성 요청, 승인, 실행 이벤트가 기록됩니다.
- [ ] 재식별 위험 평가가 수행됩니다.

## 11. 미해결 질문

- 한국 개인정보보호법 가명처리 기준 충족 여부 판단 방법 (법률 검토 필요)
- 가명처리 수준 결정 기준 및 승인 절차
