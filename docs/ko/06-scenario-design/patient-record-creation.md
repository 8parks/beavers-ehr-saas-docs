# 시나리오: 환자 기록 생성 & 서명

## 1. 비즈니스 로직 정의

의사가 환자의 진료 기록을 작성하고, 해당 기록에 디지털 서명을 적용하여 저장하는 흐름입니다.

## 2. 보안 목표

- 인가된 의사만 해당 테넌트(병원)의 환자 기록을 작성할 수 있어야 합니다.
- 기록에 디지털 서명을 적용하여 이후 변조를 탐지할 수 있어야 합니다.
- 기록 생성 이벤트는 감사 로그로 보존되어야 합니다.
- 다른 테넌트의 환자 기록에는 접근할 수 없어야 합니다.

## 3. 기능 흐름 (사용자 관점)

1. 의사가 로그인합니다.
2. 의사가 환자를 검색합니다.
3. 의사가 진료 기록을 작성합니다.
4. 의사가 기록에 서명합니다.
5. 서명된 기록이 저장됩니다.
6. 감사 로그가 생성됩니다.

## 4. 아키텍처 흐름 (AWS 서비스 관점)

<!-- 작성 예정: 아키텍처 흐름 다이어그램 -->

1. 사용자가 Cognito를 통해 인증합니다.
2. API 요청이 WAF 및 API Gateway를 통과합니다.
3. Lambda가 JWT 클레임과 테넌트 컨텍스트를 검증합니다.
4. Lambda가 테넌트 전용 스키마에서 환자 데이터를 조회합니다.
5. KMS가 기록 해시에 서명합니다.
6. RDS가 기록과 서명 메타데이터를 저장합니다.
7. CloudWatch 및 CloudTrail이 관련 이벤트를 기록합니다.

## 6. AWS 서비스 및 책임

| AWS 서비스 | 역할 |
|-----------|------|
| Amazon Cognito | 사용자 인증 및 JWT 발급 |
| AWS WAF | 웹 공격 방어 |
| Amazon API Gateway | API 진입점 및 인가자 통합 |
| AWS Lambda | 비즈니스 로직 및 테넌트 검증 |
| Amazon RDS/Aurora PostgreSQL | EHR 데이터 저장 |
| AWS KMS | 기록 서명 및 암호화 |
| Amazon CloudWatch | 애플리케이션 이벤트 로깅 |
| AWS CloudTrail | AWS API 감사 로깅 |

## 7. 보안 컨트롤

| 컨트롤 | 구현 방법 | 관련 위협 |
|--------|---------|---------|
| 테넌트 격리 | tenant_id 클레임 및 스키마 레벨 격리 | 크로스 테넌트 데이터 접근 |
| 기록 무결성 | KMS Sign/Verify | 의료 기록 변조 |
| 감사 로그 | CloudTrail 및 CloudWatch | 추적성 부재 |

## 8. 컴플라이언스 매핑

| 요구사항 범주 | 설계 대응 |
|--------------|---------|
| 접근 제어 | 역할 기반 및 테넌트 인식 인가 |
| 감사 추적성 | 기록 생성 및 서명 이벤트 로그 |
| 무결성 | 기록 해시 및 KMS 서명 저장 |

## 9. 위협 및 완화

| 위협 | 완화 방법 |
|------|---------|
| 무단 기록 수정 | KMS 서명 및 감사 로그 |
| 크로스 테넌트 접근 | 테넌트 인식 검증 및 스키마 격리 |

## 10. 설계 체크리스트

- [ ] JWT에 테넌트 및 역할 클레임이 포함되어 있습니다.
- [ ] Lambda가 PHI 조회 전 테넌트 컨텍스트를 검증합니다.
- [ ] 데이터베이스 접근이 올바른 테넌트 스키마 범위로 제한됩니다.
- [ ] 서명 전 기록 해시가 생성됩니다.
- [ ] KMS 서명 키 접근이 제한되어 있습니다.
- [ ] 감사 이벤트가 생성됩니다.
- [ ] 로그에 불필요한 PHI가 포함되지 않습니다.
